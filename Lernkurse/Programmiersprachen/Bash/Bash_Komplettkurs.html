Bash Komplettkurs

Bash (Bourne Again SHell) ist die Standard-Shell auf nahezu jedem Linux-System — und damit das Fundament jeder Automatisierung, jedes Systemskripts und jedes Werkzeugs, das TrafkHop Entertainment auf seiner Linux-Distro TrafkTux laufen lässt. Diese Shell ist keine bloße Befehlszeile: Sie ist eine vollständige Skriptsprache, ein Prozess-Orchestrator und die Klebeschicht, die Programme miteinander verbindet. Wer Bash wirklich beherrscht, kann das System nach Belieben formen.

Dieser Kurs ist kein Tutorial für Gelegenheitsnutzer. Er deckt alles ab, was Bash zu bieten hat — von den ersten Variablen bis hin zu Event-Driven Daemons, Unix Sockets und fortgeschrittenen Architektur-Patterns. Nach diesem Kurs sind externe Ressourcen überflüssig.

Voraussetzung ist ein laufendes Linux-System. Alle Beispiele beziehen sich auf die TrafkTux-Umgebung mit Hyprland als Wayland-Compositor, was aber für das Verständnis der Sprache selbst keine Rolle spielt.

Kapitel 1 — Die Shell, Skripte und das erste Programm

Bevor man irgendwelchen Code schreibt, muss man verstehen, was Bash eigentlich ist und wie es mit dem Betriebssystem zusammenarbeitet. Bash ist ein Prozess wie jeder andere — wenn man ein Terminal öffnet, startet das System eine Bash-Instanz, die auf Eingaben wartet. Skripte sind nichts anderes als Textdateien voller Bash-Befehle, die sequenziell ausgeführt werden.

1.1 — Die Shebang-Zeile

Jedes Bash-Skript beginnt mit #!/bin/bash. Diese erste Zeile heißt Shebang (kurz für "sharp-bang", also #!). Wenn das Betriebssystem eine ausführbare Textdatei startet, liest es genau diese erste Zeile und übergibt die Datei an den dort genannten Interpreter.

#!/bin/bash → startet explizit Bash #!/bin/sh → startet die System-Shell (auf Ubuntu oft dash, nicht bash!) #!/usr/bin/env bash → sucht bash im PATH — portabler für verschiedene Systeme

Auf Ubuntu ist /bin/sh meistens dash — eine schlanke, schnelle, aber mit Bash inkompatible Shell. Alles Bash-spezifische wie Arrays, [[...]], Prozesssubstitution oder Regex mit =~ funktioniert unter dash nicht. Für TrafkTux-Skripte ist #!/bin/bash immer die richtige Wahl.

Damit ein Skript ausgeführt werden kann, muss es das Execute-Bit gesetzt haben. Das erledigt chmod:

chmod +x mein_skript.sh ./mein_skript.sh

Alternativ kann man Bash direkt aufrufen — dann ist die Shebang-Zeile egal, weil man den Interpreter ja selbst angibt:

bash mein_skript.sh

1.2 — Das erste Skript

Das klassische erste Programm gibt eine Nachricht aus. echo schreibt Text auf stdout — den Standard-Ausgabekanal, der normalerweise das Terminal ist:

#!/bin/bash echo "TrafkHop Entertainment" echo "Bash läuft."

Kommentare beginnen mit #. Alles dahinter auf der Zeile wird von Bash ignoriert und dient nur der Dokumentation:

# Das ist ein Kommentar echo "Das wird ausgegeben" # Das auch hier ist ein Kommentar

Setze das Execute-Bit und führe es auf beide Arten aus — einmal mit ./ und einmal mit bash.

Kapitel 2 — Variablen, Quoting und Spezialvariablen

Variablen sind benannte Speicherplätze für Werte. In Bash sind alle Variablen standardmäßig Strings — es gibt keinen eingebauten Integer- oder Float-Typ, auch wenn man bestimmte Operationen auf Zahlen anwenden kann. Dieses Kapitel ist eines der wichtigsten, weil Quoting — also das richtige Setzen von Anführungszeichen — die häufigste Fehlerquelle im Shell-Scripting ist.

2.1 — Zuweisung und Abruf

Eine Variable wird ohne Leerzeichen um das Gleichheitszeichen zugewiesen. Das ist einer der häufigsten Fehler für Einsteiger: Bash interpretiert NAME = "TrafkHop" als den Befehl NAME mit zwei Argumenten.

NAME="TrafkHop" # richtig NAME = "TrafkHop" # FEHLER: bash denkt NAME ist ein Befehl echo "$NAME" # Abruf mit $ → TrafkHop

Doppelte Anführungszeichen beim Abruf sind fast immer Pflicht. Ohne sie passiert Word Splitting — Bash zerhackt den Wert an Leerzeichen und übergibt ihn als mehrere Argumente:

DATEI="mein dokument.txt" cat $DATEI # FEHLER: cat kriegt "mein" und "dokument.txt" als zwei Argumente cat "$DATEI" # richtig: ein Argument mit Leerzeichen

Die geschweiften Klammern ${} sind nötig, wenn direkt nach der Variable weitere Zeichen folgen sollen:

echo "${NAME}_backup" # → TrafkHop_backup echo "$NAME_backup" # → leer! Bash sucht Variable namens "NAME_backup"

2.2 — Variablen-Typen und declare

Bash kennt verschiedene Variablen-Typen, die mit declare gesetzt werden. Im Alltag reicht die normale Zuweisung, aber declare ermöglicht präzisere Kontrolle:

NAME="wert" # lokale Variable (nur im aktuellen Prozess) export NAME="wert" # Umgebungsvariable (auch für Kind-Prozesse sichtbar) local NAME="wert" # nur innerhalb einer Funktion gültig readonly NAME="wert" # kann nicht überschrieben werden declare -i ZAHL=42 # Integer-Variable (erzwingt Zahlen-Typ) declare -a ARRAY=() # normales Array declare -A HASHMAP=() # assoziatives Array (Key-Value-Map) declare -x NAME # synonym für export declare -r NAME # synonym für readonly

Umgebungsvariablen wie $HOME, $USER oder $PATH werden von der Login-Shell gesetzt und sind immer verfügbar. Sie sind mit export markiert und werden an alle Kind-Prozesse weitergegeben.

2.3 — Quoting: Das Herzstück des Shell-Scriptings

Quoting ist der komplexeste und fehlerträchtigste Teil von Bash. Es gibt drei Arten von Anführungszeichen, die sich grundlegend unterscheiden:

'einfache Anführungszeichen' # ALLES ist literal. Kein $, kein \, keine Expansion. echo '$HOME' # → gibt wirklich $HOME aus, nicht den Pfad "doppelte Anführungszeichen" # Variablen werden expandiert. Globbing (* ? []) wird verhindert. # \n, \t werden NICHT interpretiert (außer in $'...' oder echo -e). echo "$HOME" # → /home/benutzername $'...' (ANSI-C Quoting) # Escape-Sequenzen werden interpretiert: \n (Newline), \t (Tab), \x1f, \0 etc. printf $'Zeile1\nZeile2\n' # → zwei Zeilen

Kein Quoting (also nackter Text ohne Anführungszeichen) führt zu Word Splitting und Globbing. Globbing bedeutet, dass Zeichen wie *, ? und [] zu Dateinamen expandiert werden. Das ist manchmal erwünscht, aber nie bei Variablen-Abrufen.

# Globbing gewollt (alle .sh-Dateien): for f in *.sh; do echo "$f"; done # Globbing ungewollt — FEHLER: MUSTER="*.sh" for f in $MUSTER; do echo "$f"; done # expandiert zu Dateiliste! for f in "$MUSTER"; do echo "$f"; done # → gibt literal "*.sh" aus — auch falsch # Lösung: Globbing explizit, Variablen immer quoten

Faustregel: Immer "$VARIABLE" in Anführungszeichen schreiben. Die einzige Ausnahme ist, wenn man Word Splitting oder Globbing absichtlich möchte.

2.4 — Spezialvariablen

Bash setzt automatisch eine Reihe von Variablen, die Informationen über das laufende Skript und seine Umgebung enthalten:

$0 # Name des Skripts (so wie es aufgerufen wurde) $1, $2... # Argumente 1, 2, ... $@ # alle Argumente als separate Strings — immer als "$@" nutzen! $* # alle Argumente als ein String (selten nützlich) $# # Anzahl der Argumente $? # Exit-Code des letzten Befehls (0=Erfolg, alles andere=Fehler) $$ # PID (Prozess-ID) des aktuellen Skripts $! # PID des zuletzt gestarteten Hintergrundprozesses $- # aktive Shell-Optionen $_ # letztes Argument des vorherigen Befehls

Ein praktisches Beispiel: Mit realpath "$0" und dirname kann ein Skript seinen eigenen absoluten Pfad und seinen Ordner bestimmen — unabhängig davon, wie es aufgerufen wurde:

SCRIPT_PATH="$(realpath "$0")" SCRIPT_DIR="$(dirname "$(realpath "$0")")" # So findet das Skript seine "Geschwister"-Dateien, egal wo es liegt

2.5 — Arithmetik

Bash rechnet nicht automatisch mit Zahlen — alles ist erstmal Text. Es gibt drei Wege zu rechnen, wobei $(( )) die moderne und bevorzugte Methode ist:

# $(( )) — Arithmetik-Expansion (bevorzugt): ERGEBNIS=$(( 3 + 4 )) # → 7 $(( ZAHL * 2 )) # Variablen ohne $ innerhalb von (( )) $(( ZAHL ** 2 )) # Potenz $(( ZAHL % 3 )) # Modulo (Rest bei Division) # let — ältere Syntax: let ERGEBNIS=3+4 # expr — externer Befehl, langsam, vermeiden: ERGEBNIS=$(expr 3 + 4)

Fließkomma-Rechnung geht in Bash nicht mit $(( )). Dafür braucht man externe Tools wie bc oder awk:

echo "scale=2; 10 / 3" | bc # → 3.33 awk "BEGIN {print 10/3}" # → 3.33333 # Praxisbeispiel: Integer-Prozent in Dezimalzahl umwandeln: local float_val=$(awk "BEGIN {print $val/100}")

Kapitel 3 — Bedingungen: if, case und Testoperatoren

Bedingungen sind die Grundlage jeder Entscheidungslogik in einem Skript. In Bash ist dabei alles auf Exit-Codes aufgebaut: 0 bedeutet true (Erfolg), alles andere bedeutet false (Fehler). Das klingt umgekehrt, ist aber Unix-Tradition.

3.1 — if und Test-Syntax

Die Grundstruktur eines if-Statements prüft den Exit-Code eines Befehls. Ein beliebiger Befehl kann als Bedingung stehen — Bash interessiert sich nur dafür, ob er 0 oder ungleich 0 zurückgibt:

if BEFEHL; then ... elif ANDERER_BEFEHL; then ... else ... fi # Beispiel: if prüft ob ls erfolgreich war if ls /tmp; then echo "Verzeichnis existiert und ist lesbar" fi

Für Vergleiche nutzt man den [[ ]]-Operator (bash-spezifisch, bevorzugt) oder den alten [ ]-Befehl (POSIX-kompatibel). [[ ]] ist sicherer und mächtiger und sollte in Bash-Skripten immer bevorzugt werden.

if [[ "$NAME" == "TrafkHop" ]]; then echo "Willkommen" fi

3.2 — [[ ]] vs [ ] — Vergleichsoperatoren

Innerhalb von [[ ]] stehen alle Vergleichsoperatoren für Strings und Zahlen zur Verfügung. Bei Zahlen kann man alternativ auch (( )) verwenden, was normale mathematische Vergleichszeichen erlaubt:

# String-Vergleiche in [[ ]]: [[ "$A" == "$B" ]] # gleich [[ "$A" != "$B" ]] # ungleich [[ "$A" < "$B" ]] # lexikographisch kleiner [[ -z "$A" ]] # leer (zero length) [[ -n "$A" ]] # nicht leer (non-zero length) [[ "$A" =~ REGEX ]] # Regex-Match — nur in [[ ]] möglich! # Zahlen-Vergleiche in [[ ]] — mit langen Operatoren: [[ "$A" -eq "$B" ]] # gleich (equal) [[ "$A" -ne "$B" ]] # ungleich (not equal) [[ "$A" -lt "$B" ]] # kleiner als (less than) [[ "$A" -le "$B" ]] # kleiner gleich(less or equal) [[ "$A" -gt "$B" ]] # größer als (greater than) [[ "$A" -ge "$B" ]] # größer gleich (greater or equal) # Zahlen-Vergleiche in (( )) — mit normalen Zeichen: if (( MOUSE_Y >= THRESHOLD )); then echo "Über dem Schwellenwert" fi # Logische Verknüpfungen innerhalb von [[ ]]: if [[ "$MOUSE_Y" -ge "$THRESHOLD" && "$HIDDEN" == true ]]; then echo "beides erfüllt" fi # In [ ] muss man && außerhalb schreiben: if [ "$A" -eq 1 ] && [ "$B" -eq 2 ]; then ...

3.3 — Datei- und Verzeichnis-Tests

Ein sehr häufiger Anwendungsfall ist das Testen von Dateien und Verzeichnissen. Bash kennt eine Reihe von Test-Flags dafür:

[[ -f "$DATEI" ]] # ist eine reguläre Datei (file) [[ -d "$PFAD" ]] # ist ein Verzeichnis (directory) [[ -e "$PFAD" ]] # existiert — egal ob Datei oder Ordner [[ -L "$PFAD" ]] # ist ein Symlink (link) [[ -r "$DATEI" ]] # ist lesbar (readable) [[ -w "$DATEI" ]] # ist schreibbar (writable) [[ -x "$DATEI" ]] # ist ausführbar (executable) [[ -s "$DATEI" ]] # existiert und ist nicht leer (size > 0) # Praktisches Beispiel: Konfigurationsdatei laden oder Default benutzen: if [[ -f "$CONFIG_FILE" ]]; then CONFIG="$(cat "$CONFIG_FILE")" else CONFIG='{"host":"localhost"}' fi # return in Funktionen: return 1 = "false" zurückgeben is_cache_valid() { [[ -f "$CACHE_FILE" ]] || return 1 # ... weitere Prüfungen }

3.4 — case — Muster-Vergleiche

case ist Bashs switch-Statement. Es ist besonders nützlich für Befehlszeilen-Argumente und Menü-Logik. Muster nutzen Glob-Syntax, nicht Regex. Die ;; am Ende jedes Blocks sind Pflicht.

case "$VARIABLE" in muster1) echo "Muster 1 gefunden" ;; muster2|muster3) echo "Muster 2 oder 3" # | ist das Oder ;; *.sh) echo "Eine .sh-Datei" # Glob-Muster funktionieren ;; *) echo "Nichts passt — Default" ;; esac # Praxisbeispiel: Helligkeit steuern case "$ACTION" in up) set_value $(( current + 5 )) ;; down) set_value $(( current - 5 )) ;; status) echo "$current" ;; *) echo "Unbekannte Aktion: $ACTION"; exit 1 ;; esac

Kapitel 4 — Arrays: Normale und Assoziative

Arrays sind geordnete Sammlungen von Werten. In Bash sind sie ein-dimensional und 0-basiert. Es gibt zwei Arten: normale Arrays mit numerischen Indizes und assoziative Arrays mit String-Schlüsseln. Beide sind in der Praxis unverzichtbar.

4.1 — Normale Arrays (Indexed Arrays)

Ein normales Array wird mit runden Klammern erstellt. Der Zugriff erfolgt mit eckigen Klammern. Das kritischste Detail: beim Iterieren immer "${ARRAY[@]}" in Anführungszeichen, sonst werden Elemente mit Leerzeichen aufgespalten:

# Erstellen: SPIELE=("Tetris" "Pong" "Snake") SPIELE[3]="Breakout" # direkte Index-Zuweisung # Zugriff: echo "${SPIELE[0]}" # → Tetris echo "${SPIELE[@]}" # → alle Elemente echo "${#SPIELE[@]}" # → Länge: 4 echo "${!SPIELE[@]}" # → alle Indizes: 0 1 2 3 # Slicing — Teilarray: echo "${SPIELE[@]:1:2}" # → Pong Snake (ab Index 1, 2 Elemente) # Letztes Element: echo "${SPIELE[-1]}" # → Breakout (Bash 4.3+) # Element hinzufügen: SPIELE+=("Galaga") # Element löschen (Lücke bleibt, Index nicht neu nummeriert!): unset "SPIELE[1]" # Nach unset re-indexieren: SPIELE=("${SPIELE[@]}") # In Schleife — IMMER mit Anführungszeichen: for spiel in "${SPIELE[@]}"; do echo "Spiel: $spiel" done # Mit Index: for i in "${!SPIELE[@]}"; do echo "[$i] ${SPIELE[$i]}" done

Der Unterschied zwischen "${ARRAY[@]}" und ${ARRAY[@]} ohne Anführungszeichen ist entscheidend:

SPIELE=("Super Mario" "Half Life") for s in ${SPIELE[@]}; do echo "$s"; done # → 4 Elemente! (falsch) for s in "${SPIELE[@]}"; do echo "$s"; done # → 2 Elemente (korrekt)

4.2 — Assoziative Arrays (Hash Maps)

Assoziative Arrays funktionieren wie Key-Value-Maps in anderen Sprachen. Sie müssen zwingend mit declare -A deklariert werden — ohne diese Deklaration erstellt Bash ein normales Array und ignoriert die String-Schlüssel:

# Deklaration ist PFLICHT: declare -A KONFIGURATION # Befüllen: KONFIGURATION["host"]="localhost" KONFIGURATION["port"]="8080" # Oder direkt bei Deklaration: declare -A FARBEN=(["rot"]="#FF0000" ["grün"]="#00FF00") # Zugriff: echo "${KONFIGURATION["host"]}" # → localhost # Alle Keys: echo "${!KONFIGURATION[@]}" # → host port # Alle Werte: echo "${KONFIGURATION[@]}" # → localhost 8080 # Key-Value-Iteration: for key in "${!KONFIGURATION[@]}"; do echo "$key = ${KONFIGURATION[$key]}" done # Prüfen ob Key existiert: if [[ -v KONFIGURATION["host"] ]]; then echo "Key host existiert" fi # Key löschen: unset "KONFIGURATION[port]" # Als "Set" nutzen (Existenz, nicht Wert zählt): declare -A exclude exclude["firefox"]=1 if [[ -z "${exclude[$cmd]}" ]]; then echo "$cmd ist nicht ausgeschlossen" fi

4.3 — mapfile / readarray

mapfile (Synonym: readarray) liest Zeilen aus einer Datei oder einem Befehl direkt in ein Array. Das -t-Flag entfernt das abschließende Newline von jeder Zeile. Für Dateinamen mit Leerzeichen kombiniert man es mit find -print0 und dem Null-Byte als Separator:

# Aus einer Datei: mapfile -t ZEILEN < datei.txt # Aus einem Befehl (Process Substitution — mehr dazu in Kapitel 6): mapfile -t PAKETE < <(dpkg --list | awk '{print $2}') # Mit Null-Byte-Separator (für Dateinamen mit Leerzeichen — sichere Praxis!): mapfile -t -d $'\0' DATEIEN < <(find . -name "*.sh" -print0) # Iterieren: for zeile in "${ZEILEN[@]}"; do echo "Verarbeite: $zeile" done

Kapitel 5 — Funktionen: Aufbau, Parameter, Rückgabe, Scope

Funktionen sind benannte Code-Blöcke, die mehrfach aufgerufen werden können. Sie machen Skripte lesbarer, wartbarer und testbar. In Bash unterscheiden sie sich in einem wichtigen Punkt von anderen Sprachen: Rückgabewerte funktionieren anders.

5.1 — Syntax und Aufruf

Es gibt zwei äquivalente Schreibweisen. Die Bash-Stil-Variante ohne das Schlüsselwort function ist die bevorzugte. Beim Aufruf werden keine Klammern geschrieben:

# Bash-Stil (bevorzugt): meine_funktion() { echo "Hallo aus der Funktion" } # POSIX-Stil: function meine_funktion { echo "Hallo" } # Aufruf — kein () beim Aufruf! meine_funktion # Mit Argumenten: meine_funktion "arg1" "arg2" # Argumente innerhalb der Funktion: meine_funktion() { echo "$1" # erstes Argument echo "$2" # zweites Argument echo "$#" # Anzahl der Argumente echo "$@" # alle Argumente }

5.2 — Rückgabe-Werte

Das ist ein wichtiger Unterschied zu anderen Sprachen. return gibt in Bash nur einen Exit-Code zurück (0-255), keinen Wert. Um echte Werte zurückzugeben, gibt die Funktion Text über echo aus und wird mit $() aufgerufen:

# Exit-Code als Rückgabe (true/false): ist_gerade() { (( $1 % 2 == 0 )) # Arithmetik: 0 wenn wahr, 1 wenn falsch } if ist_gerade 4; then echo "Gerade"; fi # → Gerade if ist_gerade 3; then echo "Gerade"; fi # → (nichts) # Wert zurückgeben über echo + $(): get_brightness() { brightnessctl -m | cut -d',' -f4 | tr -d '%' } WERT=$(get_brightness) echo "Helligkeit: $WERT%" # Praxisbeispiel: Icons anhand von Zahlenwerten auswählen get_icon() { local v="$1" if (( v < 34 )); then echo "dunkel" elif (( v < 67 )); then echo "mittel" else echo "hell" fi } current=$(get_value) icon=$(get_icon "$current")

5.3 — local — Scope und warum er wichtig ist

Ohne local sind alle Variablen in einer Funktion global — das ist die häufigste versteckte Fehlerquelle in Shell-Skripten. Eine Funktion kann ohne Absicht globale Variablen überschreiben:

WERT="global" setze_wert() { WERT="lokal" # überschreibt global! } setze_wert echo "$WERT" # → lokal (unerwartetes Verhalten!) # Richtig: mit local setze_wert_richtig() { local WERT="lokal" # bleibt in der Funktion } setze_wert_richtig echo "$WERT" # → global (korrekt) # Faustregel: JEDE Variable am Anfang einer Funktion mit local deklarieren meine_funktion() { local val="$1" local ergebnis ergebnis=$(some_command "$val") echo "$ergebnis" }

5.4 — Rekursion

Bash unterstützt Rekursion, aber sie sollte sparsam eingesetzt werden — Bash startet für jeden Aufruf einen neuen Kontext, was bei tiefer Rekursion langsam wird:

countdown() { local n="$1" if (( n <= 0 )); then echo "Start!" return fi echo "$n..." countdown $(( n - 1 )) } countdown 5 # → 5... 4... 3... 2... 1... Start!

5.5 — Funktionen als Bibliotheken: source

Größere Projekte lagern wiederverwendbare Funktionen in separate Dateien aus. Mit source (oder dem gleichbedeutenden .) werden diese Dateien eingebunden, als wären sie Teil des Hauptskripts:

# In lib/utils.sh: log_info() { echo "[INFO] $*"; } log_error() { echo "[ERROR] $*" >&2; } # Im Haupt-Skript: source "$SCRIPT_DIR/lib/utils.sh" # oder equivalent: . "$SCRIPT_DIR/lib/utils.sh" log_info "Skript gestartet" log_error "Etwas ist schiefgelaufen"

Kapitel 6 — Pipes, Umleitungen und Streams

Das Herzstück der Unix-Philosophie ist die Fähigkeit, kleine, spezialisierte Programme miteinander zu verbinden. Pipes und Umleitungen sind der Mechanismus dahinter. Das Verstehen der drei Standard-Streams ist grundlegend für alles, was folgt.

6.1 — Die drei Standard-Streams

Jeder Prozess in Linux hat drei Standard-Datenkanäle, identifiziert durch Datei-Deskriptoren:

0 = stdin # Eingabe (normalerweise Tastatur) 1 = stdout # Ausgabe (normalerweise Terminal) 2 = stderr # Fehlerausgabe (normalerweise Terminal) echo "Hallo" # → stdout echo "Fehler" >&2 # → stderr — wichtig für Fehlermeldungen! read -r EINGABE # liest aus stdin

Fehlermeldungen gehören auf stderr, damit sie von normaler Ausgabe getrennt werden können. Das ist eine Unix-Konvention, die man einhalten sollte.

6.2 — Umleitung

Umleitungen verändern, wohin oder woher die Streams fließen. Die wichtigsten:

> datei.txt # stdout in Datei schreiben (Datei wird überschrieben) >> datei.txt # stdout anhängen < datei.txt # stdin aus Datei lesen 2> fehler.txt # stderr in Datei umleiten 2>&1 # stderr auf stdout umlenken (beide zusammenführen) &> datei.txt # stdout + stderr in Datei (Kurzform für > datei &>/dev/null) &>/dev/null # alles verwerfen — der "Mülleimer" : > datei.txt # Datei leeren (: ist no-op mit Exit-Code 0) # Praxisbeispiel: Aktion ausführen, Ausgabe komplett unterdrücken hyprctl dispatch fullscreen 1 &>/dev/null # Nur stderr unterdrücken, stdout bleibt sichtbar: git push origin HEAD 2>/dev/null

6.3 — Pipes (|)

Eine Pipe verbindet den stdout des linken Befehls mit dem stdin des rechten. Man kann beliebig viele Befehle verketten. Das ist die Unix-Philosophie in ihrer reinsten Form: viele kleine Tools, jedes macht eine Sache gut:

befehl1 | befehl2 | befehl3 # Praxisbeispiel: Bildschirmhöhe aus hyprctl extrahieren SCREEN_H=$(hyprctl monitors | grep -A 12 "focused: yes" | grep "res" | \ awk '{print $2}' | cut -d'x' -f2 | cut -d'@' -f1) # Schritt für Schritt: # 1. hyprctl monitors → alle Monitor-Infos # 2. grep -A 12 "focused" → fokussierten Monitor + 12 Folgezeilen # 3. grep "res" → Auflösungszeile # 4. awk '{print $2}' → zweites Wort, z.B. "1920x1080@60" # 5. cut -d'x' -f2 → "1080@60" # 6. cut -d'@' -f1 → "1080"

6.4 — tee: Ausgabe duplizieren

tee schreibt stdin sowohl auf stdout als auch in eine Datei — nützlich wenn man Ausgabe gleichzeitig im Terminal sehen und in eine Logdatei schreiben möchte:

befehl | tee protokoll.txt | weiterer_befehl befehl | tee -a protokoll.txt # -a für Append statt Überschreiben

6.5 — Here-Strings und Here-Documents

Here-Strings und Here-Documents erlauben es, Text direkt als stdin-Eingabe im Skript zu schreiben, ohne eine Datei anlegen zu müssen:

# Here-String: kurzer Input an einen Befehl: grep "Wort" <<< "Dieser Satz hat das gesuchte Wort" # Here-Document: mehrzeiliger Input (EOF ist frei wählbar): cat << EOF Zeile 1 Variablen werden expandiert: $USER EOF # Mit einfachen Anführungszeichen um EOF: keine Expansion: cat << 'EOF' Dies ist $wirklich literal — keine Expansion EOF # Mit - vor EOF: führende Tabs werden ignoriert: cat <<- EOF Diese Tabs werden entfernt EOF

6.6 — Named Pipes (FIFOs)

Eine Named Pipe ist eine spezielle Datei, die wie eine Pipe wirkt, aber zwischen nicht verwandten Prozessen genutzt werden kann. Sie wird mit mkfifo erstellt:

mkfifo /tmp/meine_pipe cat /tmp/meine_pipe & # Liest aus der Pipe (blockiert im Hintergrund) echo "Hallo" > /tmp/meine_pipe # Schreibt in die Pipe rm /tmp/meine_pipe # Aufräumen

Kapitel 7 — Command Substitution und Process Substitution

Command Substitution und Process Substitution sind zwei der mächtigsten Features von Bash. Sie erlauben es, Befehlsausgaben als Variablen oder als Pseudodateien zu behandeln.

7.1 — Command Substitution $()

$() führt einen Befehl aus und ersetzt den Ausdruck durch die Ausgabe des Befehls. Das ist die moderne Syntax — die alte Backtick-Syntax `` `befehl` `` ist zu vermeiden, da sie keine Verschachtelung erlaubt:

ERGEBNIS=$(befehl) # modern, bevorzugt ERGEBNIS=`befehl` # alt, vermeiden # Verschachtelung geht nur mit $(): TIEF=$(echo $(date +%Y)) # Wichtig: Zeilenumbrüche werden entfernt! ZEILEN=$(cat datei.txt) # "Zeile1\nZeile2" wird zu "Zeile1 Zeile2" in der Variable. # Für Zeilen: mapfile oder IFS-Loop nutzen (Kapitel 8, 9). # Praxisbeispiel: SCREEN_H=$(hyprctl monitors -j | jq -r '.[] | select(.focused == true) | .height') active_addr=$(hyprctl activewindow -j | jq -r '.address // empty')

7.2 — Process Substitution <() — Befehle als Dateien

Process Substitution <() macht aus der Ausgabe eines Befehls eine temporäre Pseudodatei. Das ist echte Bash-Magie — es erlaubt Befehlen, die Dateipfade als Argumente erwarten, mit Befehlsausgaben zu arbeiten:

<(befehl) # Ausgabe des Befehls als Pseudodatei >(befehl) # Eingabe geht in den Befehl als wäre er eine Datei # Beispiel: zwei Befehlsausgaben vergleichen diff <(ls /tmp) <(ls /var) # diff braucht zwei Datei-Argumente — mit <() klappt das ohne Temp-Dateien # mapfile mit Process Substitution: mapfile -t PAKETE < <(apt list --installed 2>/dev/null | grep -v "Listing") # Das < ist Umleitung, das <() ist Process Substitution — zusammen: # "lies die Ausgabe des Befehls als wäre es eine Datei"

7.3 — Das Subshell-Problem und die Lösung

Dieser Unterschied ist kritisch für jeden, der Daten aus Schleifen braucht: Wenn eine Schleife über eine Pipe läuft, läuft sie in einer Subshell — Variablen, die dort gesetzt werden, sind danach verloren:

# PROBLEM: Schleife in Subshell, Zähler geht verloren! ZAEHLER=0 ls | while read -r f; do (( ZAEHLER++ )) done echo "$ZAEHLER" # → 0! (falsch) # LÖSUNG: Process Substitution — Schleife bleibt in aktueller Shell: ZAEHLER=0 while read -r f; do (( ZAEHLER++ )) done < <(ls) echo "$ZAEHLER" # → korrekte Zahl! # Praxisbeispiel aus app-menu.sh: while IFS= read -r desktop; do grep -qE '^(NoDisplay|Hidden)=true' "$desktop" && continue name=$(grep -m1 '^Name=' "$desktop" | cut -d= -f2-) # ... Verarbeitung done < <(find "$dir" -name "*.desktop" -type f 2>/dev/null)

Kapitel 8 — Regex, grep, sed, awk und String-Operationen

Textverarbeitung ist das, wofür Shell-Scripting gemacht wurde. Bash selbst, grep, sed und awk bilden zusammen ein vollständiges Toolkit, das praktisch jede Textmanipulation ermöglicht — ohne dass man eine andere Programmiersprache benötigt.

8.1 — Bash-interne Regex mit =~

Innerhalb von [[ ]] kann man mit =~ direkt gegen Extended Regular Expressions testen. Capture-Gruppen landen in BASH_REMATCH. Wichtig: der Regex darf keine Anführungszeichen haben, sonst wird er literal genommen:

[[ "$STRING" =~ REGEX ]] # Beispiel: IP-Adresse validieren IP="192.168.1.100" [[ "$IP" =~ ^[0-9]+\.[0-9]+\.[0-9]+\.[0-9]+$ ]] && echo "Gültige IP" # Capture-Gruppen in BASH_REMATCH: [[ "2026-06-12" =~ ^([0-9]{4})-([0-9]{2})-([0-9]{2})$ ]] echo "${BASH_REMATCH[0]}" # → 2026-06-12 (ganzer Match) echo "${BASH_REMATCH[1]}" # → 2026 (erste Gruppe) echo "${BASH_REMATCH[2]}" # → 06 (zweite Gruppe) echo "${BASH_REMATCH[3]}" # → 12 (dritte Gruppe) # Praxisbeispiel: Mausposition extrahieren # hyprctl cursorpos gibt z.B. "1234, 567" aus if [[ "$pos" =~ ^[0-9]+,\ ([0-9]+)$ ]]; then MOUSE_Y="${BASH_REMATCH[1]}" fi # Regex-Variable (für komplexe Muster): REGEX="^[0-9]+$" [[ "$s" =~ $REGEX ]] # ohne Anführungszeichen um $REGEX!

8.2 — grep: Zeilen filtern

grep durchsucht Text nach Mustern und gibt passende Zeilen aus. Es ist das meistgenutzte Filter-Tool in Shell-Pipelines:

grep "muster" datei.txt # Zeilen mit "muster" grep -i "muster" datei.txt # case-insensitive grep -v "muster" datei.txt # invertiert (Zeilen OHNE muster) grep -n "muster" datei.txt # mit Zeilennummer grep -c "muster" datei.txt # zählt Treffer-Zeilen grep -l "muster" *.txt # nur Dateinamen ausgeben grep -r "muster" verzeichnis/ # rekursiv suchen grep -E "regex+" datei.txt # Extended Regex (+ ? | () ) grep -P "regex\d" datei.txt # Perl-Regex grep -q "muster" datei.txt # quiet, nur Exit-Code (0=gefunden) grep -m1 "muster" datei.txt # nur ersten Treffer grep -A 5 "muster" datei.txt # 5 Zeilen nach dem Treffer grep -B 3 "muster" datei.txt # 3 Zeilen vor dem Treffer # Mehrere Muster: grep -E "foo|bar" datei.txt grep -e "foo" -e "bar" datei.txt # äquivalent # Praxisbeispiel: .desktop-Dateien ohne Anzeige filtern grep -qE '^(NoDisplay|Hidden)=true' "$desktop" && continue # Namen aus .desktop-Datei extrahieren: name=$(grep -m1 '^Name=' "$desktop" | cut -d= -f2-)

8.3 — sed: Stream Editor

sed verarbeitet Text Zeile für Zeile und führt Transformationen durch. Die häufigste Operation ist Suchen-und-Ersetzen mit s/alt/neu/:

# Einfaches Ersetzen (erstes Vorkommen): echo "Hallo Welt" | sed 's/Welt/Shell/' # → Hallo Shell # Global (alle Vorkommen): echo "a a a" | sed 's/a/b/g' # → b b b # Case-insensitive: echo "Hallo HALLO" | sed 's/hallo/Hi/gI' # → Hi Hi # Extended Regex mit -E: echo "2026-06-12" | sed -E 's/([0-9]+)-([0-9]+)-([0-9]+)/\3.\2.\1/' # → 12.06.2026 (Datum umdrehen mit Capture-Groups) # Zeilen löschen: sed '/muster/d' datei.txt # Zeilen mit "muster" entfernen # Bestimmte Zeilen ausgeben: sed -n '5,10p' datei.txt # nur Zeilen 5-10 # In-place Bearbeitung (Datei direkt ändern): sed -i 's/alt/neu/g' datei.txt sed -i.bak 's/alt/neu/g' datei.txt # mit Backup # Mehrere Operationen in einem sed-Aufruf (durch ; getrennt): sed 's/%[a-zA-Z]//g;s/^[[:space:]]*//;s/[[:space:]]*$//' datei.txt # 1. Desktop-Placeholders entfernen (%f, %u, %F etc.) # 2. führende Leerzeichen entfernen # 3. abschließende Leerzeichen entfernen

8.4 — awk: Spaltenverarbeitung und Programmierung

awk ist eine vollständige Programmiersprache für Textverarbeitung. Es verarbeitet Zeilen und kennt Spalten (Fields). Es ist besonders stark für tabellarische Daten und Berechnungen:

# Grundstruktur: awk 'MUSTER { AKTION }' datei.txt # Spalten ausgeben ($1=erste, $2=zweite, $NF=letzte, $0=ganze Zeile): echo "Hallo Welt Bash" | awk '{print $2}' # → Welt echo "Hallo Welt Bash" | awk '{print $NF}' # → Bash # Mit Trennzeichen (-F): echo "user:1000:gruppe" | awk -F: '{print $1}' # → user echo "user:1000:gruppe" | awk -F: '{print $3}' # → gruppe # Berechnung: echo "10 20 30" | awk '{print $1 + $2 + $3}' # → 60 # Bedingungen: ps aux | awk '$3 > 5 {print $1, $11}' # Prozesse mit CPU > 5% # BEGIN und END (vor/nach der Verarbeitung): ps aux | awk 'BEGIN{print "Prozesse:"} NR>1{print $11} END{print "Fertig"}' # NR = aktuelle Zeilennummer (Number of Record) # Summieren: ps aux | awk 'NR>1{cpu+=$3} END{printf "CPU gesamt: %.1f%%\n", cpu}' # Fließkomma-Berechnung: awk "BEGIN {print $val/100}"

8.5 — String-Manipulation in Bash (intern)

Bash hat eingebaute String-Operationen — sie sind schneller als externe Tools wie sed oder awk, weil kein neuer Prozess gestartet wird:

STR="Hallo Welt" # Länge: echo "${#STR}" # → 10 # Substring: echo "${STR:6}" # → Welt (ab Position 6) echo "${STR:0:5}" # → Hallo (Position 0, Länge 5) echo "${STR: -4}" # → Welt (von hinten — Leerzeichen wichtig!) # Groß/Kleinschreibung (Bash 4+): echo "${STR,,}" # → hallo welt (alles klein) echo "${STR^^}" # → HALLO WELT (alles groß) echo "${STR,}" # → hallo Welt (nur erster Buchstabe klein) echo "${STR^}" # → Hallo Welt (nur erster Buchstabe groß) # Muster vom Anfang entfernen: DATEI="foto.backup.tar.gz" echo "${DATEI#*.}" # → backup.tar.gz (kürzestes Muster) echo "${DATEI##*.}" # → gz (längstes Muster) # Muster vom Ende entfernen: echo "${DATEI%.*}" # → foto.backup.tar (kürzestes Muster) echo "${DATEI%%.*}" # → foto (längstes Muster) # Ersetzen: echo "${STR/Welt/Shell}" # → Hallo Shell (erstes Vorkommen) echo "${STR//l/L}" # → HaLLo WeLt (alle Vorkommen) echo "${STR/#Hallo/Hi}" # → Hi Welt (nur am Anfang) echo "${STR/%Welt/Du}" # → Hallo Du (nur am Ende) # Praxisbeispiele: local clean_addr="${target_addr#0x}" # "0x" Prefix entfernen event="${line%%>>*}" # alles ab ">>" entfernen data="${line#*>>}" # alles bis ">>" entfernen name="${name//[$'\t\n\0']/ }" # Tabs, Newlines → Leerzeichen

8.6 — Default-Werte und Fehler-Variablen

Bash bietet eine elegante Syntax für Variablen mit Default-Werten. Diese sogenannte Parameter-Expansion ist sehr nützlich für robuste Skripte:

${VAR:-default} # Wert von VAR, oder "default" wenn VAR leer/ungesetzt ${VAR:=default} # wie :-, setzt aber VAR auch auf default ${VAR:+ersatz} # "ersatz" wenn VAR gesetzt, sonst leer ${VAR:?fehlermsg} # Fehler + Meldung wenn VAR leer/ungesetzt # Praxisbeispiele: STATE_FILE="${XDG_RUNTIME_DIR:-/tmp}/rofi-menu-state" # XDG_RUNTIME_DIR ist normalerweise /run/user/1000 # Falls nicht gesetzt: /tmp als Fallback

Kapitel 9 — Fehlerbehandlung, Exit Codes und Debugging

Robuste Skripte fallen nicht stillschweigend auseinander — sie erkennen Fehler, behandeln sie sauber und hinterlassen keine inkonsistenten Zustände. Dieses Kapitel zeigt, wie man Skripte defensiv schreibt.

9.1 — Exit Codes

Jeder Befehl gibt beim Beenden einen Exit-Code zurück. 0 bedeutet Erfolg, alles andere bedeutet Fehler. $? enthält immer den Exit-Code des letzten Befehls. Konventionelle Exit-Code-Bedeutungen:

ls /tmp echo "$?" # → 0 (Erfolg) ls /gibts_nicht echo "$?" # → 2 (Fehler) exit 0 # Skript erfolgreich beenden exit 1 # allgemeiner Fehler exit 2 # falsche Verwendung (Konvention) exit 127 # Befehl nicht gefunden

9.2 — set -e, set -u, set -o pipefail

Diese drei Shell-Optionen machen Skripte deutlich robuster. Sie werden typischerweise ganz am Anfang eines Skripts gesetzt:

set -euo pipefail # Standard für robuste Skripte — alle drei zusammen # Was sie bedeuten: set -e # "exit on error": abbricht bei jedem nicht-0-Exit-Code set -u # "undefined vars": Fehler bei ungesetzten Variablen set -o pipefail # Pipeline schlägt fehl wenn ein Teil scheitert set -x # Debug-Modus: jede Zeile wird vor Ausführung ausgegeben # Problem mit set -e: manchmal will man Fehler ignorieren ls /gibts_nicht || true # || true: verhindert Abbruch grep "nicht_da" datei.txt || true # Oder mit if: if ls /gibts_nicht; then echo "existiert" fi # set -e für einen Block deaktivieren: set +e riskanter_befehl set -e

9.3 — trap: Signal Handler

trap erlaubt es, Code auszuführen wenn bestimmte Events auftreten — z.B. wenn das Skript endet, unterbrochen wird oder ein Signal empfängt. Das ist unverzichtbar für sauberes Aufräumen:

trap 'BEFEHLE' SIGNAL # Wichtige Signale: # EXIT → immer wenn das Skript endet (normal oder Fehler) # ERR → bei jedem Fehler # INT → Ctrl+C (SIGINT) # TERM → kill (SIGTERM) # HUP → Terminal geschlossen (SIGHUP) # USR1 → benutzerdefiniertes Signal 1 # USR2 → benutzerdefiniertes Signal 2 # Cleanup beim Beenden (der wichtigste Anwendungsfall): TMP_FILE=$(mktemp) trap "rm -f '$TMP_FILE'" EXIT # egal wie das Skript endet — die Datei wird immer gelöscht # Mehrere Signale und Cleanup-Funktion: cleanup() { rm -f /tmp/meine_lock_datei echo "Aufgeräumt" >&2 } trap cleanup EXIT INT TERM # Externes Steuern des Skript-Zustands via Signale: HIDDEN=false trap "HIDDEN=false" USR2 # kill -USR2 <PID> setzt HIDDEN von außen auf false

9.4 — Signale senden: kill und pkill

kill sendet Signale an Prozesse. Es beendet nicht zwingend einen Prozess — das hängt davon ab, welches Signal gesendet wird:

kill PID # sendet SIGTERM (15) — freundliche Aufforderung kill -9 PID # SIGKILL — sofort beenden, nicht abfangbar kill -SIGUSR1 PID # benutzerdefiniertes Signal kill -SIGUSR2 PID pkill -SIGUSR1 waybar # Signal an Prozess nach Name pkill -f "$SCRIPT" # Signal per vollem Kommandozeilen-String

9.5 — Debugging-Techniken

Bash bietet mehrere eingebaute Methoden zum Debuggen von Skripten:

# 1. set -x — Trace-Modus (zeigt jede ausgeführte Zeile): set -x VARIABLE="test" echo "$VARIABLE" set +x # Ausgabe: + VARIABLE=test + echo test → test # Nur bestimmten Teil debuggen: bash -x skript.sh # ganzes Skript im Trace-Modus # 2. Eigene Debug-Funktion (ein-/ausschaltbar): DEBUG="${DEBUG:-false}" debug() { [[ "$DEBUG" == "true" ]] && echo "[DEBUG] $*" >&2 } debug "Variable ist: $VARIABLE" # Aufruf: DEBUG=true ./skript.sh # 3. Syntax prüfen ohne Ausführen: bash -n skript.sh # nur Syntax-Check # 4. shellcheck — externer Linter (empfohlen): shellcheck skript.sh # findet häufige Fehler und Antipatterns

Kapitel 10 — Schleifen und Kontrollfluss

Schleifen sind das Werkzeug für Wiederholungen. Bash bietet while, until, for und verschiedene Kontrollbefehle. Jede Schleifenart hat ihre Stärken, und das richtige Werkzeug für die richtige Aufgabe macht den Unterschied.

10.1 — while, until, for

# while: läuft solange Bedingung true ist ZAEHLER=0 while (( ZAEHLER < 5 )); do echo "$ZAEHLER" (( ZAEHLER++ )) done # until: läuft solange Bedingung false ist (umgekehrtes while) until pgrep -x "waybar" > /dev/null; do echo "Warte auf waybar..." sleep 1 done # Endlosschleife: while true; do echo "Läuft..." sleep 1 done # Alternative: while :; do ... done (: ist immer true) # for mit Zahlen-Bereich (Bash-spezifisch): for i in {1..10}; do echo $i; done for i in {0..100..5}; do echo $i; done # in 5er-Schritten # for C-Stil: for (( i=0; i<10; i++ )); do echo "$i" done # for über Glob statt ls (korrekt, weil ls bei Leerzeichen versagt): for datei in *.sh; do echo "Skript: $datei" done

10.2 — read in Schleifen: zeilenweise Verarbeitung

Die häufigste Schleifenform in Shell-Skripten ist das zeilenweise Lesen von Dateien oder Befehlsausgaben. Das richtige Muster nutzt IFS= und -r:

# Datei zeilenweise (IFS= verhindert Whitespace-Trimming, -r kein Backslash-Escaping): while IFS= read -r zeile; do echo "Zeile: $zeile" done < datei.txt # Aus Befehl — mit Process Substitution (Schleife bleibt in aktueller Shell!): while IFS= read -r zeile; do echo "Zeile: $zeile" done < <(find . -name "*.sh") # Mit Trennzeichen (z.B. TSV — Tab-Separated): while IFS=$'\t' read -r name cmd icon; do echo "Name: $name, Cmd: $cmd" done < tsv_datei.txt # Null-Byte getrennt (für Dateinamen mit Leerzeichen — beste Praxis): while IFS= read -r -d '' datei; do echo "Datei: $datei" done < <(find . -name "*.sh" -print0)

10.3 — break, continue, return

# break: Schleife sofort verlassen for i in {1..10}; do [[ "$i" -eq 5 ]] && break echo "$i" done # → 1 2 3 4 # break N: N Ebenen verlassen (bei verschachtelten Schleifen): for i in {1..3}; do for j in {1..3}; do [[ "$i" -eq 2 && "$j" -eq 2 ]] && break 2 done done # continue: nächste Iteration for i in {1..5}; do [[ "$i" -eq 3 ]] && continue echo "$i" done # → 1 2 4 5 # Praxisbeispiel: .desktop-Dateien überspringen die versteckt sind while IFS= read -r desktop; do grep -qE '^NoDisplay=true' "$desktop" && continue # ... verarbeiten done < <(find ...)

10.4 — Parallele Ausführung: & und wait

Mit & werden Befehle im Hintergrund gestartet. Das ist nützlich für Aufgaben, die parallel laufen können. wait wartet auf den Abschluss:

# Im Hintergrund starten: langer_befehl & echo "Gestartet, PID: $!" # Mehrere parallel, dann warten bis alle fertig: befehl1 & befehl2 & befehl3 & wait # Exit-Code eines Hintergrundprozesses: befehl & PID=$! wait $PID echo "Exit: $?" # Praxisbeispiel: Cache im Hintergrund aufbauen damit der Nutzer nicht wartet if ! is_cache_valid; then build_desktop_cache & fi

Kapitel 11 — jq: JSON aus der Shell verarbeiten

Moderne System-Tools geben JSON aus — hyprctl, systemctl --output=json, REST-APIs und viele andere. jq ist der Standard-Tool dafür: eine Mini-Programmiersprache speziell für JSON-Verarbeitung in der Shell.

11.1 — Grundlagen

# Basis-Syntax: echo '{"name":"TrafkHop","version":3}' | jq '.name' # → "TrafkHop" echo '{"name":"TrafkHop","version":3}' | jq '.version' # → 3 echo '{"name":"TrafkHop","version":3}' | jq -r '.name' # → TrafkHop (raw, ohne Anführungszeichen!) # -r ist crucial: ohne -r sind Strings immer mit Anführungszeichen — nicht als Shell-Variable nutzbar

11.2 — Arrays und Objekte

JSON='[{"id":1,"name":"Alpha"},{"id":2,"name":"Beta"}]' echo "$JSON" | jq '.[0]' # → {"id":1,"name":"Alpha"} echo "$JSON" | jq '.[0].name' # → "Alpha" echo "$JSON" | jq '.[] | .name' # → "Alpha" "Beta" (alle) echo "$JSON" | jq -r '.[] | .name' # → Alpha \n Beta (raw) # Länge: echo "$JSON" | jq 'length' # → 2 # Filtern: echo "$JSON" | jq '.[] | select(.id == 2)' # → {"id":2,"name":"Beta"} # Ausgabe als Array: echo "$JSON" | jq '[.[] | .name]' # → ["Alpha","Beta"] # String-Interpolation: echo "$JSON" | jq '.[] | "\(.id): \(.name)"' # → "1: Alpha" "2: Beta"

11.3 — jq in der Praxis

Typische Muster bei der Verwendung von jq mit System-Tools und APIs:

# Hyprland: fokussierten Monitor finden SCREEN_H=$(hyprctl monitors -j | jq -r '.[] | select(.focused == true) | .height') # Aktives Fenster: erst JSON in Variable, dann mehrfach parsen (nur ein hyprctl-Aufruf!) active=$(hyprctl activewindow -j) addr=$(echo "$active" | jq -r '.address') ws=$(echo "$active" | jq -r '.workspace.id') # jq -e: Exit-Code 1 wenn Ergebnis null/false (nützlich für if-Bedingungen) if ! hyprctl clients -j | jq -e ".[] | select(.address == \"$addr\")" >/dev/null; then echo "Fenster nicht mehr vorhanden" fi # endswith() für Adressen-Matching: win_data=$(echo "$clients_json" | jq ".[] | select(.address | endswith(\"$clean_addr\"))") # Null-Fallback: gibt "" zurück wenn .address null ist, statt "null" active_addr=$(hyprctl activewindow -j | jq -r '.address // empty')

11.4 — Fortgeschrittenes jq

# Neues JSON-Objekt erstellen: jq -n '{"status":"ok","data":["a","b"]}' # Shell-Variablen sicher in jq übergeben (verhindert Injection): NAME="TrafkHop" jq -n --arg name "$NAME" '{"name":$name}' # Zwei JSON-Dateien mergen: jq -s '.[0] * .[1]' datei1.json datei2.json # Alle Keys eines Objekts: echo '{"a":1,"b":2}' | jq 'keys' # → ["a","b"] # Bedingte Logik in jq: echo '{"val":5}' | jq '.val | if . > 3 then "groß" else "klein" end' # Rekursiv alle Strings suchen: echo '{"a":{"b":{"c":"ziel"}}}' | jq '.. | strings' # Komplexes Beispiel: verschachteltes JSON traversieren echo "$MENU_JSON" | jq -r --arg p "$path" ' (if $p == "." then . else getpath($p | split(".")) end) | to_entries[] | "\(.key)\t\(if (.value|type)=="object" then "dir" else "cmd" end)" '

Kapitel 12 — Subshells, Prozesse und Jobverwaltung

Bash ist ein vollständiger Prozess-Manager. Das Verstehen von Subshells, Hintergrundprozessen und Jobverwaltung ist entscheidend für fortgeschrittene Skripte, die mehrere Aufgaben gleichzeitig koordinieren.

12.1 — Was ist eine Subshell?

Wenn Bash einen Befehl in runden Klammern ausführt, startet es eine Subshell — eine Kopie der aktuellen Shell. Variablen, die in der Subshell gesetzt werden, sind in der Eltern-Shell nicht sichtbar:

( LOKAL="nur in Subshell" echo "$LOKAL" # → "nur in Subshell" ) echo "$LOKAL" # → leer # Wann passiert das implizit? # - Bei jeder Pipe: cmd1 | cmd2 (cmd2 in Subshell) # - Bei $() (technisch fork+exec) # - Bei Hintergrundprozessen (&) # - Bei <() und >() (Process Substitution) # Nützlich: Subshell für Isolation (Verzeichniswechsel ohne Auswirkung): ( cd /tmp && ls ) # Aktuelles Verzeichnis im Eltern-Prozess unverändert

12.2 — setsid: vom Terminal entkoppeln

setsid erstellt eine neue Session und trennt den Prozess vollständig vom Terminal. Das ist wichtig für GUI-Anwendungen, die von einem Skript gestartet werden und auch dann laufen sollen, wenn das Skript oder Terminal geschlossen wird:

# Normal: stirbt wenn das Skript endet firefox & # Mit setsid: läuft weiter, komplett unabhängig setsid firefox & # disown ist eine Alternative (Bash "vergisst" den Job): firefox & disown # Praxisbeispiel: Anwendung aus einem Launcher starten setsid bash -c "$value" >/dev/null 2>&1 &

Kapitel 13 — Unix Sockets, socat und IPC

Inter-Process Communication (IPC) ist die Fähigkeit von Prozessen, miteinander zu kommunizieren. Unix Domain Sockets sind der Standard dafür auf Linux — schneller als TCP, sicherer weil dateisystembasiert, und von praktisch allen System-Daemons verwendet.

13.1 — Was sind Unix Domain Sockets?

Unix Sockets sind spezielle Dateien (Typ 's' in ls -la) die als Kommunikations-Endpunkte fungieren. Sie sind bidirektional und für komplexere Kommunikation als Named Pipes geeignet. Viele System-Programme kommunizieren ausschließlich darüber: Hyprland, systemd, dbus, X11, Wayland, Docker, Podman:

ls -la /tmp/*.sock # Socket-Dateien anzeigen ls -la /run/user/1000/hypr/*/ # Hyprland-Sockets

13.2 — socat: SOcket CAT

socat verbindet zwei beliebige Datenströme miteinander — Stdin/Stdout, Dateien, TCP, UDP, Unix Sockets und mehr. Es ist das universelle Werkzeug für Socket-Kommunikation:

# Unix Socket lesen: socat - UNIX-CONNECT:/pfad/zu/socket.sock # Als Empfänger (Server-Seite): socat UNIX-LISTEN:/tmp/test.sock - # Mit TCP: socat TCP:example.com:80 - # Hyprland Events empfangen: socat -U - "UNIX-CONNECT:$XDG_RUNTIME_DIR/hypr/$HYPRLAND_INSTANCE_SIGNATURE/.socket2.sock" # -U: UNIX-Domain Socket # Format der Events: "eventname>>daten"

13.3 — Hyprland IPC

Hyprland (der Wayland-Compositor auf TrafkTux) hat eine Socket-basierte API. socket.sock empfängt Befehle, socket2.sock sendet Events:

# Alle Events ansehen (dann Fenster bewegen, klicken etc.): socat -U - "UNIX-CONNECT:$XDG_RUNTIME_DIR/hypr/$HYPRLAND_INSTANCE_SIGNATURE/.socket2.sock" # Beispiel-Event-Zeilen: # "openwindow>>addr,1,workspace.name,firefox" # "activewindowv2>>addr" # "workspace>>name" # Event-Parsing: event="${line%%>>*}" # alles vor ">>" data="${line#*>>}" # alles nach ">>"

13.4 — Event-Driven Architektur in Shell

Ein Event-Driven Daemon öffnet eine permanente Socket-Verbindung, liest Events in einer Endlosschleife und reagiert auf relevante Events. Das ist das Shell-Äquivalent von EventEmitter oder addEventListener:

socat -U - "UNIX-CONNECT:$SOCKET" | while read -r event_line; do event="${event_line%%>>*}" data="${event_line#*>>}" case "$event" in openwindow) # Kleines Delay damit Hyprland den Zustand aktualisiert hat, # dann in Subshell im Hintergrund — Event-Schleife blockiert nicht! ( sleep 0.08; handle_open "$data" ) & ;; closewindow) handle_close "$data" & ;; workspace|workspacev2) handle_ws "$data" & ;; *) ;; # unbekannte Events ignorieren esac done

Kapitel 14 — Rofi-Integration und Skript-Architekturen

Rofi ist ein Launcher und Menü-Tool für Linux. Es kann externe Skripte als dynamische Datenquellen nutzen — ein elegantes IPC-System über stdout/stdin und Environment-Variablen. Dieses Kapitel zeigt die Architektur dahinter am konkreten Beispiel.

14.1 — Wie Rofi mit Skripten kommuniziert

Rofi ruft das Skript mehrfach auf und kommuniziert über Environment-Variablen:

rofi -show menu -modi "menu:/pfad/zum/skript.sh" # Rofi ruft das Skript so auf: # 1. Erster Aufruf (ROFI_RETV=0): Skript gibt Einträge aus. # 2. Bei Auswahl (ROFI_RETV=1): Skript kriegt die Selektion als $1. # 3. Bei Auswahl+Enter (ROFI_RETV=2): alternative Aktion. # 4. Bei Hotkey (ROFI_RETV=10, 11, ...): Hotkey-Aktionen. # $ROFI_RETV → 0=erster Aufruf, 1=Auswahl, 2=Auswahl+Enter # $ROFI_INFO → Info-String des ausgewählten Eintrags

14.2 — Das Rofi-Ausgabeformat

Rofi versteht ein spezielles Format für Einträge mit Metadaten. Das Null-Byte trennt Anzeigetext von Metadaten, ASCII 31 trennt Key und Value:

# Format: "Anzeigetext\0icon\x1fICON_NAME\x1finfo\x1fMEIN_INFO\n" # \0 = Null-Byte: trennt Anzeigetext von Metadaten # \x1f = ASCII 31 (Unit Separator): trennt Key und Value # \n = neue Zeile = neuer Eintrag # Eintrag mit Icon und Info: printf "← Zurück\0icon\x1fgo-previous\x1finfo\x1fback:\n" # Dynamisch mit printf %s: printf "%s\0icon\x1f%s\x1finfo\x1fcmd:%s\n" "$name" "$icon" "$exec_cmd" # Steueranweisungen an Rofi (kein Eintrag, nur Rofi-Konfiguration): printf "\0prompt\x1fNeuer Prompt\n" # Prompt-Text ändern printf "\0no-custom\x1ffalse\n" # Freie Eingabe erlauben printf "\0keep-selection\x1ftrue\n" # Auswahl behalten

14.3 — State-Management zwischen Aufrufen

Da Rofi das Skript neu aufruft für jeden Schritt, muss der Zustand (welcher Ordner bin ich gerade?) irgendwo gespeichert werden. Die Lösung ist eine State-Datei in XDG_RUNTIME_DIR:

STATE_FILE="${XDG_RUNTIME_DIR:-/tmp}/rofi-menu-state" get_path() { [[ -f "$STATE_FILE" ]] && cat "$STATE_FILE" || echo "."; } set_path() { echo "$1" > "$STATE_FILE"; } # Navigation: case "$type" in dir) set_path "$value" ;; back) set_path "${path%.*}" ;; # ein Level hoch cmd) setsid bash -c "$value" >/dev/null 2>&1 &; exit 0 ;; quit) rm -f "$STATE_FILE"; exit 0 ;; esac # Entry-Point — unterscheidet ob Rofi oder Nutzer aufgerufen hat: if [[ -n "$ROFI_RETV" ]]; then run_mode "$1" # Rofi hat das Skript aufgerufen else rm -f "$STATE_FILE" rofi -show menu -modi "menu:$SCRIPT_PATH" ... fi

14.4 — Cache-Strategien

Teure Operationen (z.B. hunderte .desktop-Dateien lesen) sollen nicht bei jedem Aufruf wiederholt werden. Eine Cache-Datei mit Ablaufzeit löst das Problem:

CACHE_FILE="${XDG_RUNTIME_DIR:-/tmp}/rofi-desktop-cache" CACHE_MAX_AGE=120 # Sekunden is_cache_valid() { [[ -f "$CACHE_FILE" ]] || return 1 local now mod age now=$(date +%s) # aktuelle Zeit als Unix-Timestamp mod=$(stat -c %Y "$CACHE_FILE" 2>/dev/null) || return 1 # Datei-Änderungszeit age=$(( now - mod )) # Alter in Sekunden (( age < CACHE_MAX_AGE )) # true wenn jung genug } # Cache im Hintergrund auffrischen wenn nötig — Nutzer wartet nicht: if [[ -z "$ROFI_RETV" ]] && ! is_cache_valid; then build_desktop_cache & fi

Kapitel 15 — Fortgeschrittene Patterns, Architektur und Professionelles Scripting

Die letzten Kapitel haben die Werkzeuge gezeigt. Dieses Kapitel zeigt, wie man sie zu guten Skripten zusammenfügt — Skripte, die robust, wartbar und professionell sind.

15.1 — Das Dispatcher/Handler Pattern

Viele Shell-Skripte folgen einem klaren Muster: Eingabe entgegennehmen, entscheiden was zu tun ist, Aktion ausführen. Dieses Pattern ist die Grundlage für wartbare Skripte:

# Eingabe: Event oder Argument # Dispatch: case-Statement wählt den Handler # Handler: Funktion führt die Aktion aus # Aus einem Daemon: case "$event" in openwindow) handler_open "$data" ;; closewindow) handler_close "$data" ;; workspace) handler_ws "$data" ;; *) ;; esac # Aus einem Power-Menü: case "$chosen" in Shutdown) systemctl poweroff ;; Reboot) systemctl reboot ;; Suspend) systemctl suspend ;; Lock) hyprlock ;; Logout) hyprctl dispatch exit ;; *) exit 0 ;; esac

15.2 — Idempotenz: Skripte die man beliebig oft aufrufen kann

Ein idempotentes Skript hat beim zweiten Aufruf dieselbe Wirkung wie beim ersten. Das ist besonders wichtig für Setup- und Konfigurations-Skripte:

# Nicht idempotent → jeder Aufruf fügt eine Zeile hinzu: echo "Zeile" >> datei.txt # Idempotent → nur hinzufügen wenn noch nicht vorhanden: grep -qF "Zeile" datei.txt || echo "Zeile" >> datei.txt # mkdir -p: kein Fehler wenn schon vorhanden mkdir -p neuer_ordner # ln -sf: überschreibt bestehenden Symlink ln -sf quelle ziel # -f macht das idempotent

15.3 — Lock-Files: Race Conditions vermeiden

Wenn ein Skript mehrfach parallel gestartet werden kann (z.B. durch Hotkeys), braucht man manchmal Mutex-Mechanismen. flock ist die atomare Lösung:

LOCK_FILE="/tmp/mein_skript.lock" # Einfaches Lock (hat eine Race-Condition bei sehr gleichzeitigem Start): if [[ -f "$LOCK_FILE" ]]; then echo "Läuft bereits"; exit 1 fi touch "$LOCK_FILE" trap "rm -f '$LOCK_FILE'" EXIT # Atomares Lock mit flock (keine Race-Condition — die richtige Lösung): exec 9>"$LOCK_FILE" flock -n 9 || { echo "Läuft bereits"; exit 1; } # flock öffnet und sperrt die Datei in einem atomaren Schritt

15.4 — Config-Dateien lesen und schreiben

Konfigurationsdateien im INI-Stil oder JSON sind in Shell-Skripten eine häufige Anforderung:

# INI-Stil lesen (host=localhost, port=8080): while IFS='=' read -r key value; do [[ "$key" =~ ^[[:space:]]*# ]] && continue # Kommentare überspringen [[ -z "$key" ]] && continue # Leerzeilen überspringen key="${key// /}" # Leerzeichen entfernen value="${value//\"}" # Anführungszeichen entfernen declare -g "CONFIG_${key^^}=$value" # Als Variable speichern done < "$CONFIG_FILE" echo "$CONFIG_HOST" # → was in config stand: host=localhost # JSON-Config mit jq: CONFIG=$(cat "$CONFIG_FILE") HOST=$(echo "$CONFIG" | jq -r '.host') PORT=$(echo "$CONFIG" | jq -r '.port // 8080') # mit Default-Wert

15.5 — Testen von Shell-Skripten

Kein Standard-Test-Framework existiert für Bash, aber es gibt bewährte Muster für Unit-Tests:

# Einfache Assert-Funktion: assert() { local erwartet="$1" local tatsaechlich="$2" local beschreibung="$3" if [[ "$erwartet" == "$tatsaechlich" ]]; then echo "[OK] $beschreibung" else echo "[FAIL] $beschreibung" echo " Erwartet: $erwartet" echo " Tatsaechlich: $tatsaechlich" return 1 fi } # Funktion importieren und testen: source ./brightness-utils.sh result=$(get_icon 20) assert "dunkel" "$result" "get_icon unter 34 gibt dunkles Icon" result=$(get_icon 50) assert "mittel" "$result" "get_icon 34-66 gibt mittleres Icon" # BATS (Bash Automated Testing System) ist ein echtes Test-Framework: # github.com/bats-core/bats-core — für größere Projekte empfohlen

15.6 — Performance-Optimierung

Shell-Skripte sind langsam für intensive Berechnungen, aber schnell für Prozess-Orchestrierung. Diese Muster vermeiden die häufigsten Performance-Fallen:

# Langsam: externe Tools in Schleifen — 1000 Prozesse! for i in {1..1000}; do echo "$(date)" done # Schnell: einmal aufrufen, dann wiederverwenden DATE=$(date) for i in {1..1000}; do echo "$DATE" done # Langsam: unnötiges cat + pipe (Useless Use of Cat) cat datei.txt | grep "muster" # Schnell: direkte Umleitung grep "muster" datei.txt # Bash-intern statt externe Tools (schneller, kein Prozess-Start): # Statt: echo "$STR" | tr '[:upper:]' '[:lower:]' echo "${STR,,}" # bash-interne Kleinschreibung # Teure Operationen cachen (einmal ausführen, mehrfach nutzen): clients_json=$(hyprctl clients -j) # einmal abrufen for addr in ...; do echo "$clients_json" | jq ... # JSON wiederverwenden done
