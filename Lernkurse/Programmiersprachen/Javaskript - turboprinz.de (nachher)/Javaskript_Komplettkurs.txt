================================================================================
MODUL 1: JavaScript und der Browser — wie alles zusammenhängt
================================================================================

Bevor wir in Syntax einsteigen, ein Schritt zurück: WO läuft JavaScript
überhaupt, und WARUM nennt man es "die Sprache des Web"? Das hilft dir,
jede einzelne Regel später richtig einzuordnen — und erklärt, warum dein
bestehender Code so aussieht, wie er aussieht.

------------------------------------------------------------------------
1.1  Die drei Schichten einer Website
------------------------------------------------------------------------

Aus dem HTML- und CSS-Kurs kennst du das Prinzip schon, aber zur
Erinnerung — für JS ist es zentral:

  HTML  → STRUKTUR/INHALT   "Was gibt es?"          (das Skelett)
  CSS   → DARSTELLUNG       "Wie sieht es aus?"     (die Kleidung)
  JS    → VERHALTEN         "Was passiert, wann?"   (Muskeln + Nerven)

Alles, was sich auf deiner Seite BEWEGT, REAGIERT, NACHLÄDT oder RECHNET,
ist JavaScript. Konkret in TrafkSite:

  - Die Navigation in #header-nav wird zur Laufzeit aus sitemap.xml
    aufgebaut (nav.js) — ohne JS: ein leeres <nav>-Tag, sonst nichts.
  - Die animierten "Bubbles" hinter den Nav-Punkten (placeholder.js) —
    ohne JS: gar nichts da, nicht mal ein Platzhalter.
  - Der Lautstärkeregler merkt sich deine Einstellung über Seiten-
    aufrufe hinweg (volume.js) — ohne JS: der Regler würde bei jedem
    Neuladen wieder auf 0.5 zurückspringen.
  - Der Alfonz-Chat (AskAlfonz.js) — ohne JS: keine Eingabe möglich,
    kein einziger Request würde abgeschickt.

Streiche gedanklich alle drei <script>-Tags aus index.html — was bliebe?
Eine völlig statische Seite: Logo, Projektliste, Mitgliederliste. ALLES
andere — Header, Footer, Navigation, Lautstärke-Merkfunktion, Chat —
kommt aus JavaScript. Genau DAS macht JS zur "Sprache des Web": Es ist die
einzige Programmiersprache, die JEDER Browser von Haus aus versteht — ganz
ohne Installation, ohne Compiler, ohne irgendetwas. Eine .html-Datei mit
einem <script>-Tag reicht.

------------------------------------------------------------------------
1.2  Wie der Browser deine Seite aufbaut — die Ladereihenfolge
------------------------------------------------------------------------

Dieser Ablauf erklärt, WARUM Skripte da stehen, wo sie stehen — und warum
manche Dinge "null" sind, wenn man zu früh danach fragt.

  1. Der Browser bekommt index.html und liest sie von OBEN nach UNTEN.
  2. Für jedes HTML-Element baut er das DOM (Document Object Model) auf —
     einen Baum aus JS-Objekten, der deine Seite repräsentiert. Jedes
     <div>, <p>, <img> wird zu einem JS-Objekt, mit dem du arbeiten kannst.
  3. Trifft er auf <link rel="stylesheet">, lädt er die CSS-Datei
     parallel und baut das CSSOM (CSS Object Model) auf.
  4. Trifft er auf <script> (ohne defer/async), PASSIERT etwas Wichtiges:
     Das HTML-Parsen PAUSIERT, der Browser lädt das Skript (falls extern)
     und FÜHRT ES SOFORT AUS. Erst danach geht das Parsen weiter.
  5. Sobald das gesamte HTML geparst ist (DOM komplett, Bilder evtl. noch
     nicht): Event "DOMContentLoaded" wird ausgelöst.
  6. Sobald wirklich ALLES geladen ist (auch Bilder, Videos, Schriften):
     Event "load" wird ausgelöst.

Punkt 4 ist der Grund, warum in deinem index.html die Skripte GANZ AM
ENDE von <body> stehen, kurz vor </body>:

    <script src="files/placeholder.js"></script>
    <script src="files/nav.js"></script>
    <script src="files/volume.js"></script>
    </body>

Wenn placeholder.js dort z.B. document.getElementById('header-placeholder')
aufruft, GIBT ES DIESES ELEMENT BEREITS — der Browser hat den ganzen
<body> schon geparst, bevor er bei diesen <script>-Tags ankommt.

Würdest du DASSELBE Skript stattdessen oben in den <head> packen (ohne
defer), wäre #header-placeholder zu diesem Zeitpunkt noch GAR NICHT im
DOM — document.getElementById('header-placeholder') würde null
zurückgeben, und der nächste Zugriff (z.B. .innerHTML = ...) würde mit
"Cannot set properties of null" abstürzen.

HÄUFIGER FEHLER: "Mein Skript findet mein Element nicht!"
  → In 90% der Fälle: Das Skript läuft, BEVOR das Element im DOM
    existiert. Lösung: Skript ans Ende von <body> verschieben, ODER den
    Code in einen DOMContentLoaded-Listener packen (siehe Modul 5).

------------------------------------------------------------------------
1.3  <script>-Tags — alle Spielarten
------------------------------------------------------------------------

  // Inline — Code direkt im HTML (für kleine Sachen ok, sonst meiden):
  <script>
    console.log("Hallo aus index.html!");
  </script>

  // Extern — sauberer, vom Browser cachebar, wiederverwendbar:
  <script src="files/volume.js"></script>

  // Mehrere externe Skripte werden in der angegebenen Reihenfolge
  // AUSGEFÜHRT (Standardverhalten, "blockierend", siehe 1.2):
  <script src="files/placeholder.js"></script>
  <script src="files/nav.js"></script>
  <script src="files/volume.js"></script>
  // nav.js ruft makeBubbleVideo() auf, das in placeholder.js definiert
  // ist — die Reihenfolge ist hier also nicht egal!

Drei moderne Attribute, die das Ladeverhalten ändern:

  defer
    Lädt das Skript PARALLEL zum HTML-Parsen (blockiert NICHT die Seite),
    führt es aber erst aus, WENN das HTML komplett geparst ist — UND in
    der Reihenfolge, in der die <script>-Tags stehen. Kann deshalb
    problemlos im <head> stehen:

      <script src="files/placeholder.js" defer></script>
      <script src="files/nav.js" defer></script>
      <script src="files/volume.js" defer></script>

    → Für TrafkSite wäre das eine kleine, saubere Verbesserung: alle drei
    Dateien würden BEREITS WÄHREND das restliche HTML noch geparst wird
    heruntergeladen (schneller!), aber erst danach ausgeführt — Reihen-
    folge bleibt erhalten, #header-placeholder existiert garantiert.

  async
    Lädt PARALLEL, führt aber aus, SOBALD FERTIG — egal in welcher
    Reihenfolge zu anderen Skripten, egal ob das DOM schon fertig ist.
    Gut für komplett UNABHÄNGIGE Skripte (z.B. Analytics-Tracker).
    SCHLECHT, wenn Skripte sich gegenseitig brauchen — nav.js würde
    crashen, wenn es vor placeholder.js fertig wird und makeBubbleVideo()
    noch nicht existiert.

  type="module"
    Macht aus der Datei ein ES-Modul (siehe Modul 13). Effekte:
      - automatisch wie defer (lädt parallel, führt nach DOM-Aufbau aus)
      - eigener Geltungsbereich (Scope) — keine globalen Variablen mehr,
        die sich zwischen Dateien überschreiben können
      - automatisch im "strict mode" (strengere Fehlerprüfung)
      - erlaubt import/export zwischen Dateien

        <script type="module" src="files/main.js"></script>

------------------------------------------------------------------------
1.4  Die globalen Objekte — deine Werkzeugkiste
------------------------------------------------------------------------

Jedes JS, das im Browser läuft, hat automatisch Zugriff auf ein paar
riesige, vorgefertigte Objekte. Die wichtigsten:

  window
    Das Browserfenster selbst. ALLES Globale ist eigentlich eine
    Eigenschaft von window — auch deine eigenen Top-Level-Funktionen!
    Die Funktion initNav() aus nav.js ist technisch window.initNav().
    Praktisch:
      window.innerWidth / window.innerHeight  → aktuelle Fenstergröße
      window.location                          → aktuelle URL (Modul 9)
      window.addEventListener(...)             → globale Events (Modul 5)

  document
    Repräsentiert die aktuelle Seite — dein wichtigstes Werkzeug ab
    Modul 3. document.title, document.body, document.getElementById(...),
    document.querySelector(...), document.createElement(...), ...

  console
    Dein Debug-Werkzeug. console.log/warn/error/table/group — die volle
    Liste in Modul 19.

  navigator
    Infos über Browser & Gerät: navigator.userAgent, navigator.language,
    navigator.onLine, navigator.clipboard (siehe Modul 9).

  location  (Kurzform für window.location)
    Die aktuelle URL, in ihre Einzelteile zerlegt — sehr wichtig für
    Websites, eigenes Thema in Modul 9.

------------------------------------------------------------------------
1.5  Die DevTools — deine Werkbank
------------------------------------------------------------------------

Öffne auf JEDER Seite (auch auf trafkhop-entertainment.github.io) die
DevTools: Rechtsklick → "Untersuchen" (oder F12 / Strg+Shift+I).

  Console-Tab
    Ein JS-Interpreter, LIVE, in der laufenden Seite. Hier kannst du:
      document.title                          → aktuellen Titel sehen
      document.title = "Test"                 → Titel SOFORT ändern
      document.querySelector('h1')            → erstes <h1> finden
      document.body.style.background = 'red'  → Seite einfärben

    Jede Zeile, die du hier eintippst, läuft SOFORT auf der echten Seite.
    Das ist der beste Ort, um neue JS-Konzepte risikofrei auszuprobieren,
    bevor du sie in eine Datei schreibst.

  Elements-Tab
    Zeigt das LEBENDE DOM. Wenn dein JS ein Element ändert oder neu
    erzeugt — z.B. wenn nav.js die komplette Navigation in #header-nav
    hineinschreibt — siehst du das hier LIVE entstehen. Der beste Ort,
    um "warum ist mein Element nicht da / sieht falsch aus" zu debuggen.

  Sources-Tab
    Hier liegen deine .js-Dateien (so wie der Browser sie sieht). Du
    kannst Breakpoints setzen (Zeilennummer anklicken) — der Code stoppt
    dort, und du siehst alle aktuellen Variablenwerte. Mehr in Modul 19.

  Network-Tab
    Zeigt JEDEN Request, den die Seite macht — auch
    fetch('sitemap.xml') aus nav.js oder den POST an PROXY_URL aus
    AskAlfonz.js. Status-Code, Antwortzeit, Response-Body: alles
    sichtbar. Unschätzbar wertvoll, sobald du mit fetch arbeitest
    (Modul 8).

------------------------------------------------------------------------
1.6  Ein Blick voraus: JavaScript außerhalb des Browsers
------------------------------------------------------------------------

JavaScript ist längst nicht mehr NUR die Sprache des Browsers. Mit
Node.js läuft derselbe Sprachkern auch auf einem Server — OHNE window,
OHNE document, dafür mit Dateisystem-Zugriff und der Fähigkeit, selbst
ein HTTP-Server zu sein.

Dein index.js ist genau das:

  const express = require('express');
  // ...
  app.listen(PORT, () => { ... });

Das ist Node.js-Code — kein DOM, kein window. Stattdessen process.env
(Umgebungsvariablen, wie $VAR in Bash) und ein eigener Server, der
Anfragen entgegennimmt. Wir kommen in Modul 17 ausführlich darauf zurück.
Für jetzt reicht: GLEICHE Sprache (Variablen, Funktionen, Schleifen, alles
aus diesem Kurs gilt dort 1:1), ANDERE Umgebung (andere globale Objekte,
andere "Bühne").

MODUL 2: Variablen, Typen und Operatoren — die Bausteine

------------------------------------------------------------------------
2.1  var, let, const — der Unterschied
------------------------------------------------------------------------

In Shell hast du einfach NAME="Wert" geschrieben — kein Typ, kaum
Scope-Konzept. In JavaScript gibt es drei Schlüsselwörter zum Deklarieren
von Variablen, und der Unterschied ist wichtig — gerade weil du in
Web-Code oft Elemente "für später" referenzierst, bevor sie existieren.

  var   → veraltet, fast nie mehr benutzen. Hat seltsame Regeln (function
          scope, hoisting). Kommt in altem Code vor.

  let   → moderne Variable, kann überschrieben werden, nur innerhalb des
          Blocks {} gültig, in dem sie deklariert wurde.

  const → Konstante. Kann nicht neu zugewiesen werden. Trotzdem Standard
          für fast alles, weil Objekte/Arrays die in const stecken, noch
          verändert werden können (dazu mehr in Modul 6).

Faustregel: Immer const. Nur wenn du die Variable wirklich neu zuweisen
musst (z.B. Zähler in einer Schleife, oder ein DOM-Element, das erst
SPÄTER zugewiesen wird), nimm let. var: nie.

Beispiel aus deinem Code (AskAlfonz.js):

  let chatWindow, inputField, sendBtn, quickActions;
  // ↑ richtig! Diese Variablen sind beim Programmstart NICHTS
  // (undefined). Erst im DOMContentLoaded-Handler (siehe Modul 1 und 5!)
  // bekommen sie ihre echten DOM-Elemente zugewiesen. Deshalb MÜSSEN sie
  // "let" sein — sie werden NACH der Deklaration neu zugewiesen.

  const PROXY_URL = "https://trafkhop-entertainment-alfonzproxy.hf.space/chat";
  // ↑ richtig für URLs/Konstanten — wird im ganzen Skript nie verändert.

HÄUFIGER FEHLER: "Cannot access 'x' before initialization"
  Mit let/const gibt es eine "Temporal Dead Zone" — die Variable EXISTIERT
  zwar schon (im Gegensatz zu var), ist aber bis zur Deklarationszeile
  "gesperrt". Du kannst sie also nicht VOR ihrer Zeile benutzen. Das ist
  Absicht: Es verhindert Bugs durch Variablen, die "irgendwie schon da"
  waren, bevor man sie eigentlich definiert hat.

------------------------------------------------------------------------
2.2  Datentypen
------------------------------------------------------------------------

JavaScript ist "dynamisch getypt" — wie Shell, anders als C. Eine
Variable kann zuerst eine Zahl sein, dann ein String. Das ist Fluch und
Segen zugleich.

Die 8 Typen (davon 7 "primitiv", 1 komplex):

  string      "Hallo", 'Welt', `Template ${variable}`
  number      42, 3.14, -7, Infinity, NaN (ja, NaN ist ein Typ number!)
  boolean     true, false
  undefined   Variable deklariert, aber kein Wert zugewiesen
  null        Absichtlich "kein Wert" (von dir gesetzt)
  bigint      Für riesige ganze Zahlen: 9007199254740991n
  symbol      Einzigartiger Bezeichner, selten gebraucht
  object      Alles andere: Arrays, Objekte, Funktionen, null (Falle!)

typeof-Operator (zum Debuggen sehr nützlich):

  typeof "Hallo"      → "string"
  typeof 42           → "number"
  typeof true         → "boolean"
  typeof undefined    → "undefined"
  typeof null         → "object"  ← bekannter Bug in JS, merken!
  typeof []           → "object"
  typeof {}           → "object"
  typeof function(){} → "function"

WICHTIG FÜR WEBSEITEN — die "Alles ist ein String"-Falle:

  JEDER Wert, der aus dem HTML/DOM kommt, ist IMMER ein String — auch
  wenn er aussieht wie eine Zahl!

    const eingabe = document.getElementById('alter');
    eingabe.value           → "25"        (STRING — auch bei type="number"!)
    typeof eingabe.value     → "string"

  Willst du damit RECHNEN, musst du explizit umwandeln:

    Number(eingabe.value)     → 25    (number)
    parseInt(eingabe.value)   → 25    (number, ignoriert Nachkommastellen)
    parseFloat(eingabe.value) → 25.5  (number, MIT Nachkommastellen)

  Umgekehrt: Willst du eine Zahl als CSS-Wert setzen, reicht "Zahl" oft
  NICHT — CSS braucht eine Einheit:

    el.style.width = 100;        // wird ignoriert (CSS versteht "100" nicht)
    el.style.width = "100px";    // richtig
    el.style.width = `${100}px`; // richtig — Template Literal, siehe 2.3

  Dieser "Alles ist String"-Stolperstein ist DER häufigste Anfängerfehler
  bei Formularen — ausführlich in Modul 7.

------------------------------------------------------------------------
2.3  String-Besonderheiten
------------------------------------------------------------------------

Drei Arten von Strings:

  'einfache Anführungszeichen'    → Standard
  "doppelte Anführungszeichen"    → Standard, austauschbar
  `Backtick Template Literals`    → Modern, erlaubt Interpolation

Template Literals sind dein Freund — du benutzt sie in deinem Code schon:

  const name = "Alfonz";
  const alter = 400_000_000_000;  // Unterstriche erlaubt als Lesehilfe

  // Alt (String-Konkatenation):
  "Ich bin " + name + " und " + alter + " Jahre alt."

  // Neu (Template Literal):
  `Ich bin ${name} und ${alter} Jahre alt.`

  // Beliebige Ausdrücke sind erlaubt:
  `Ergebnis: ${2 + 2}`            → "Ergebnis: 4"
  `Groß: ${name.toUpperCase()}`   → "Groß: ALFONZ"

  // Mehrzeilig — ohne \n, einfach Zeilenumbruch im Code:
  const text = `Zeile 1
  Zeile 2`;

  // Aus deinem AskAlfonz.js — HTML per Template Literal zusammenbauen:
  msgDiv.innerHTML = `<b style="color:#7FFFD4;">Traveler:</b> <p>${text}</p>`;
  // Template Literals sind DER Standardweg, um dynamisches HTML zu bauen
  // (mehr dazu in Modul 3 — UND eine Sicherheitswarnung dazu in Modul 18!)

Wichtige String-Methoden (du brauchst fast alle davon):

  const s = "  Hallo Welt  ";

  s.length                → 14
  s.trim()                → "Hallo Welt"        (Whitespace entfernen)
  s.toLowerCase()         → "  hallo welt  "
  s.toUpperCase()         → "  HALLO WELT  "
  s.includes("Welt")      → true
  s.startsWith("Hallo")   → false  (wegen Leerzeichen vorne!)
  s.startsWith("  Hallo") → true
  s.endsWith("Welt  ")    → true
  s.replace("Welt", "JS") → "  Hallo JS  "        (ersetzt NUR erstes)
  s.replaceAll("l", "L")  → "  HaLLo WeLt  "      (alle Vorkommen)
  s.split(" ")            → ["", "", "Hallo", "Welt", "", ""]
  s.slice(2, 7)           → "Hallo"   (von Index 2, exklusiv Index 7)
  s.indexOf("Welt")       → 8
  s.padStart(20, "0")     → "000000  Hallo Welt  "  (auf Länge auffüllen)

  // Regex-Replace (wie in deinem Code: "${game// /%20}" in Bash):
  "Hallo Welt Foo".replace(/ /g, "%20")   → "Hallo%20Welt%20Foo"
  // Das /g = global flag = alle Vorkommen ersetzen (ohne g: nur erstes)

------------------------------------------------------------------------
2.4  Operatoren — Rechnen, Vergleichen, Verknüpfen
------------------------------------------------------------------------

Arithmetische Operatoren:

  +  -  *  /  %  (Modulo/Rest)  **  (Potenz)
  ++ (Increment)  -- (Decrement) — als Präfix oder Postfix

  let punkte = 0;
  punkte++;        // punkte = 1 (Postfix: gibt ALTEN Wert zurück, dann +1)
  ++punkte;        // punkte = 2 (Präfix: erhöht ZUERST, gibt neuen Wert)

  10 % 3   → 1     (Rest bei Division) — extrem nützlich für "jedes
                    2. Element", "jeden 2. Frame" etc.

  // Aus deinem placeholder.js:
  if (_frameCount % 2 !== 0) return;
  // "Wenn der Frame-Zähler bei Division durch 2 einen Rest hat (also
  // ungerade ist), überspringe diesen Frame." → effektiv 30fps statt 60fps.

  2 ** 8   → 256   (Potenz, modern statt Math.pow(2, 8))

DIE "STRING-PLUS"-FALLE:
  Der + Operator ist ÜBERLADEN — bei Strings bedeutet er "verbinden"
  (concat), bei Zahlen "addieren". JS entscheidet anhand der Operanden,
  und sobald EINER ein String ist, wird der ANDERE zu String konvertiert:

    "5" + 3     → "53"   (string + number → string concat!)
    5 + "3"     → "53"
    "5" - 3     → 2      (- gibt's NUR für Zahlen → "5" wird zu 5)
    "5" * "2"   → 10     (* genauso → beide werden zu Zahlen)

  Das ist DER Klassiker bei Formularen:

    const a = document.getElementById('a').value;  // "5"  (string!)
    const b = document.getElementById('b').value;  // "3"  (string!)
    a + b                  → "53"   ← BUG! Nutzer erwartet 8
    Number(a) + Number(b)  → 8      ← richtig

Vergleichsoperatoren: >  <  >=  <=  (siehe 2.5 für == und ===)

  Bei Zahlen funktionieren sie wie erwartet. Bei Strings: lexikografisch
  (vereinfacht: alphabetisch, genauer nach Unicode-Wert):

    "Apfel" < "Banane"   → true
    "10" < "9"           → true!  (String-Vergleich: Zeichen "1" < "9")
    10 < 9               → false  (Zahlen-Vergleich)

  → Noch ein Grund, Werte aus Formularen VOR dem Vergleich zu konvertieren!

Logische Operatoren: &&  (UND)   ||  (ODER)   !  (NICHT)

  Die Wahrheitstabellen kennst du sicher — viel wichtiger im echten Code
  ist aber SHORT-CIRCUIT EVALUATION: JS wertet von links nach rechts aus
  und STOPPT, sobald das Ergebnis feststeht.

    &&  stoppt bei FALSY  (gibt diesen falsy-Wert zurück)
    ||  stoppt bei TRUTHY (gibt diesen truthy-Wert zurück)

  Das wird als KURZFORM für if benutzt — extrem verbreitet:

    // Statt:
    if (callback) { callback(); }
    // Kurzform:
    callback && callback();

    // Sinngemäß im Stil von AskAlfonz.js:
    images && images.length > 0 && addImages(images);
    // "Wenn images existiert UND mind. 1 Element hat → addImages aufrufen"

    // || für Fallback-Werte (siehe auch ?? in 2.6):
    const priority = urlPriorityMap.get(url) || 0.65;
    // "Hole priority — falls das falsy ist (z.B. undefined), nimm 0.65"

  ! kehrt um:
    !true        → false
    !""          → true   (leerer String ist falsy → ! macht daraus true)

  !! ("Doppel-Nicht") wandelt JEDEN Wert in ein echtes Boolean um — ein
  häufig gesehener Trick:
    !!"Hallo"    → true
    !!""         → false
    !!0          → false
    !!42         → true

Der Ternäre Operator (Bedingungs-Operator) — ? :

    bedingung ? wertWennWahr : wertWennFalsch

  Die kompakte Alternative zu if/else, wenn du einen WERT brauchst, statt
  eine Anweisung auszuführen. In Web-Code überall zu finden, weil man
  ständig Texte/Klassen/Styles abhängig von einer Bedingung setzt:

    // if/else — eine ANWEISUNG, braucht eine vorab deklarierte Variable:
    let status;
    if (indexReady) { status = "Bereit"; } else { status = "Lädt..."; }

    // Ternär — ein AUSDRUCK, direkt verwendbar:
    const status = indexReady ? "Bereit" : "Lädt...";

    // Sinngemäß im Stil von AskAlfonz.js:
    const label = addedFiles === 0
        ? `[MAIN SOURCE]\nSOURCE: ${doc.url}`
        : `SOURCE: ${doc.url}`;

    // DER Klassiker für bedingtes Rendering, direkt im Template Literal:
    el.innerHTML = `<span class="${istAktiv ? 'aktiv' : ''}">${name}</span>`;

  Verschachtelte Ternäre sind möglich, aber werden schnell unleserlich —
  ab 3 Bedingungen lieber if/else oder switch (Modul 4) verwenden.

------------------------------------------------------------------------
2.5  Vergleiche — die Falle mit == vs ===
------------------------------------------------------------------------

Das ist einer der größten Stolpersteine in JS.

  ==  (lose Gleichheit): versucht Typen umzuwandeln vor dem Vergleich
  === (strenge Gleichheit): Wert UND Typ müssen gleich sein

  0 == false        → true   (JS konvertiert false zu 0)
  0 === false       → false  (number != boolean)
  "1" == 1          → true   (JS konvertiert "1" zu 1)
  "1" === 1         → false
  null == undefined → true
  null === undefined→ false

Regel: IMMER === benutzen. Nur in sehr seltenen Fällen brauchst du ==.

  // Aus deinem Code:
  if (e.key === 'Enter') sendMessage();   ← richtig!

------------------------------------------------------------------------
2.6  Truthy und Falsy
------------------------------------------------------------------------

In Shell: if [ "$VAR" ]; dann... — in JS ist das ähnlich, aber anders.

Folgende Werte sind "falsy" (werden in if-Bedingungen als false behandelt):

  false, 0, -0, 0n, "", '', ``, null, undefined, NaN

ALLES andere ist "truthy" — also auch leere Arrays [] und Objekte {}!

Das benutzt du schon unbewusst:

  if (!url) return false;     ← url ist "" oder null oder undefined → false
  if (!text) return;          ← text ist Leer-String → falsy → return

Für Webseiten besonders relevant: ein leeres Eingabefeld (<input>) hat
.value === "" — und "" ist falsy. So prüfst du mit einem einzigen if, ob
ein Feld ausgefüllt wurde:

  if (!inputField.value.trim()) {
      // Feld ist leer (oder nur Leerzeichen) → nicht absenden
  }

------------------------------------------------------------------------
2.7  Nullish Coalescing und Optional Chaining (sehr modern, sehr nützlich)
------------------------------------------------------------------------

Nullish Coalescing (??) — Fallback wenn null oder undefined:

  const vol = localStorage.getItem('site-volume') ?? '0.5';
  // Du benutzt das genau so in volume.js! parseFloat(... ?? '0.5')
  // Liefert den linken Wert, AUSSER er ist null oder undefined.
  // Unterschied zu ||: 0 ?? 'fallback' → 0 (weil 0 nicht null ist)
  //                     0 || 'fallback' → 'fallback' (weil 0 falsy ist)

  → Genau DAS ist der Grund, warum volume.js ?? statt || benutzt: Wäre
  die gespeicherte Lautstärke 0 (also stumm), würde || das fälschlich
  als "nichts gespeichert" werten und auf 0.5 zurückspringen. ?? lässt
  0 als gültigen, gespeicherten Wert durch.

Optional Chaining (?.) — kein Crash, wenn ein Objekt null ist:

  error.response?.data
  // Du benutzt das in index.js!
  // Bedeutet: wenn error.response nicht null/undefined ist, greif auf
  // .data zu. Sonst: undefined (kein TypeError-Crash).

  // Weitere Beispiele:
  const len = user?.name?.length ?? 0;
  // Kein Crash, auch wenn user oder user.name undefined ist.

  document.getElementById(loadingId)?.remove();
  // Aus deinem AskAlfonz.js — entfernt das Element NUR, wenn es existiert.

MODUL 3: Das DOM — deine Seite aus JS heraus steuern
================================================================================

Das ist DAS Kapitel, das JS vom "rechnet irgendwo im Hintergrund" zum
"verändert live, was du auf dem Bildschirm siehst" macht. Alles, was
nav.js, placeholder.js, volume.js und AskAlfonz.js auf der Seite TUN,
basiert auf diesem Modul.

------------------------------------------------------------------------
3.1  Vom HTML-Baum zum DOM — die Brücke zwischen HTML und JS
------------------------------------------------------------------------

Aus dem HTML-Kurs kennst du das Prinzip: Eine Seite ist ein BAUM aus
verschachtelten Elementen. Genau dieser Baum existiert auch als
JS-Objektstruktur — das DOM (Document Object Model). Jedes <div>, <p>,
<img> wird zu einem JS-Objekt mit Eigenschaften und Methoden.

document ist die Wurzel dieses Baums — repräsentiert das gesamte Dokument:

  document
   └── html
        ├── head
        │    ├── meta, meta
        │    ├── link (standard.css, index.css)
        │    └── title
        └── body
             ├── div#header-placeholder
             ├── div#nav-placeholder
             ├── div#footer-placeholder
             ├── main
             │    ├── section#Logo
             │    ├── section#Projects
             │    └── section#Members
             └── script, script, script

WICHTIG: Das DOM ist NICHT dasselbe wie deine .html-Datei! Die .html-
Datei ist der "Bauplan", das DOM ist das tatsächlich gebaute "Haus" — und
JS kann dieses Haus NACH dem Bau noch umbauen: Wände verschieben, Räume
hinzufügen, Möbel (Inhalte) austauschen. Schaust du im Elements-Tab nach,
siehst du das DOM, NICHT die Originaldatei — deshalb erscheint dort die
komplette Navigation, obwohl in index.html nur ein leeres
<div id="header-placeholder"></div> steht. placeholder.js hat das "Haus"
nach dem Bauen umgebaut.

------------------------------------------------------------------------
3.2  Elemente finden
------------------------------------------------------------------------

  // Nach ID (gibt EIN Element zurück, oder null):
  document.getElementById('chat-window')

  // Nach CSS-Selektor (ERSTER Treffer):
  document.querySelector('.tree-ul')
  document.querySelector('section > h2')
  document.querySelector('#header-nav .nav')

  // Nach CSS-Selektor (ALLE Treffer — eine NodeList, KEIN Array!):
  document.querySelectorAll('.nav')

  // NodeList zu Array konvertieren (für map/filter/reduce nötig):
  Array.from(document.querySelectorAll('.nav'))
  [...document.querySelectorAll('.nav')]

  // Innerhalb EINES Elements suchen, nicht im ganzen Dokument:
  const nav = document.getElementById('header-nav');
  nav.querySelectorAll('.tree-li')   // nur innerhalb von nav

  // Aus deinem placeholder.js:
  document.getElementById(elementId).innerHTML = data;
  root.querySelectorAll('.nav').forEach(el => { ... });

querySelector vs. getElementById — was nehmen?

  getElementById ist minimal schneller, geht aber NUR nach ID.
  querySelector kann ALLES, was CSS auch kann — Klassen, Attribute,
  Kombinatoren, sogar Pseudoklassen wie :first-child. In der Praxis:
  nimm querySelector/querySelectorAll für (fast) alles. getElementById
  ist okay, wenn eine ID sowieso feststeht und du es minimal lesbarer
  findest — beide Schreibweisen sind in deinem Code vertreten.

------------------------------------------------------------------------
3.3  Inhalte lesen und ändern — textContent, innerHTML, innerText, value
------------------------------------------------------------------------

Drei verschiedene Wege an den "Inhalt" eines Elements zu kommen — und sie
sind NICHT dasselbe:

  textContent
    Der RAW-Text — inklusive Text in versteckten Elementen, OHNE
    HTML-Parsing. Schnell und sicher (kein XSS-Risiko, siehe Modul 18).
    BEIM SETZEN: el.textContent = "Hallo <b>Welt</b>" zeigt LITERAL
    "Hallo <b>Welt</b>" an — das <b> wird NICHT als Tag interpretiert,
    sondern als Text dargestellt.

  innerHTML
    Der HTML-Code als String — INKLUSIVE Tags. BEIM SETZEN wird der
    String als HTML GEPARST und ins DOM eingebaut (es entstehen also
    wirklich NEUE Elemente!). el.innerHTML = "Hallo <b>Welt</b>" macht
    "Welt" tatsächlich fett.
    GEFAHR: Stammt der String von einem NUTZER, kann hier theoretisch
    Schadcode landen (XSS) — Details und Lösungen in Modul 18.

  innerText
    Ähnlich wie textContent, ABER: respektiert CSS! Text in
    display:none-Elementen wird NICHT zurückgegeben, Zeilenumbrüche
    entsprechen dem, was im Browser sichtbar ist. Dafür LANGSAMER —
    erzwingt ein "Reflow" (der Browser muss das Layout neu berechnen, um
    zu wissen, was überhaupt sichtbar ist).

  Faustregel:
    - Reinen TEXT anzeigen/lesen          → textContent (schnell, sicher)
    - HTML-STRUKTUR erzeugen (mehrere
      Elemente auf einmal, eigene Daten)  → innerHTML
    - innerText: selten nötig, fast immer ist textContent die
      bessere Wahl.

  Aus deinem nav.js:
    link.textContent = n.name;
    link.title = n.name;
    // → reiner Text, sicher, schnell. n.name kommt aus der Sitemap,
    // wird aber NICHT als HTML interpretiert — gut so.

  Aus deinem AskAlfonz.js:
    msgDiv.innerHTML = `<b style="color:#7FFFD4;">Traveler:</b> <p>${text}</p>`;
    // → HTML-Struktur (<b> und <p>) wird tatsächlich erzeugt. <script>-
    // Tags würden hier NICHT ausgeführt — innerHTML führt nie Skripte
    // aus. Trotzdem: bei eigenen Projekten immer kurz überlegen, WOHER
    // ein String kommt, bevor er per innerHTML landet (Modul 18 vertieft
    // das mit konkreten Beispielen).

Eingabefelder sind ANDERS — sie haben kein textContent/innerHTML für
ihren aktuellen Wert, sondern .value:

  const input = document.getElementById('chat-input');
  input.value           // aktueller Text im Feld (IMMER ein String!)
  input.value = "";     // Feld leeren
  input.value = "Hallo" // Feld vorbefüllen

  // Aus deinem AskAlfonz.js:
  let text = inputField.value.trim();
  inputField.value = '';   // Feld nach dem Senden leeren

3.4  Attribute, Klassen und data-*-Attribute
------------------------------------------------------------------------

  // Attribute lesen/setzen:
  link.href = 'https://...';
  link.getAttribute('href')
  link.setAttribute('data-id', '42')
  link.removeAttribute('disabled')
  link.hasAttribute('disabled')   → true / false

  // Klassen — DAS Werkzeug für "Zustand" auf Webseiten:
  el.classList.add('open')
  el.classList.remove('open')
  el.classList.toggle('open')        // an wenn aus, aus wenn an
  el.classList.toggle('open', true)  // ZWEITER Parameter: erzwingt
                                       // an (true) oder aus (false) —
                                       // statt nur umzuschalten!
  el.classList.contains('open')      → true / false
  el.classList.replace('alt', 'neu')

  // Aus deinem nav.js:
  li.querySelector('.tree-ul').classList.toggle('open');
  arrow.classList.toggle('open');
  // Zwei Klassen gleichzeitig toggeln → CSS macht den Rest, siehe
  // standard.css: #header-nav .tree-ul.open { display: flex; }

data-*-Attribute und .dataset — eigene Daten am Element "festkleben"

  Du kannst an JEDES HTML-Element eigene Attribute hängen, die mit
  "data-" beginnen. Der Browser ignoriert sie fürs Rendering, aber JS
  kann sie auslesen — perfekt, um Infos direkt am Element zu speichern,
  die du z.B. bei einem Klick brauchst:

    <button data-projekt-id="42" data-status="aktiv">Details</button>

    // In JS — automatische camelCase-Umwandlung von kebab-case:
    btn.dataset.projektId   → "42"     (data-projekt-id → projektId)
    btn.dataset.status      → "aktiv"
    btn.dataset.neu = "ja"  // setzt im DOM: data-neu="ja"

  Praktisches Beispiel für TrafkSite: Du könntest jeder Projekt-Kachel
  in #projectsdiv ein data-projekt="TrafkTux" geben und beim Klick darauf
  reagieren — OHNE für jede Kachel eine eigene Funktion zu schreiben
  (siehe Event Delegation in Modul 5).

3.5  Styles direkt ändern — und warum man es meistens NICHT tun sollte
------------------------------------------------------------------------

  // Inline-Styles direkt setzen:
  el.style.display = 'none'
  el.style.color = 'red'
  el.style.cssText = 'position:absolute; top:0; left:0;'
  // Aus deinem placeholder.js!

  // CSS-Eigenschaften mit Bindestrich → camelCase in JS:
  el.style.backgroundColor = 'blue';   // background-color
  el.style.fontSize = '1.5rem';        // font-size

  // Den TATSÄCHLICH berechneten Stil lesen (egal woher er kommt —
  // CSS-Datei, Browser-Default, oder Inline-Style):
  const styles = getComputedStyle(el);
  styles.color           → tatsächliche Farbe als rgb(...)

WANN el.style.xyz, WANN classList?

  - Einmalige, sehr SPEZIFISCHE Werte, oft aus Berechnungen (Canvas-
    Größen, dynamische Positionen) → el.style.xyz direkt, wie in
    placeholder.js:
      canvas.style.cssText = 'position:absolute;top:0;left:0;width:100%;...';

  - "Zustände" (offen/geschlossen, aktiv/inaktiv, hell/dunkel) →
    classList.toggle() + die eigentlichen Werte in der CSS-Datei
    definieren. Das hält DESIGN (CSS) und LOGIK (JS) GETRENNT — du
    kannst später das Aussehen ändern, ohne JS anzufassen.

    // Schlecht — Design verstreut sich in JS:
    el.style.display = istOffen ? 'flex' : 'none';
    el.style.borderLeft = istOffen ? '2px dashed purple' : 'none';

    // Besser — CSS entscheidet, JS sagt nur "an" oder "aus":
    el.classList.toggle('open', istOffen);
    /* in standard.css definiert:
       #header-nav .tree-ul.open { display: flex; } */

  Genau dieses Prinzip siehst du bei #header-nav .tree-ul.open in
  standard.css: nav.js toggelt NUR die Klasse "open", ALLE visuellen
  Details (display, Pfeil-Rotation etc.) stehen im CSS.

  // Element erstellen:
  const li = document.createElement('li');

  li.classList.add('tree-li');       // FÜGT eine Klasse hinzu (sicherer)

  // Aus deinem nav.js — kompletter Aufbau eines Elements:
  const ul = document.createElement('ul');
  ul.className = 'tree-ul root-ul';

  const link = document.createElement('a');
  link.className = 'do';
  link.href = n.path || '#';

  // Einfügen — verschiedene Positionen:
  parentEl.appendChild(ul);                    // ans Ende anfügen
  el.insertBefore(hovCanvas, el.firstChild);   // vor erstem Kind einfügen
  el.prepend(canvas);                          // modern: ans Anfang
  el.append(canvas);                           // modern: ans Ende
  ref.after(newEl);                            // direkt NACH ref einfügen
  ref.before(newEl);                           // direkt VOR ref einfügen

  // Entfernen:
  el.remove();                  // modern, einfach
  parent.removeChild(child);    // älterer Weg, falls .remove() fehlt

  // Jedes appendChild kann (im schlimmsten Fall) ein Neu-Zeichnen der

  const fragment = document.createDocumentFragment();
  for (const name of ["Apfel", "Banane", "Kirsche"]) {
      const li = document.createElement('li');
      li.textContent = name;
      fragment.appendChild(li);
  }
  ul.appendChild(fragment);   // EIN Einfüge-Vorgang statt drei

3.7  NodeList vs. Array — kleine, aber wichtige Unterschiede
------------------------------------------------------------------------

  const liste = document.querySelectorAll('.nav');  // NodeList

  liste.length         // funktioniert, wie bei Arrays
  liste.forEach(...)   // funktioniert, wie bei Arrays
  liste.map(...)       // FEHLER! map gibt es auf NodeList nicht.

  // Umwandeln, wenn du map/filter/reduce brauchst:
  const arr  = Array.from(liste);
  const arr2 = [...liste];

  // children vs. childNodes vs. querySelectorAll('*'):
  el.children               → NUR direkte Kind-ELEMENTE (HTMLCollection)
  el.childNodes              → ALLE Kind-NODES, auch Text/Kommentare
  el.querySelectorAll('*')   → ALLE Nachfahren, beliebig tief

------------------------------------------------------------------------
4.1  Function Declaration vs Function Expression vs Arrow Function
------------------------------------------------------------------------

Es gibt drei Wege, Funktionen zu schreiben:

  // 1. Function Declaration (wird "gehoisted" — kann VOR der eigenen
  //    Deklaration im Code aufgerufen werden):
  function greet(name) {
      return `Hallo, ${name}!`;
  }

  // 2. Function Expression (in const — KEIN Hoisting):
  const greet = function(name) {
      return `Hallo, ${name}!`;
  };

  // 3. Arrow Function (modern, kompakt, KEIN eigenes "this" — siehe 4.6):
  const greet = (name) => {
      return `Hallo, ${name}!`;
  };

  // Noch kürzer, wenn nur EIN Ausdruck zurückgegeben wird:
  const greet = name => `Hallo, ${name}!`;

  // Mehrere Parameter → immer Klammern:
  const add = (a, b) => a + b;

  // Kein Parameter → leere Klammern:
  const rand = () => Math.random();

  // Objekt zurückgeben → extra Klammern nötig! (sonst denkt JS, {...}
  // ist der Funktionskörper, kein Objekt):
  const makeUser = name => ({ name, active: true });

Wann welche?
  - Function Declaration: für "richtige", benannte Top-Level-Funktionen
    (so wie initNav, buildTree, renderTree in nav.js).
  - Arrow Function: für KURZE Callbacks (Event-Handler, map/filter/...),
    und überall, wo du "this" vom umgebenden Code übernehmen willst.

------------------------------------------------------------------------
4.2  Parameter: Defaults, Rest, Destructuring
------------------------------------------------------------------------

  // Default-Werte (wie in Bash: ${VAR:-default}):
  function makeBubbleVideo(src = BUBBLE_VIDEO, className = 'nav-bubble-video') {
      // Aus deinem placeholder.js! Wird makeBubbleVideo() ohne Argumente
      // aufgerufen, springen BUBBLE_VIDEO und 'nav-bubble-video' ein.
  }

  // Rest-Parameter — alle weiteren Argumente als Array:
  function log(prefix, ...messages) {
      messages.forEach(m => console.log(`[${prefix}] ${m}`));
  }
  log("INFO", "Start", "Laufe", "Ende");

  // Objekt-Destructuring im Parameter — extrem verbreitet für "Optionen":
  function render({ name, active = false, role = "user" }) {
      console.log(name, active, role);
  }
  render({ name: "Alfonz", active: true });
  // → "Alfonz" true "user"  — role bekommt den Default, weil im
  // übergebenen Objekt kein role-Key existiert.

------------------------------------------------------------------------
4.3  Scope — wo eine Variable "sichtbar" ist
------------------------------------------------------------------------

In Shell hast du vielleicht erlebt, dass eine Variable in einer Funktion
plötzlich überall verfügbar war (oder eben nicht). "Scope" beschreibt
genau das: WO im Code eine Variable existiert — und wo nicht.

JS kennt drei Arten von Scope:

  Globaler Scope
    Außerhalb jeder Funktion/jedes Blocks deklariert → überall im
    Skript sichtbar — und bei normalen <script>-Tags OHNE
    type="module" sogar in ANDEREN <script>-Tags derselben Seite!

      let chatHistory = [];   // ganz oben in AskAlfonz.js — global

  Funktions-Scope
    Mit var ODER let/const INNERHALB einer Funktion deklariert → nur
    INNERHALB dieser Funktion sichtbar.

      function sendMessage() {
          const text = inputField.value;  // nur HIER sichtbar
      }
      console.log(text);  // FEHLER: text is not defined

  Block-Scope (nur let/const — NICHT var!)
    Innerhalb von {} (if, for, while, ...) deklariert → nur INNERHALB
    dieses Blocks sichtbar.

      if (indexReady) {
          const meldung = "Bereit!";  // nur in diesem if-Block
      }
      console.log(meldung);  // FEHLER: meldung is not defined

      // var "sickert" dagegen aus dem Block heraus — verwirrend:
      if (true) { var x = 5; }
      console.log(x);  // 5 — funktioniert mit var (einer der Gründe,
                        // var zu meiden!)

Die Scope-Kette: Eine innere Funktion sieht alle Variablen ihrer
"Eltern"-Funktionen/Blöcke — aber NICHT umgekehrt:

  let aussen = "A";
  function f() {
      let innen = "B";
      console.log(aussen);  // "A" — sieht nach AUSSEN
  }
  f();
  console.log(innen);  // FEHLER — außen sieht NICHT nach innen

WARUM DAS FÜR WEBSEITEN WICHTIG IST:

  Lädst du mehrere <script>-Dateien OHNE type="module" — so wie
  index.html mit placeholder.js, nav.js, volume.js — landen ALLE
  Top-Level-Variablen und -Funktionen im GLOBALEN Scope, also im SELBEN
  "Topf". Das erklärt zwei Dinge:

    - WARUM nav.js die Funktion makeBubbleVideo() aus placeholder.js
      einfach aufrufen kann (typeof makeBubbleVideo === 'function') —
      beide Dateien teilen sich den globalen Scope.

    - WARUM das auch ein RISIKO ist: Zwei Dateien könnten sich
      versehentlich überschreiben, wenn beide global eine Variable mit
      demselben Namen anlegen.

  Genau dieses Risiko ist einer der Gründe für IIFEs (siehe 4.5) und für
  ES-Module mit type="module" (Modul 13) — dort hat jede Datei ihren
  EIGENEN Scope, nichts "sickert" mehr ungewollt global durch.

------------------------------------------------------------------------
4.4  Return-Werte — und das "früh-return"-Muster
------------------------------------------------------------------------

Eine Funktion ohne return gibt automatisch undefined zurück. return
beendet die Funktion SOFORT — alles danach wird NICHT mehr ausgeführt.

Das "früh-return"-Muster vermeidet tiefe Verschachtelung — extrem
verbreitet bei defensivem Code (Modul 12 vertieft das):

  // Verschachtelt (schwer lesbar):
  function shouldIndexUrl(url) {
      if (url) {
          const lower = url.toLowerCase();
          if (!lower.includes('/.git/')) {
              return true;
          } else {
              return false;
          }
      } else {
          return false;
      }
  }

  // Früh-Return — genau wie in deinem AskAlfonz.js!
  function shouldIndexUrl(url) {
      if (!url) return false;
      const lower = url.toLowerCase();
      if (lower.includes('/.git/') || lower.includes('\\.git\\')) return false;
      return true;
  }

  // Beide tun dasselbe — Version 2 liest sich wie eine Checkliste von
  // "raus-Bedingungen" statt sich zu verschachteln. Je mehr
  // Sonderfälle eine Funktion hat, desto deutlicher der Unterschied.

------------------------------------------------------------------------
4.5  Closures — Funktionen, die ihren Kontext "einschließen"
------------------------------------------------------------------------

Eines der mächtigsten (und anfangs verwirrendsten) Konzepte in JS. Eine
innere Funktion hat Zugriff auf Variablen der äußeren Funktion — AUCH
NACHDEM die äußere schon zurückgekehrt ist.

  function makeCounter() {
      let count = 0;          // Diese Variable lebt WEITER!
      return function() {
          count++;
          return count;
      };
  }

  const counter = makeCounter();
  counter();  → 1
  counter();  → 2
  counter();  → 3

  // counter hat die Variable count "eingeschlossen" (closure).
  // count ist PRIVAT — von außen nicht erreichbar, nur über counter().

Aus deinem volume.js — das IIFE (Immediately Invoked Function Expression):

  (function() {
      const getStoredVol = () => parseFloat(localStorage.getItem('...') ?? '0.5');
      // ...alles hier ist privat, verschmutzt den globalen Namespace nicht
  })();

  // Das () am Ende führt die Funktion SOFORT aus.
  // Technik aus der Zeit vor Modulen — verhindert globale Variablen.
  // Genau das Problem aus 4.3 (globaler Scope bei mehreren <script>-
  // Tags) wird hier gezielt umgangen: alles in volume.js, was NICHT
  // global gebraucht wird, bleibt PRIVAT innerhalb dieser einen
  // Funktion. Mit ES-Modulen (Modul 13) löst man das heute eleganter.

------------------------------------------------------------------------
4.6  this — der gefürchtete Kontext (und warum Arrow Functions helfen)
------------------------------------------------------------------------

In klassischen Funktionen zeigt "this" auf den AUFRUFER. Das ist oft
überraschend:

  const obj = {
      name: "Alfonz",
      greet: function() {
          console.log(this.name);  // "Alfonz" ✓ — über obj.greet() aufgerufen
      },
      greetArrow: () => {
          console.log(this.name);  // undefined! Arrow hat KEIN eigenes this
      }
  };

  const fn = obj.greet;
  fn();  // undefined! "this" ist jetzt window (bzw. undefined im strict mode)
         // — der AUFRUF-KONTEXT ist verloren gegangen.

  // Arrow Functions ERBEN "this" vom UMGEBENDEN Scope — daher sicherer
  // in Callbacks:

  class Timer {
      constructor() {
          this.sekunden = 0;
      }
      start() {
          // OHNE Arrow wäre "this" hier undefined/window:
          setInterval(() => {
              this.sekunden++;
              console.log(this.sekunden);
          }, 1000);
      }
  }

Merksatz: Wenn du nicht sicher bist, was "this" gerade ist, nimm Arrow
Functions in Callbacks. Nur "echte" Methoden auf Objekten/Klassen
brauchen "function" mit eigenem "this".

------------------------------------------------------------------------
4.7  Höhere Ordnung — Funktionen als Argumente
------------------------------------------------------------------------

Funktionen sind in JS "First Class Citizens" — sie können wie Variablen
übergeben, zurückgegeben und gespeichert werden.

  // Funktion als Argument:
  [1,2,3].filter(n => n > 1);   // Die Arrow ist hier die "Callback-Funktion"

  // Funktion zurückgeben (Factory-Pattern):
  function makeMultiplier(faktor) {
      return n => n * faktor;
  }
  const verdoppeln = makeMultiplier(2);
  const verdreifachen = makeMultiplier(3);
  verdoppeln(5)    → 10
  verdreifachen(5) → 15

  // Aus deinem nav.js:
  arrow.onclick = e => {
      e.stopPropagation();
      li.querySelector('.tree-ul').classList.toggle('open');
      arrow.classList.toggle('open');
  };
  // Die Arrow-Funktion wird als EVENT-HANDLER übergeben — genau das
  // Thema von Modul 5, das jetzt direkt folgt.

================================================================================
MODUL 5: Events — auf den Nutzer reagieren
================================================================================

Eine Website ohne Interaktivität ist nur ein digitales Plakat. Events sind
die Brücke zwischen "der Nutzer tut etwas" und "deine Funktion läuft".
Jeder Klick, jede Eingabe, jedes Scrollen in deinem Code (Buttons,
Lautstärkeregler, Suchfeld, Pfeile in der Navigation) basiert hierauf.

------------------------------------------------------------------------
5.1  addEventListener — die Grundlage aller Interaktivität
------------------------------------------------------------------------

  element.addEventListener('eventName', handlerFunktion);

  // Aus deinem AskAlfonz.js:
  sendBtn.addEventListener('click', sendMessage);

  // Arrow-Funktion direkt als Handler:
  inputField.addEventListener('keypress', e => {
      if (e.key === 'Enter') sendMessage();
  });

Warum addEventListener und NICHT onclick="..." direkt im HTML?
  - Mehrere Listener für dasselbe Element/Event sind möglich (onclick="
    " im HTML erlaubt effektiv nur EINEN Handler).
  - Trennung von HTML (Struktur) und JS (Verhalten) — "Unobtrusive
    JavaScript", ein zentrales Web-Prinzip: dein HTML bleibt lesbar, ohne
    dass überall onclick="..." dazwischensteht.
  - Mehr Kontrolle: Optionen wie { once: true } oder { passive: true }
    (siehe 5.5), und Listener lassen sich gezielt wieder entfernen.

------------------------------------------------------------------------
5.2  Das Event-Objekt
------------------------------------------------------------------------

Jeder Handler bekommt automatisch ein Event-Objekt übergeben (oft "e" oder
"event" genannt) — es beschreibt, WAS passiert ist:

  btn.addEventListener('click', event => {
      event.preventDefault();    // Standard-Verhalten unterbinden
                                  // (z.B. Link-Navigation, Form-Submit)
      event.stopPropagation();   // Event nicht weiter "nach oben" geben
      event.target                // Element, das TATSÄCHLICH den Event
                                   // ausgelöst hat (z.B. ein <span> in
                                   // einem Button)
      event.currentTarget          // Element, AUF DEM der Listener sitzt
      event.type                   // "click", "keydown", ...
  });

  // Tastatur-Events — wichtige Properties:
  document.addEventListener('keydown', e => {
      e.key        // "Enter", "a", "Escape", "ArrowDown", ...
      e.code       // physische Taste: "KeyA", "Enter", ...
      e.ctrlKey, e.shiftKey, e.altKey, e.metaKey   // Modifier-Tasten
  });

  // Maus-Events — wichtige Properties:
  el.addEventListener('mousemove', e => {
      e.clientX, e.clientY   // Position relativ zum BROWSERFENSTER
      e.offsetX, e.offsetY   // Position relativ zum ELEMENT
  });

------------------------------------------------------------------------
5.3  Häufige Events — Übersicht
------------------------------------------------------------------------

  MAUS:
    click, dblclick, contextmenu (Rechtsklick)
    mouseenter, mouseleave   → NICHT bubbling, pro Element genau 1x
    mouseover, mouseout      → bubbling, können mehrfach feuern
    mousedown, mouseup, mousemove

    // Aus deinem placeholder.js — genau dieses Paar:
    el.addEventListener('mouseenter', () => { /* Hover-Canvas zeigen */ });
    el.addEventListener('mouseleave', () => { /* Normal-Canvas zeigen */ });

  TASTATUR:
    keydown   → bei JEDEM Tastendruck, wiederholt sich bei Halten
    keyup     → wenn Taste losgelassen wird
    keypress  → veraltet, nicht mehr verwenden

  FORMULAR (Details in Modul 7):
    input    → bei JEDER Änderung (auch Buchstabe für Buchstabe)
    change   → wenn Feld den FOKUS verliert UND sich der Wert geändert hat
    submit   → wenn das <form> abgeschickt wird
    focus, blur → Feld bekommt/verliert den Fokus
    invalid  → Browser-Validierung schlägt fehl

  FENSTER/DOKUMENT:
    DOMContentLoaded  → DOM fertig aufgebaut (siehe 5.4)
    load              → WIRKLICH ALLES fertig (Bilder, Videos, ...)
    resize            → Fenstergröße geändert
    scroll            → gescrollt — Achtung: feuert SEHR oft (→ throttle,
                        Modul 15!)
    visibilitychange  → Tab wird sichtbar/unsichtbar — genau das nutzt
                        placeholder.js über document.hidden, um die
                        Animation zu pausieren, wenn der Tab im
                        Hintergrund ist.

  DRAG & DROP:
    dragstart, dragover, drop, dragend

------------------------------------------------------------------------
5.4  DOMContentLoaded vs. window.load
------------------------------------------------------------------------

  // DOMContentLoaded: HTML geparst, DOM fertig — Bilder/CSS evtl. noch
  // nicht. Der häufigste Ort, um JS-Code zu "starten":
  document.addEventListener('DOMContentLoaded', () => {
      // DOM-Elemente sind jetzt sicher verfügbar
  });

  // window.load: WIRKLICH alles fertig (Bilder, CSS, Schriften, ...)
  window.addEventListener('load', () => {
      // Komplett geladen
  });

  // Aus deinem volume.js — BEIDE werden bewusst eingesetzt!
  document.addEventListener('DOMContentLoaded', () => {
      applyVolume(getStoredVol());
  });
  window.addEventListener('load', () => applyVolume(getStoredVol()));
  // Warum beide? DOMContentLoaded greift früh (Slider reagiert sofort),
  // load greift NOCHMAL als Sicherheitsnetz — falls <audio>/<video>-
  // Elemente erst durch spätere Skripte (placeholder.js!) entstehen und
  // beim ersten Durchlauf noch nicht existierten.

------------------------------------------------------------------------
5.5  Optionen für addEventListener — once, passive, und Entfernen
------------------------------------------------------------------------

  // once — Listener läuft genau EINMAL, wird danach automatisch entfernt:
  document.addEventListener('click', () => v.play(), { once: true });
  // Aus deinem placeholder.js — der "Sicherheits-Play für Firefox":
  // sobald EINMAL irgendwo geklickt wurde, startet das Video, und der
  // Listener verschwindet wieder.

  // passive — sagt dem Browser "ich rufe NIEMALS preventDefault auf",
  // erlaubt flüssigeres Scrollen bei touch/wheel-Events:
  el.addEventListener('touchstart', handler, { passive: true });

  // Listener entfernen — braucht eine Referenz auf DIESELBE Funktion!
  const handler = () => console.log("Klick");
  btn.addEventListener('click', handler);
  btn.removeEventListener('click', handler);
  // Funktioniert NICHT mit anonymen Arrow-Funktionen direkt inline —
  // jede neue Arrow-Funktion ist eine ANDERE Funktion, auch wenn der
  // Code identisch aussieht. Für removeEventListener brauchst du eine
  // benannte Referenz wie oben.

------------------------------------------------------------------------
5.6  Event Delegation — effizienter als 1000 Listener
------------------------------------------------------------------------

Statt auf JEDEM Kind-Element einen eigenen Listener zu registrieren,
registrierst du EINEN auf dem Elternelement und prüfst event.target:

  // Ineffizient — UND bricht bei später hinzugefügten Elementen:
  document.querySelectorAll('.btn').forEach(btn => {
      btn.addEventListener('click', handler);
  });

  // Effizient — EIN Listener auf dem Elternelement:
  document.getElementById('container').addEventListener('click', e => {
      if (e.target.matches('.btn')) {
          handler(e);
      }
  });
  // Funktioniert AUCH für Elemente, die SPÄTER per JS hinzugefügt
  // werden — der Listener sitzt ja am Container, nicht am Kind!

Kombiniert mit data-* (siehe 3.4) — ein sehr verbreitetes Pattern:

  document.getElementById('projektliste').addEventListener('click', e => {
      const karte = e.target.closest('.projekt-karte');
      if (!karte) return;  // Klick war nicht auf/in einer Karte
      const id = karte.dataset.projektId;
      zeigeDetails(id);
  });
  // .closest() sucht VOM GEKLICKTEN ELEMENT aus nach OBEN, bis es ein
  // Element findet, das auf den Selektor passt — perfekt, wenn man auf
  // VERSCHACHTELTE Elemente innerhalb einer "Karte" klickt (z.B. auf
  // ein <img> oder <span> INNERHALB der Karte).

------------------------------------------------------------------------
5.7  Eigene Events — CustomEvent und dispatchEvent
------------------------------------------------------------------------

Du kannst nicht nur auf Browser-Events reagieren, sondern auch EIGENE
Events erzeugen und "senden" — nützlich, um Komponenten voneinander zu
entkoppeln (das "Observer Pattern" aus Modul 20 baut genau darauf auf).

  // Eigenes Event mit Zusatzdaten erstellen:
  const event = new CustomEvent('lautstaerke-geaendert', {
      detail: { wert: 0.7 },
      bubbles: true   // das Event "blubbert" zu Elternelementen hoch
  });

  // Senden:
  slider.dispatchEvent(event);

  // Empfangen — wie jedes andere Event:
  document.addEventListener('lautstaerke-geaendert', e => {
      console.log('Neue Lautstärke:', e.detail.wert);
  });

Idee für TrafkSite: volume.js könnte beim Ändern der Lautstärke ein
'lautstaerke-geaendert'-Event feuern. Dann müsste KEIN anderes Skript
wissen, WIE volume.js intern funktioniert — es müsste nur auf dieses
Event lauschen (z.B. um eine Mute-Anzeige im Header zu aktualisieren,
ohne dass volume.js diese Anzeige überhaupt kennt).

------------------------------------------------------------------------
5.8  Praxis: klassische Website-Interaktionsmuster
------------------------------------------------------------------------

Mit den Werkzeugen aus Modul 3 (DOM) und diesem Modul kannst du bereits
die meisten "Standard-Komponenten" einer Website bauen. Hier die
Grundmuster — du baust sie in der Übung selbst:

  AKKORDEON / ausklappbarer Bereich:
    button.addEventListener('click', () => {
        inhalt.classList.toggle('open');
        button.setAttribute('aria-expanded', inhalt.classList.contains('open'));
    });
    /* CSS: .inhalt { display: none; } .inhalt.open { display: block; } */
    // Das aria-expanded-Attribut ist ein Barrierefreiheits-Hinweis für
    // Screenreader (mehr dazu im HTML-Kurs, Modul "Barrierefreiheit").

  TABS:
    tabButtons.forEach(btn => {
        btn.addEventListener('click', () => {
            // zuerst ALLE deaktivieren...
            tabButtons.forEach(b => b.classList.remove('aktiv'));
            panels.forEach(p => p.classList.remove('aktiv'));
            // ...dann genau EINEN aktivieren
            btn.classList.add('aktiv');
            document.getElementById(btn.dataset.tab).classList.add('aktiv');
        });
    });

  MODAL/LIGHTBOX:
    openBtn.addEventListener('click', () => modal.classList.add('offen'));
    closeBtn.addEventListener('click', () => modal.classList.remove('offen'));
    modal.addEventListener('click', e => {
        if (e.target === modal) modal.classList.remove('offen');
        // Klick auf den dunklen Hintergrund (NICHT den Inhalt) schließt
    });
    document.addEventListener('keydown', e => {
        if (e.key === 'Escape') modal.classList.remove('offen');
    });

================================================================================
MODUL 6: Arrays und Objekte — Daten strukturieren
================================================================================

Webseiten sind voller Listen (Navigationspunkte, Projekte, Chat-Verlauf,
Suchergebnisse) und voller "Dinge mit Eigenschaften" (ein Nutzer, ein
Projekt, eine Chat-Nachricht). Arrays und Objekte sind genau dafür da.

------------------------------------------------------------------------
6.1  Arrays — geordnete Listen
------------------------------------------------------------------------

  const projekte = ["RSA", "TrafkSite", "TrafkTux"];

  projekte[0]            → "RSA"
  projekte.length         → 3
  projekte[projekte.length - 1]  → "TrafkTux"   (letztes Element)

  // Hinzufügen/Entfernen am Ende/Anfang:
  projekte.push("Neu");        // ans Ende
  projekte.pop();              // letztes entfernen, gibt es zurück
  projekte.unshift("Erstes");  // an den Anfang
  projekte.shift();            // erstes entfernen, gibt es zurück

  // splice — Allzweck-Werkzeug zum Einfügen/Entfernen MITTEN im Array:
  projekte.splice(1, 0, "Eingefügt");  // bei Index 1, 0 entfernen, "Eingefügt" einfügen
  projekte.splice(1, 1);               // bei Index 1, 1 Element entfernen

  // slice — Kopie eines Ausschnitts (verändert das Original NICHT!):
  projekte.slice(0, 2)   → ["RSA", "TrafkSite"]
  projekte.slice(-1)     → ["TrafkTux"]  (letztes Element, als Array)

  // Suchen:
  projekte.includes("RSA")     → true
  projekte.indexOf("TrafkTux")  → 2
  projekte.find(p => p.startsWith("Trafk"))   → "TrafkSite" (erster Treffer)
  projekte.findIndex(p => p === "TrafkTux")   → 2

  // Sortieren — ACHTUNG: verändert das Original-Array!
  [3, 1, 2].sort()                → [1, 2, 3]
  [3, 1, 2].sort((a, b) => b - a) → [3, 2, 1]   (absteigend)
  // Für eine SORTIERTE KOPIE (modern): [...arr].sort(...) oder arr.toSorted(...)

------------------------------------------------------------------------
6.2  map, filter, reduce, forEach — die "Daten-Pipeline"
------------------------------------------------------------------------

Diese vier Methoden sind das Rückgrat moderner JS-Programmierung. Statt
klassischer for-Schleifen mit Index-Variable beschreibst du, WAS mit
JEDEM Element passieren soll — JS übernimmt das "Wie".

  forEach — für JEDES Element etwas TUN (kein Rückgabewert):
    projekte.forEach(p => console.log(p));
    // Aus deinem nav.js, placeholder.js: ÜBERALL forEach.

  map — JEDES Element in etwas NEUES UMWANDELN (gleiche Länge!):
    const grossbuchstaben = projekte.map(p => p.toUpperCase());
    → ["RSA", "TRAFKSITE", "TRAFKTUX"]

    // Aus deinem nav.js — URLs zu lesbaren Texten umwandeln:
    const urls = Array.from(xml.querySelectorAll("loc"))
        .map(el => el.textContent.trim())
        .filter(u => u.endsWith('.html'));
    // map: <loc>-Element → Text. filter: nur .html-URLs behalten.
    // Das ist eine KETTE — map gibt ein neues Array zurück, auf dem
    // sofort .filter aufgerufen wird. Kein Zwischenspeichern nötig!

  filter — NUR Elemente behalten, die eine Bedingung erfüllen:
    const trafkProjekte = projekte.filter(p => p.startsWith("Trafk"));
    → ["TrafkSite", "TrafkTux"]

  reduce — ALLE Elemente zu EINEM einzigen Wert "zusammenfalten":
    const gesamtlaenge = projekte.reduce((summe, p) => summe + p.length, 0);
    // (summe, aktuellesElement) => neueSumme, Startwert: 0
    // Durchlauf: 0+3=3, 3+9=12, 12+8=20 → 20

    // Häufiges Beispiel: Objekt aus Array bauen (z.B. für schnelles
    // Nachschlagen per ID statt Durchsuchen):
    const nachId = projekte.reduce((acc, p, i) => {
        acc[i] = p;
        return acc;
    }, {});

Verkettung — der eigentliche Grund, warum diese Methoden so mächtig sind:

  const ergebnis = chatHistory
      .filter(m => m.role === 'user')      // nur Nutzer-Nachrichten
      .map(m => m.content.toUpperCase())   // alle in Großbuchstaben
      .slice(-5);                          // nur die letzten 5

  Jede Methode gibt ein NEUES Array zurück — das Original bleibt
  UNVERÄNDERT (Ausnahme: sort, push, pop, splice, unshift, shift
  verändern das Original DIREKT — "mutierend").

------------------------------------------------------------------------
6.3  Objekte — benannte Eigenschaften
------------------------------------------------------------------------

  const projekt = {
      name: "TrafkTux",
      jahr: 2026,
      aktiv: true,
      tags: ["Spiel", "Linux"],
      autor: { name: "Daniel Raberger", rolle: "HoppiTex" }
  };

  // Zugriff — zwei Schreibweisen:
  projekt.name           → "TrafkTux"        (Punkt-Notation)
  projekt["name"]        → "TrafkTux"        (Klammer-Notation)
  projekt["jahr-info"]   → nur Klammer möglich bei Sonderzeichen im Key!

  // Verschachtelt:
  projekt.autor.name     → "Daniel Raberger"

  // Property SETZEN/ÄNDERN — geht auch bei const! const verbietet nur
  // das NEU-ZUWEISEN der VARIABLEN selbst, nicht das Verändern des
  // Objekt-INHALTS:
  projekt.jahr = 2027;        // ✓ erlaubt
  projekt.neuesFeld = "x";    // ✓ erlaubt — fügt neue Property hinzu
  projekt = {};               // ✗ FEHLER — const darf nicht neu zugewiesen werden

  // Existenz prüfen:
  'tags' in projekt              → true
  projekt.hasOwnProperty('tags') → true
  projekt.nichtVorhanden         → undefined  (kein Fehler! aber: Vorsicht
                                                bei .irgendwas.nochwas →
                                                "Cannot read properties of
                                                undefined" — siehe 2.7
                                                Optional Chaining!)

  // Object-Methoden:
  Object.keys(projekt)    → ["name", "jahr", "aktiv", "tags", "autor"]
  Object.values(projekt)  → ["TrafkTux", 2026, true, [...], {...}]
  Object.entries(projekt) → [["name","TrafkTux"], ["jahr",2026], ...]

  for (const [key, value] of Object.entries(projekt)) {
      console.log(`${key}: ${value}`);
  }

------------------------------------------------------------------------
6.4  Destructuring — Werte "auspacken"
------------------------------------------------------------------------

  // Array-Destructuring:
  const [erstes, zweites, ...rest] = projekte;
  // erstes = "RSA", zweites = "TrafkSite", rest = ["TrafkTux"]

  // Swap zweier Variablen — der klassische "Trick":
  let a = 1, b = 2;
  [a, b] = [b, a];   // a=2, b=1

  // Objekt-Destructuring:
  const { name, jahr } = projekt;
  // name = "TrafkTux", jahr = 2026

  // Mit Umbenennung und Default:
  const { name: projektName, beschreibung = "Keine Beschreibung" } = projekt;
  // projektName = "TrafkTux", beschreibung = "Keine Beschreibung" (Default!)

  // Verschachtelt:
  const { autor: { name: autorName } } = projekt;
  // autorName = "Daniel Raberger"

  // In Funktionsparametern — siehst du SEHR oft (siehe auch 4.2):
  function zeigeProjekt({ name, jahr, tags = [] }) {
      console.log(`${name} (${jahr}): ${tags.join(', ')}`);
  }

------------------------------------------------------------------------
6.5  Spread-Operator (...) — Kopieren und Zusammenführen
------------------------------------------------------------------------

  // Array kopieren (FLACHE Kopie!) und erweitern:
  const alle = [...projekte, "Neu1", "Neu2"];
  // ["RSA", "TrafkSite", "TrafkTux", "Neu1", "Neu2"] — projekte bleibt unverändert

  // Objekt kopieren und ERWEITERN/ÜBERSCHREIBEN:
  const projektV2 = { ...projekt, jahr: 2027, neu: true };
  // Alle Properties von projekt, ABER jahr ist jetzt 2027, plus neu: true
  // SPÄTERE Properties überschreiben FRÜHERE — Reihenfolge zählt!

  // Mehrere Arrays/Objekte zusammenführen:
  const a1 = [1, 2], a2 = [3, 4];
  const zusammen = [...a1, ...a2];   → [1, 2, 3, 4]

  const obj1 = { a: 1 }, obj2 = { b: 2 };
  const merged = { ...obj1, ...obj2 };  → { a: 1, b: 2 }

  // FLACHE Kopie — ACHTUNG bei verschachtelten Objekten/Arrays!
  const original = { tags: ["a", "b"] };
  const kopie = { ...original };
  kopie.tags.push("c");
  original.tags  → ["a", "b", "c"]  ← AUCH original wurde verändert!
  // Grund: tags ist ein ARRAY (also ein Objekt) — die Kopie enthält nur
  // eine REFERENZ auf dasselbe Array, kein eigenständiges Duplikat. Für
  // eine "tiefe" Kopie (alle Ebenen unabhängig): structuredClone(original)
  // (modern, eingebaut) oder JSON.parse(JSON.stringify(original)) (älter,
  // funktioniert NICHT mit Funktionen/undefined/Date in den Daten).

6.6  Map und Set — Alternativen zu Objekt und Array
------------------------------------------------------------------------

  Map — wie ein Objekt, aber:
    - JEDER Wert kann Key sein (nicht nur Strings)
    - behält Einfüge-Reihenfolge garantiert
    - .size statt .length, eigene Methoden

    const cache = new Map();
    cache.set('TrafkTux', { jahr: 2026 });
    cache.get('TrafkTux')      → { jahr: 2026 }
    cache.has('TrafkTux')      → true
    cache.delete('TrafkTux');
    cache.size                  → 0

    // Sinngemäß denkbar im Stil von AskAlfonz.js — eine Priority-Map
    // für die Sitemap-Generierung:
    const urlPriorityMap = new Map();
    urlPriorityMap.set('/index.html', 1.0);
    urlPriorityMap.get('/index.html') ?? 0.65   // Fallback mit ??, siehe 2.7

  Set — eine Liste OHNE Duplikate:
    const tags = new Set(["Spiel", "Linux", "Spiel"]);
    tags.size            → 2  (Duplikat "Spiel" automatisch entfernt!)
    tags.add("Neu");
    tags.has("Linux")    → true

    // Praktisch: Array ohne Duplikate erzwingen:
    const eindeutig = [...new Set([1, 2, 2, 3, 3, 3])];  → [1, 2, 3]

6.7  JSON — Daten zwischen JS, Server und Speicher austauschen
------------------------------------------------------------------------

JSON (JavaScript Object Notation) ist das STANDARDFORMAT, um Daten als
TEXT zu transportieren — zwischen Browser und Server (fetch, Modul 8),
oder um Objekte im localStorage (Modul 9) zu speichern, das ja nur
Strings kann.

  // JS-Objekt → JSON-String:
  JSON.stringify({ name: "Alfonz", aktiv: true })
  → '{"name":"Alfonz","aktiv":true}'

  // JSON-String → JS-Objekt:
  JSON.parse('{"name":"Alfonz","aktiv":true}')
  → { name: "Alfonz", aktiv: true }

  // Aus deinem index.js — der Server schickt JSON zurück:
  res.json(response.data);
  // Express wandelt response.data automatisch in JSON um UND setzt den
  // passenden Content-Type-Header.

  // Aus deinem AskAlfonz.js — fetch-Antwort als JSON lesen:
  const data = await res.json();   // (await/Promises: Modul 8)

  // Pretty-Print für Debugging (3. Argument = Einrückung):
  JSON.stringify(projekt, null, 2)
  /* {
       "name": "TrafkTux",
       "jahr": 2026,
       ...
     } */

WICHTIG: JSON kennt KEINE Funktionen, undefined, oder Kommentare! Beim
Stringify werden solche Werte ENTFERNT (bei Objekten) oder zu null
(in Arrays). Auch Date-Objekte werden zu Strings — beim Parse bleiben sie
Strings, du musst sie manuell wieder in new Date(...) umwandeln.

MODUL 7: Formulare — Eingaben verarbeiten und validieren
================================================================================

Formulare sind die häufigste Art, wie Nutzer DATEN AN deine Website
geben — Suchfelder, Kontaktformulare, Lautstärkeregler, Chat-Eingaben.
Dieses Modul gab es im alten Kurs nicht explizit — dabei steckt hier ein
GROSSER Teil der "Alles ist ein String"-Fallen aus Modul 2.

------------------------------------------------------------------------
7.1  Formularelemente und ihre .value
------------------------------------------------------------------------

  <input type="text">      → input.value  (string)
  <input type="number">    → input.value  (STRING, auch wenn type="number"!)
  <input type="checkbox">  → input.checked  (boolean!) — NICHT .value für
                              den "angehakt"-Zustand (input.value ist bei
                              Checkboxen oft "on" oder ein fest gesetzter
                              String, unabhängig vom Haken)
  <input type="radio">     → input.checked, mehrere mit demselben "name"
                              bilden eine Gruppe — nur EINS kann checked sein
  <input type="range">     → input.value  (string — "0.5", nicht 0.5!)
  <select>                  → select.value (value des gewählten <option>)
  <textarea>                → textarea.value

  // Aus deinem footer.html + volume.js:
  <input type="range" id="volume" min="0" max="1" step="0.01" value="0.5">

  const slider = document.getElementById('volume');
  slider.value              → "0.5"           (STRING!)
  parseFloat(slider.value)  → 0.5              (number — für Berechnungen)

  slider.addEventListener('input', (e) => {
      const vol = parseFloat(e.target.value);   // ⚠ ohne parseFloat: "0.7"
      localStorage.setItem('site-volume', vol); // localStorage.setItem
                                                   // konvertiert wieder zu
                                                   // String — siehe Modul 9!
      applyVolume(vol);
  });

------------------------------------------------------------------------
7.2  input vs. change vs. submit — WANN feuert was?
------------------------------------------------------------------------

  input
    Feuert bei JEDER ÄNDERUNG, in Echtzeit — bei <input type="text">
    nach JEDEM Tastendruck, bei <input type="range"> bei JEDEM Pixel
    Bewegung des Schiebers. Perfekt für: Live-Vorschau, Live-Suche,
    Live-Lautstärkeregelung (genau wie in volume.js!).

  change
    Feuert erst, wenn das Feld den FOKUS VERLIERT (z.B. Tab oder Klick
    woanders hin) UND sich der Wert seit Fokus-Erhalt geändert hat. Bei
    <select>, <input type="checkbox/radio"> feuert es sofort bei Auswahl
    (da gibt's kein "tippen"). Gut für: Aktionen, die NICHT bei jedem
    Tastendruck sinnvoll sind (z.B. eine teure Berechnung, ein API-Call).

  submit
    Feuert auf dem <form>-ELEMENT (nicht dem Button!), wenn das Formular
    abgeschickt wird — per Klick auf <button type="submit">, ODER per
    Enter-Taste in einem Textfeld INNERHALB des Formulars.

    document.getElementById('mein-formular').addEventListener('submit', e => {
        e.preventDefault();   // verhindert Seiten-Neuladen (Standard bei
                               // <form>!) — FAST IMMER nötig bei JS-
                               // gesteuerten Formularen
        // ... eigene Logik
    });

  HÄUFIGER FEHLER: Den Listener auf den BUTTON statt das FORM legen UND
  e.preventDefault() vergessen → die Seite lädt komplett neu, alle
  JS-Variablen (chatHistory, etc.) gehen verloren!

------------------------------------------------------------------------
7.3  Validierung — drei Ebenen
------------------------------------------------------------------------

EBENE 1 — HTML-Attribute (aus dem HTML-Kurs bekannt, aber JS-relevant):

  <input type="email" required minlength="3" maxlength="50" pattern="...">

  Der Browser zeigt AUTOMATISCH eine Fehlermeldung, wenn das Formular
  abgeschickt wird und die Regel verletzt ist — OHNE eine Zeile JS. ABER:
  Diese Validierung läuft NUR beim submit, das Aussehen der Meldung ist
  je Browser unterschiedlich, und sie lässt sich leicht umgehen
  (DevTools, oder direkter API-Aufruf ohne Formular).

EBENE 2 — Live-Validierung mit JS (für sofortiges Feedback):

  const constraintApi = input.validity;
  // ValidityState-Objekt mit booleans:
  constraintApi.valid          → true/false (Gesamtergebnis)
  constraintApi.valueMissing   → true, wenn required und leer
  constraintApi.typeMismatch   → true, wenn z.B. type="email" kein @ hat
  constraintApi.tooShort        → true, wenn unter minlength
  constraintApi.patternMismatch → true, wenn pattern nicht passt

  input.checkValidity()    → true/false, löst KEIN visuelles Feedback aus
  input.reportValidity()   → true/false, zeigt Browser-Fehlermeldung

  // Live-Feedback bei jeder Eingabe:
  input.addEventListener('input', () => {
      if (input.validity.valid) {
          input.classList.remove('error');
          input.classList.add('valid');
      } else {
          input.classList.remove('valid');
          input.classList.add('error');
      }
  });
  /* CSS:
     input.error { border-color: red; }
     input.valid { border-color: green; }
  */

EBENE 3 — Eigene Logik (Dinge, die HTML-Attribute nicht können):

  function validateNachricht(text) {
      const fehler = [];
      if (!text.trim()) fehler.push("Nachricht darf nicht leer sein.");
      if (text.length > 500) fehler.push("Maximal 500 Zeichen erlaubt.");
      if (text.trim().length < 2) fehler.push("Mindestens 2 Zeichen.");
      return fehler;   // leeres Array = alles ok
  }

  // Sinngemäß für den Chat — Eingabe vor dem Senden prüfen:
  function sendMessage() {
      let text = inputField.value.trim();
      if (!text) return;   // genau wie in AskAlfonz.js — kurzer früher
                            // return statt verschachteltem if (Modul 4.4)
      // ... weiter mit dem Senden
  }

WICHTIG: Client-seitige Validierung ist KOMFORT für den Nutzer — sie
ersetzt NIEMALS Server-seitige Validierung! Jeder kann den Browser-Code
umgehen und direkt Requests an deinen Server schicken (mehr dazu in
Modul 18 — Sicherheit).

7.4  FormData — ein ganzes Formular auf einmal auslesen

Statt jedes Feld einzeln per getElementById abzufragen, kannst du das

================================================================================
MODUL 8: Asynchrones JavaScript — fetch und Promises
================================================================================

Bis jetzt lief dein Code SOFORT, Zeile für Zeile, von oben nach unten —
"synchron". Aber: ein Server antwortet nicht SOFORT. Eine Animation
braucht Zeit. Ein Timer wartet. JS braucht einen Weg, "warte auf etwas,
ohne die ganze Seite einzufrieren" — das ist ASYNCHRONES JavaScript, und
es ist die Grundlage für JEDEN Netzwerk-Request auf deiner Seite.

------------------------------------------------------------------------
8.1  Das Grundproblem — warum "einfach warten" nicht geht
------------------------------------------------------------------------

JS läuft im Browser auf EINEM einzigen Thread — es kann nur EINE Sache
gleichzeitig tun. Würde fetch() den Code "einfrieren", bis die Antwort da
ist, könnte währenddessen NICHTS auf der Seite passieren — kein Scrollen,
kein Klicken, kein Rendern. Die Seite würde "hängen".

Stattdessen: fetch() (und setTimeout, Animationen, Events, ...) sind
ASYNCHRON — sie STARTEN eine Operation, geben SOFORT die Kontrolle
zurück, und "melden sich" SPÄTER mit dem Ergebnis. In der Zwischenzeit
läuft alles andere normal weiter.

------------------------------------------------------------------------
8.2  setTimeout und setInterval — die einfachsten Async-Werkzeuge
------------------------------------------------------------------------

  setTimeout(() => console.log("3 Sekunden später"), 3000);
  console.log("Sofort!");
  // Reihenfolge der Ausgabe: "Sofort!" zuerst, DANN (nach 3s) die andere
  // Zeile — auch wenn setTimeout VOR dem console.log steht!

  const id = setInterval(() => console.log("Tick"), 1000);
  clearTimeout(timeoutId);   // geplanten Timeout abbrechen
  clearInterval(id);          // laufendes Intervall stoppen

  // Aus deinem placeholder.js — die Render-Loop nutzt KEIN setInterval,
  // sondern requestAnimationFrame (für flüssige Animationen optimiert,
  // synchronisiert mit der Bildschirm-Wiederholrate, ~60fps):
  function _drawLoop() {
      requestAnimationFrame(_drawLoop);   // ruft sich selbst wieder auf
      if (document.hidden) return;        // Tab im Hintergrund? Pause.
      // ... zeichnen
  }
  _drawLoop();   // einmal "anstoßen" — von da an läuft es selbst weiter

------------------------------------------------------------------------
8.3  Promises — ein "Versprechen" auf einen späteren Wert
------------------------------------------------------------------------

Ein Promise ist ein Objekt, das einen von DREI Zuständen hat:

  pending   → noch nicht entschieden (Operation läuft)
  fulfilled → erfolgreich abgeschlossen, hat einen WERT
  rejected  → fehlgeschlagen, hat einen FEHLER

  promise
    .then(ergebnis => { /* fulfilled */ })
    .catch(fehler => { /* rejected */ })
    .finally(() => { /* IMMER, egal ob Erfolg oder Fehler */ });

  // Aus deinem nav.js — fetch GIBT ein Promise zurück:
  fetch('sitemap.xml')
      .then(r => r.text())           // Promise → nächstes .then
      .then(str => { /* str ist jetzt der TEXT-INHALT */ })
      .catch(err => {
          console.error("Sitemap Fehler:", err);
          navContainer.innerHTML = '<p style="color:red">Fehler: ' + err.message + '</p>';
      });

  WICHTIG: .then gibt SELBST ein neues Promise zurück — deshalb kann man
  KETTEN. Der Rückgabewert von .then(callback) wird zum "Wert" des
  nächsten .then.

------------------------------------------------------------------------
8.4  async/await — Promises lesbar machen
------------------------------------------------------------------------

async/await ist SYNTAKTISCHER ZUCKER über Promises — derselbe
Mechanismus, aber liest sich wie SYNCHRONER Code:

  // Promise-Kette:
  function ladeDaten() {
      fetch('/api/daten')
          .then(r => r.json())
          .then(data => console.log(data))
          .catch(err => console.error(err));
  }

  // async/await — GLEICHE Funktionsweise, andere Syntax:
  async function ladeDaten() {
      try {
          const r = await fetch('/api/daten');
          const data = await r.json();
          console.log(data);
      } catch (err) {
          console.error(err);
      }
  }

Regeln:
  - await darf NUR innerhalb einer async-Funktion stehen.
  - await "pausiert" die Funktion, bis das Promise erfüllt ist — OHNE die
    Seite einzufrieren! Andere Funktionen/Events laufen währenddessen
    normal weiter.
  - Fehler bei await → try/catch fängt sie ab (Modul 12 vertieft das).
  - Eine async-Funktion gibt SELBST IMMER ein Promise zurück.

  // Aus deinem AskAlfonz.js — genau dieses Muster:
  async function sendMessage() {
      let text = inputField.value.trim();
      if (!text) return;
      // ... UI updaten ...
      try {
          const res = await fetch(PROXY_URL, {
              method: 'POST',
              headers: { 'Content-Type': 'application/json' },
              body: JSON.stringify({ messages: chatHistory })
          });
          const data = await res.json();
          // ... Antwort anzeigen ...
      } catch (err) {
          // ... Fehlermeldung anzeigen ...
      }
  }

------------------------------------------------------------------------
8.5  fetch() im Detail
------------------------------------------------------------------------

  fetch(url, optionen)

  // GET (Standard, Optionen können weggelassen werden):
  const res = await fetch('sitemap.xml');

  // POST mit JSON-Body — der Standard für "Daten an einen Server senden":
  const res = await fetch(PROXY_URL, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ messages: chatHistory })
  });

  // Die "response"-Objekt-Eigenschaften:
  res.ok            → true bei Status 200-299, sonst false
  res.status        → 200, 404, 500, ...
  res.statusText    → "OK", "Not Found", ...
  res.headers.get('Content-Type')

  // Body lesen — NUR EINMAL möglich! (Stream wird "verbraucht"):
  await res.text()    → roher Text
  await res.json()     → als JSON geparst (Modul 6.7)
  await res.blob()     → Binärdaten (z.B. für Bilder/Dateien)

WICHTIGER FALLSTRICK: fetch() wirft KEINEN Fehler bei HTTP-Fehlercodes
(404, 500, ...)! .catch() bzw. der catch-Block fängt nur NETZWERKFEHLER
(kein Internet, DNS-Fehler, CORS-Block). Einen 404 oder 500 musst du
SELBST prüfen:

  const res = await fetch(url);
  if (!res.ok) {
      throw new Error(`HTTP ${res.status}: ${res.statusText}`);
  }
  const data = await res.json();

------------------------------------------------------------------------
8.6  Mehrere Requests gleichzeitig — Promise.all
------------------------------------------------------------------------

  // Sequenziell (langsam — wartet auf JEDEN einzeln):
  const a = await fetch('/api/a').then(r => r.json());
  const b = await fetch('/api/b').then(r => r.json());
  // Gesamtzeit ≈ Zeit(a) + Zeit(b)

  // Parallel mit Promise.all (schnell — alle gleichzeitig):
  const [a, b] = await Promise.all([
      fetch('/api/a').then(r => r.json()),
      fetch('/api/b').then(r => r.json())
  ]);
  // Gesamtzeit ≈ max(Zeit(a), Zeit(b))

  // Promise.all bricht SOFORT ab, wenn EIN Request fehlschlägt (auch
  // wenn andere noch laufen). Promise.allSettled wartet auf ALLE,
  // egal ob fulfilled oder rejected — gibt für jedes ein
  // { status, value/reason }-Objekt zurück. Praktisch, wenn ein
  // einzelner fehlschlagender Request die restlichen nicht "vergiften"
  // soll.

------------------------------------------------------------------------
8.7  Praxis: Lade- und Fehlerzustände auf der Seite anzeigen
------------------------------------------------------------------------

Ein zentrales Muster für JEDE Webseite, die Daten nachlädt:

  async function ladeProjekte() {
      const container = document.getElementById('projekte');
      container.innerHTML = '<p class="lade">Lädt...</p>';   // 1. Ladezustand

      try {
          const res = await fetch('/api/projekte');
          if (!res.ok) throw new Error(`HTTP ${res.status}`);
          const projekte = await res.json();

          if (projekte.length === 0) {
              container.innerHTML = '<p>Keine Projekte gefunden.</p>';  // 2. Leer
              return;
          }

          // 3. Erfolg — Inhalt rendern (Modul 3 + 6):
          container.innerHTML = '';
          projekte.forEach(p => {
              const div = document.createElement('div');
              div.textContent = p.name;
              container.appendChild(div);
          });

      } catch (err) {
          container.innerHTML = `<p class="fehler">Fehler: ${err.message}</p>`; // 4. Fehler
      }
  }

DREI Zustände, die JEDE asynchrone UI-Operation behandeln sollte:
LADEN → ERFOLG (ggf. mit Unterfall "leer") → FEHLER. Genau dieses Muster
findest du (etwas erweitert) in AskAlfonz.js — z.B. ein "Lädt..."-Element
per ID, das nach Erhalt der Antwort wieder entfernt wird:
document.getElementById(loadingId)?.remove(); (Optional Chaining, Modul 2.7
— falls das Element schon weg ist, kein Crash).

================================================================================
MODUL 9: Storage, Cookies, URL & History
================================================================================

Eine Webseite ist von Natur aus "vergesslich" — jeder Seitenaufruf
startet bei null. Dieses Modul zeigt die Werkzeuge, mit denen Webseiten
sich trotzdem ETWAS merken — von "letzte Lautstärke" (volume.js!) bis
"in welchem Tab war ich zuletzt".

------------------------------------------------------------------------
9.1  localStorage und sessionStorage
------------------------------------------------------------------------

Beide speichern KEY-VALUE-Paare, NUR ALS STRINGS, pro ORIGIN (Protokoll +
Domain + Port — trafkhop-entertainment.github.io hat einen eigenen
Speicher, getrennt von z.B. localhost).

  localStorage    → bleibt, bis EXPLIZIT gelöscht (auch nach Browser-
                     Neustart, Tage/Wochen/Monate später noch da)
  sessionStorage  → wird gelöscht, wenn der TAB geschlossen wird

  // Setzen — KONVERTIERT automatisch zu String!
  localStorage.setItem('site-volume', 0.7);
  localStorage.getItem('site-volume')   → "0.7"   (STRING, nicht 0.7!)

  // Aus deinem volume.js — genau dieser Ablauf:
  localStorage.setItem('site-volume', vol);              // vol war number
  const getStoredVol = () =>
      parseFloat(localStorage.getItem('site-volume') ?? '0.5');
  //              ^ wieder zu number   ^ Fallback als STRING '0.5'
  //                (wichtig — parseFloat('0.5') funktioniert, parseFloat(0.5)
  //                 würde auch funktionieren, aber konsistent: alles String)

  // Existenz prüfen / Entfernen:
  localStorage.getItem('nicht-vorhanden')   → null  (NICHT undefined!)
  localStorage.removeItem('site-volume');
  localStorage.clear();   // ALLES löschen — Vorsicht!

  // Objekte speichern — über JSON (Modul 6.7)!
  localStorage.setItem('einstellungen', JSON.stringify({ theme: 'dark', vol: 0.7 }));
  const einstellungen = JSON.parse(localStorage.getItem('einstellungen') ?? '{}');
  // ?? '{}' als Fallback — sonst würde JSON.parse(null) crashen!

  // sessionStorage — IDENTISCHE API, nur anderer Lebenszyklus:
  sessionStorage.setItem('aktiverTab', 'projekte');

HINWEIS für ARTIFACTS (falls du in Claude-Artefakten experimentierst):
localStorage/sessionStorage funktionieren dort NICHT — innerhalb dieses
Kurses und auf deiner echten GitHub-Pages-Seite (trafkhop-entertainment.
github.io) funktionieren sie aber ganz normal, wie hier beschrieben.

------------------------------------------------------------------------
9.2  Cookies — der ältere, aber teils noch nötige Mechanismus
------------------------------------------------------------------------

Cookies sind älter als localStorage und werden bei JEDEM Request
AUTOMATISCH an den SERVER mitgeschickt (localStorage NIE!) — relevant,
wenn der Server selbst etwas über den Nutzer wissen muss (z.B. Login-
Sessions).

  // Lesen — document.cookie ist EIN String, "key1=val1; key2=val2":
  document.cookie
  → "theme=dark; lang=de"

  // Setzen — JEDER Aufruf setzt/überschreibt NUR DAS EINE genannte
  // Cookie, NICHT den gesamten String:
  document.cookie = "theme=dark; max-age=3600; path=/";
  //                  ^name=wert  ^Gültigkeit in Sek.  ^für welche Pfade

  // Eigene Helper-Funktionen sind praktisch üblich, da die Roh-API
  // umständlich ist:
  function getCookie(name) {
      const match = document.cookie.match(new RegExp('(^| )' + name + '=([^;]+)'));
      return match ? match[2] : null;
  }
  // (RegExp wird in Modul 10 erklärt!)

  // Löschen — Trick: max-age auf 0 oder negative Zahl setzen:
  document.cookie = "theme=; max-age=0; path=/";

WICHTIGE Cookie-Attribute (Sicherheitsrelevant — Modul 18 vertieft das):

  Secure     → Cookie wird NUR über HTTPS gesendet
  HttpOnly   → Cookie ist von JS GAR NICHT lesbar (nur Server kann es
                setzen/lesen) — kann NUR vom SERVER per HTTP-Header
                gesetzt werden, NICHT per document.cookie!
  SameSite   → steuert, ob das Cookie bei Requests von ANDEREN Seiten
                mitgeschickt wird (Schutz gegen CSRF-Angriffe)

  Für TrafkSite (statisches GitHub Pages) sind Cookies aktuell kaum
  relevant — localStorage reicht für "Lautstärke merken" völlig aus, weil
  der Server (GitHub Pages) diese Info gar nicht braucht.

9.3  URL und URLSearchParams — die Adresse als Datenquelle
------------------------------------------------------------------------

  location.href          → komplette URL, z.B.
                            "https://trafkhop-entertainment.github.io/TrafkSite/index.html?ref=nav#Projects"
  location.protocol       → "https:"
  location.host            → "trafkhop-entertainment.github.io"
  location.pathname        → "/TrafkSite/index.html"
  location.search          → "?ref=nav"
  location.hash             → "#Projects"

URLSearchParams — Query-Parameter LESEN und SCHREIBEN, ohne selbst Strings
zu zerlegen:

  // Aktuelle URL: ...?suche=trafktux&sort=neu
  const params = new URLSearchParams(location.search);
  params.get('suche')      → "trafktux"
  params.get('sort')       → "neu"
  params.has('suche')      → true
  params.get('fehlt')      → null

  // Neue Parameter setzen (z.B. um einen "teilbaren Link" zu bauen):
  params.set('sort', 'alt');
  params.toString()        → "suche=trafktux&sort=alt"

  // Eine Such-URL aus einem Eingabefeld bauen — denkbar für #search im
  // Header (header.html):
  const suchbegriff = document.getElementById('search').value;
  const url = new URL('such-seite.html', location.origin);
  url.searchParams.set('q', suchbegriff);
  location.href = url.toString();   // Navigiert zur neuen URL

  // new URL(...) — relative URLs sicher zu vollständigen machen:
  new URL('/TrafkSite/projects/TrafkTux/TrafkTux.html', location.origin).href
  → "https://trafkhop-entertainment.github.io/TrafkSite/projects/TrafkTux/TrafkTux.html"

  // Aus deinem nav.js wird new URL bereits genutzt:
  const u = new URL(url);
  const parts = u.pathname.split('/').filter(Boolean).slice(1);
  // .pathname extrahiert NUR den Pfad-Teil — egal, ob die URL absolut
  // oder relativ zur Sitemap-Domain war.

9.4  History API — Navigation ohne Neuladen
------------------------------------------------------------------------

  history.back()       // wie der "Zurück"-Button
  history.forward()     // wie "Vorwärts"
  history.go(-2)         // 2 Schritte zurück

  // Die URL ändern, OHNE die Seite neu zu laden — Grundlage von
  // "Single Page Applications" (vereinfacht):
  history.pushState({ tab: 'projekte' }, '', '?tab=projekte');
  // Browser-Adresse ändert sich, Seite lädt NICHT neu, ein neuer
  // History-Eintrag entsteht ("Zurück"-Button geht dahin zurück).

  history.replaceState({ tab: 'mitglieder' }, '', '?tab=mitglieder');
  // Wie pushState, ABER ersetzt den AKTUELLEN Eintrag (kein neuer
  // "Zurück"-Schritt entsteht).

  // Auf "Zurück"/"Vorwärts" reagieren:
  window.addEventListener('popstate', e => {
      console.log('Neuer State:', e.state);   // { tab: 'projekte' }, etc.
      // → hier würdest du z.B. den richtigen Tab wieder aktivieren
  });

Für eine Seite wie TrafkSite (mehrere echte .html-Dateien, GitHub Pages)

wenn du z.B. Tabs/Filter SO bauen willst, dass ihr Zustand in der URL

9.5  Vergleich: Wann was benutzen?

  localStorage     → Einstellungen, die der NUTZER lokal behalten soll

  History API        → Navigation/Zustand OHNE Neuladen, mit funktionie-

================================================================================
MODUL 10: Reguläre Ausdrücke (Regex)
================================================================================

Regex sind "Muster" zum Suchen, Prüfen und Ersetzen von Text — extrem
mächtig für Formularvalidierung (Modul 7), URL-Manipulation (Modul 9) und
das Säubern von Eingaben. Du hast sie in deinem Bash-Kurs vermutlich schon
in grep/sed gesehen — die Grundsyntax ist sehr ähnlich.

------------------------------------------------------------------------
10.1  Regex erstellen und testen
------------------------------------------------------------------------

  // Literal-Syntax — am häufigsten:
  const muster = /hallo/;

  // Mit Flags:
  const muster2 = /hallo/gi;
  //                 ^g = global (alle Treffer, nicht nur ersten)
  //                  ^i = ignore case (Groß-/Kleinschreibung egal)
  //               m = multiline, s = dotAll (. matcht auch Zeilenumbrüche)

  // Aus einem String (z.B. wenn das Muster aus einer Variable kommt —
  // wichtig für getCookie aus Modul 9!):
  const name = "theme";
  const muster3 = new RegExp('(^| )' + name + '=([^;]+)');

  // Testen — gibt true/false:
  /hallo/i.test("Hallo Welt")    → true

  // Treffer finden:
  "Hallo Welt".match(/[A-Z]\w+/g)   → ["Hallo", "Welt"]

------------------------------------------------------------------------
10.2  Wichtige Muster-Bausteine
------------------------------------------------------------------------

  .         → ein beliebiges Zeichen (außer Zeilenumbruch, ohne /s)
  \d        → eine Ziffer (0-9)        \D → KEINE Ziffer
  \w        → Wortzeichen (a-z, A-Z, 0-9, _)   \W → kein Wortzeichen
  \s        → Whitespace (Leerzeichen, Tab, Zeilenumbruch)   \S → kein Whitespace
  ^         → Anfang des Strings (oder Zeile, mit /m)
  $         → Ende des Strings (oder Zeile, mit /m)
  [abc]     → eines von a, b, c
  [^abc]    → KEINS von a, b, c
  [a-z]     → Bereich a bis z
  a|b       → a ODER b
  (...)     → Gruppe (kann später referenziert werden)
  *         → 0 oder mehr (gierig)
  +         → 1 oder mehr
  ?         → 0 oder 1 (macht vorheriges Element OPTIONAL)
  {3}       → genau 3-mal
  {2,5}     → 2 bis 5-mal
  {2,}      → mindestens 2-mal

  Sonderzeichen ESCAPEN mit \ wenn sie WÖRTLICH gemeint sind:
  \. \/ \( \) \[ \] \{ \} \+ \* \? \^ \$ \|

------------------------------------------------------------------------
10.3  Praktische Beispiele — Formularvalidierung & mehr
------------------------------------------------------------------------

  // E-Mail (EINFACHE Prüfung — für "echte" Validierung lieber
  // type="email" + Server-Check, siehe Modul 7 und 18!):
  const emailMuster = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
  emailMuster.test("max@example.com")   → true
  emailMuster.test("ungültig")           → false

  // Nur Buchstaben und Zahlen (z.B. für einen "Benutzername"):
  /^[a-zA-Z0-9]+$/.test("Trafk2026")     → true
  /^[a-zA-Z0-9]+$/.test("Trafk 2026")    → false  (Leerzeichen!)

  // Leerzeichen durch %20 ersetzen — wie in deinem Bash-Kurs
  // ("${game// /%20}"), aber in JS:
  "Hallo Welt Foo".replace(/ /g, "%20")
  → "Hallo%20Welt%20Foo"
  // OHNE /g würde NUR das ERSTE Leerzeichen ersetzt!

  // Aus deinem AskAlfonz.js — typische "Bereinigungs"-Aufgabe:
  // Mehrere aufeinanderfolgende Leerzeichen/Zeilenumbrüche zu EINEM
  // Leerzeichen zusammenfassen:
  "Hallo   Welt\n\nFoo".replace(/\s+/g, " ")
  → "Hallo Welt Foo"

  // Gruppen + replace mit "$1", "$2" — Teile eines Treffers
  // wiederverwenden:
  "2026-06-14".replace(/(\d{4})-(\d{2})-(\d{2})/, "$3.$2.$1")
  → "14.06.2026"   (von ISO-Datum zu deutschem Format)

  // matchAll — ALLE Treffer MIT Gruppen-Details (modern):
  const text = "Projekt: RSA (2024), Projekt: TrafkSite (2026)";
  for (const m of text.matchAll(/Projekt: (\w+) \((\d{4})\)/g)) {
      console.log(m[1], m[2]);   // "RSA" "2024", dann "TrafkSite" "2026"
  }
  // m[0] = der GESAMTE Treffer, m[1]/m[2]/... = die Gruppen in ()

------------------------------------------------------------------------
10.4  Wann Regex, wann String-Methoden?
------------------------------------------------------------------------

Für SIMPLE Fälle sind String-Methoden (Modul 2.3) oft lesbarer:

  text.includes("Spam")        statt   /spam/i.test(text)  (für EXAKTEN
                                         String — bei "egal Groß/Klein"
                                         ist Regex hier kürzer)
  text.toLowerCase().includes("spam")   ← oft die bessere Wahl für
                                          case-insensitive EXAKTEN
                                          Teilstring-Check

Regex lohnt sich, wenn:
  - das Muster VARIABEL ist (z.B. "eine Zahl, gefolgt von px"),
  - mehrere ALTERNATIVEN gleichzeitig geprüft werden sollen,
  - du etwas ERSETZEN willst, basierend auf einem MUSTER statt einem
    exakten String,
  - du Teile eines Treffers EXTRAHIEREN willst (Gruppen).

Faustregel: Wenn dein Regex-Ausdruck länger ist als die String-Methoden-
Alternative UND nicht deutlich mehr leistet — nimm die String-Methoden.
Lesbarkeit zählt.

================================================================================
MODUL 11: Klassen und Objektorientierung
================================================================================

Bisher hast du einzelne Funktionen und "lose" Objekte gesehen. Klassen
sind BAUPLÄNE für Objekte, die GLEICHE Struktur und GLEICHES Verhalten
haben — z.B. "eine Chat-Nachricht", "ein Projekt", "ein Modal-Dialog".

------------------------------------------------------------------------
11.1  Klassen-Grundlagen
------------------------------------------------------------------------

  class ChatNachricht {
      constructor(rolle, text) {
          this.rolle = rolle;       // "user" oder "assistant"
          this.text = text;
          this.zeitstempel = Date.now();
      }

      istVomNutzer() {
          return this.rolle === 'user';
      }

      zuHTML() {
          const farbe = this.istVomNutzer() ? '#FFD700' : '#7FFFD4';
          const label = this.istVomNutzer() ? 'Du' : 'Alfonz';
          return `<b style="color:${farbe};">${label}:</b> <p>${this.text}</p>`;
      }
  }

  const m1 = new ChatNachricht('user', 'Hallo Alfonz!');
  m1.istVomNutzer()   → true
  m1.zuHTML()         → '<b style="color:#FFD700;">Du:</b> <p>Hallo Alfonz!</p>'

  // new ChatNachricht(...) erstellt eine NEUE INSTANZ — ein eigenständiges
  // Objekt, das die Struktur der Klasse hat, aber EIGENE Werte für
  // rolle, text, zeitstempel.

------------------------------------------------------------------------
11.2  Vererbung — extends und super
------------------------------------------------------------------------

  class Komponente {
      constructor(elementId) {
          this.el = document.getElementById(elementId);
          if (!this.el) throw new Error(`Element #${elementId} nicht gefunden`);
      }
      zeigen() { this.el.classList.remove('versteckt'); }
      verstecken() { this.el.classList.add('versteckt'); }
  }

  class Modal extends Komponente {
      constructor(elementId, closeBtnId) {
          super(elementId);   // ruft den constructor von Komponente auf
          this.closeBtn = document.getElementById(closeBtnId);
          this.closeBtn.addEventListener('click', () => this.verstecken());
          // Pfeilfunktion (Modul 4.6!) — "this" bleibt das Modal-Objekt,
          // nicht der Button.
      }
      zeigen() {            // überschreibt die Methode der Elternklasse
          super.zeigen();   // ruft TROTZDEM die Original-Methode auf
          document.body.classList.add('modal-offen');
      }
  }

  const meinModal = new Modal('modal', 'modal-close');
  meinModal.zeigen();   // → ruft Modal.zeigen() → ruft Komponente.zeigen()
                         //   UND fügt zusätzlich die body-Klasse hinzu

------------------------------------------------------------------------
11.3  Getter, Setter, statische Methoden, private Felder
------------------------------------------------------------------------

  class Projekt {
      #intern = 0;   // PRIVATES Feld (# Präfix) — NUR innerhalb der
                      // Klasse zugreifbar, von außen unsichtbar!

      constructor(name, jahr) {
          this.name = name;
          this._jahr = jahr;
      }

      // Getter — wird wie eine PROPERTY benutzt, nicht wie eine Methode:
      get jahr() { return this._jahr; }

      // Setter — mit eigener Logik beim ZUWEISEN:
      set jahr(wert) {
          if (wert < 2020) throw new Error("Jahr unrealistisch");
          this._jahr = wert;
      }

      erhoeheIntern() { this.#intern++; return this.#intern; }

      // STATISCHE Methode — gehört zur KLASSE, nicht zu einer Instanz:
      static ausObjekt(obj) {
          return new Projekt(obj.name, obj.jahr);
      }
  }

  const p = new Projekt("TrafkTux", 2026);
  p.jahr           → 2026     (Getter — KEINE Klammern!)
  p.jahr = 2027;    // Setter
  p.jahr = 2019;    // wirft Error: "Jahr unrealistisch"
  p.erhoeheIntern();         → 1
  p.#intern                   → SyntaxError — von außen unzugreifbar!

  const p2 = Projekt.ausObjekt({ name: "RSA", jahr: 2024 });
  // Projekt.ausObjekt — OHNE "new", direkt auf der KLASSE aufgerufen

------------------------------------------------------------------------
11.4  Wann Klassen, wann "einfache" Objekte/Funktionen?
------------------------------------------------------------------------

Klassen lohnen sich, wenn:
  - du MEHRERE Instanzen mit GLEICHEM Verhalten brauchst (mehrere
    Chat-Nachrichten, mehrere Modals, mehrere "Karten" mit eigenem
    State),
  - Vererbung sinnvoll Code spart (mehrere ÄHNLICHE Komponenten, die
    eine Basis teilen),
  - du PRIVATEN, gekapselten Zustand brauchst (#feld).

Einfache Objekte/Funktionen reichen, wenn:
  - es nur EINE Sache gibt (genau EIN Lautstärkeregler, EIN Header —
    volume.js, nav.js sind dafür Funktionen, KEINE Klassen, und das ist
    völlig richtig so!),
  - "Konfiguration" oder "Daten" beschrieben werden, kein Verhalten.

Faustregel für TrafkSite: Die aktuellen Skripte (nav.js, volume.js,
placeholder.js) sind als Funktionen/IIFE genau richtig — es gibt jeweils
nur EINE Navigation, EINEN Lautstärkeregler. Würdest du aber z.B. MEHRERE
unabhängige "Such-Widgets" auf einer Seite brauchen, die jedes seinen
eigenen Zustand (letzter Suchbegriff, eigene Ergebnis-Liste) haben sollen
— DAS wäre ein guter Fall für eine Klasse SuchWidget, von der du mehrere
new SuchWidget(...)-Instanzen erzeugst.

================================================================================
MODUL 12: Fehlerbehandlung — robuster Code
================================================================================

Code, der NUR im "Idealfall" funktioniert, ist auf einer echten Website
WERTLOS — Nutzer haben langsames Internet, blockieren Requests,
tippen Unsinn in Formulare, oder ein Server ist mal down. Dieses Modul
macht aus "funktioniert bei mir" → "funktioniert auch, wenn was schiefgeht".

------------------------------------------------------------------------
12.1  try/catch/finally
------------------------------------------------------------------------

  try {
      // Code, der FEHLSCHLAGEN könnte
      const daten = JSON.parse(unsicherer_string);
  } catch (error) {
      // läuft NUR, wenn im try-Block ein Fehler AUFTRAT
      console.error("Parse-Fehler:", error.message);
  } finally {
      // läuft IMMER — egal ob Erfolg oder Fehler
      console.log("Fertig.");
  }

  // error-Objekt-Eigenschaften:
  error.message     → Beschreibung des Fehlers
  error.name        → "TypeError", "SyntaxError", "Error", ...
  error.stack        → Stack-Trace (wo genau ist es passiert)

  // Aus deinem AskAlfonz.js — der gesamte fetch-Block läuft in try/catch:
  try {
      const res = await fetch(PROXY_URL, { ... });
      const data = await res.json();
      // ... Erfolg verarbeiten ...
  } catch (err) {
      // ... Fehlermeldung im Chat anzeigen ...
  }

------------------------------------------------------------------------
12.2  Eigene Fehler werfen — throw
------------------------------------------------------------------------

  function teile(a, b) {
      if (b === 0) {
          throw new Error("Division durch Null nicht erlaubt");
      }
      return a / b;
  }

  try {
      teile(10, 0);
  } catch (e) {
      console.error(e.message);   // "Division durch Null nicht erlaubt"
  }

  // Eigene Fehler-Klassen (Vererbung aus Modul 11!) — für
  // unterscheidbare Fehlerarten:
  class ValidierungsFehler extends Error {
      constructor(feld, message) {
          super(message);
          this.name = 'ValidierungsFehler';
          this.feld = feld;
      }
  }

  function pruefeAlter(wert) {
      const n = Number(wert);
      if (Number.isNaN(n)) throw new ValidierungsFehler('alter', 'Keine Zahl');
      if (n < 0 || n > 150) throw new ValidierungsFehler('alter', 'Unrealistisch');
      return n;
  }

  try {
      pruefeAlter("abc");
  } catch (e) {
      if (e instanceof ValidierungsFehler) {
          console.log(`Fehler im Feld "${e.feld}": ${e.message}`);
      } else {
          throw e;   // unbekannten Fehler WEITERWERFEN, nicht verschlucken!
      }
  }

------------------------------------------------------------------------
12.3  Defensive Programmierung — Annahmen prüfen, BEVOR sie crashen
------------------------------------------------------------------------

  // Aus deinem nav.js — der ALLERERSTE Schritt jeder Funktion:
  function initNav() {
      const navContainer = document.getElementById('header-nav');
      if (!navContainer) return;   // ← KEIN Crash, einfach nichts tun,
                                     // falls dieses Element auf der
                                     // aktuellen Seite gar nicht existiert
      // ... weiter
  }

  // Optional Chaining (Modul 2.7) als defensives Werkzeug:
  document.getElementById(loadingId)?.remove();
  // Statt:
  //   const el = document.getElementById(loadingId);
  //   if (el) el.remove();

  // Array.isArray() statt typeof — typeof [] === 'object' ist nicht
  // hilfreich, um Arrays von "normalen" Objekten zu unterscheiden:
  function verarbeite(daten) {
      if (!Array.isArray(daten)) {
          console.warn("Erwartete ein Array, bekam:", typeof daten);
          return [];
      }
      return daten.map(/* ... */);
  }

  // typeof für FUNKTIONEN prüfen — bevor man sie aufruft (genau wie
  // in deinem nav.js!):
  if (typeof makeBubbleVideo === 'function') {
      bubble.appendChild(makeBubbleVideo());
  }
  // → falls placeholder.js NICHT geladen wurde (z.B. weil jemand das
  // <script>-Tag entfernt hat), crasht nav.js NICHT — es lässt die
  // Bubble einfach weg.

------------------------------------------------------------------------
12.4  Fehler global abfangen — window.onerror und unhandledrejection
------------------------------------------------------------------------

Für Fehler, die NICHT in einem try/catch landen (z.B. ein Tippfehler in
einem Event-Handler), gibt es globale "Auffangnetze":

  window.addEventListener('error', e => {
      console.error('Globaler Fehler:', e.message, 'in', e.filename, 'Zeile', e.lineno);
      // z.B. an einen Logging-Dienst senden
  });

  window.addEventListener('unhandledrejection', e => {
      console.error('Unbehandeltes Promise:', e.reason);
      e.preventDefault();   // verhindert die Standard-Konsolenausgabe
  });

  // Praktisch: ein Promise OHNE .catch() / try-catch:
  fetch('/nicht-vorhanden').then(r => r.json());
  // → würde bei Netzwerkfehler hier landen, NICHT in deinem normalen Code

Diese globalen Handler sind kein Ersatz für try/catch an der richtigen
Stelle — aber ein gutes "letztes Sicherheitsnetz", besonders während der
Entwicklung (oder für ein eigenes Mini-Error-Logging).

------------------------------------------------------------------------
12.5  Console-Methoden für besseres Debugging
------------------------------------------------------------------------

  console.log(...)      → normale Ausgabe
  console.warn(...)     → gelb, "Achtung"
  console.error(...)    → rot, "Fehler" (+ Stack-Trace im Browser)
  console.info(...)     → wie log, semantisch "Info"
  console.table(array)   → Array von Objekten als TABELLE darstellen
  console.group("X")     → Gruppe öffnen — folgende logs eingerückt
  console.groupEnd()      → Gruppe schließen
  console.assert(bedingung, "Nachricht")  → loggt NUR, wenn bedingung FALSE ist
  console.time("x"); ... console.timeEnd("x")   → Zeitmessung

  // Aus deinem nav.js:
  console.log("HTML URLs gefunden:", urls.length, urls.slice(0, 5));
  console.log("Tree root keys:", Object.keys(tree));
  console.warn(e);

================================================================================
MODUL 13: Module — Code über mehrere Dateien organisieren
================================================================================

In Modul 4.3 hast du gesehen: ohne type="module" landen ALLE Top-Level-
Deklarationen mehrerer <script>-Dateien im SELBEN globalen Scope. Das
funktioniert für ein kleines Projekt wie TrafkSite gut (solange man
aufpasst), wird aber mit wachsender Codebasis riskant. ES-Module sind die
moderne Lösung.

------------------------------------------------------------------------
13.1  export und import — die Grundlagen
------------------------------------------------------------------------

  // datei: utils.js
  export function formatiereDatum(d) {
      return d.toLocaleDateString('de-DE');
  }

  export const VERSION = "1.0";

  export default function haupt() {
      console.log("Hauptfunktion");
  }

  // datei: main.js
  import haupt, { formatiereDatum, VERSION } from './utils.js';
  //     ^default-Export    ^benannte Exports (geschweifte Klammern!)

  haupt();
  console.log(formatiereDatum(new Date()), VERSION);

  // Im HTML — type="module" ist PFLICHT für import/export:
  <script type="module" src="main.js"></script>

Unterschiede zu "klassischen" Skripten (type="module" bewirkt automatisch):
  - eigener Scope PRO DATEI — keine globalen Variablen mehr, die sich
    überschreiben (löst das Problem aus Modul 4.3!)
  - automatisch defer-Verhalten (lädt parallel, führt nach DOM-Parsing aus)
  - automatisch "strict mode" (Modul 14 erklärt, was das bedeutet)
  - import/export funktioniert NUR zwischen Modulen — ein normales
    <script> ohne type="module" kann NICHT importieren

------------------------------------------------------------------------
13.2  Benannte Exports vs. Default-Export
------------------------------------------------------------------------

  // Benannte Exports — beliebig viele pro Datei, MÜSSEN beim Import
  // mit GLEICHEM Namen (oder per "as" umbenannt) importiert werden:
  export const A = 1;
  export const B = 2;
  import { A, B as Zwei } from './modul.js';

  // Default-Export — HÖCHSTENS EINER pro Datei, beim Import beliebiger
  // Name möglich:
  export default function() { ... }
  import irgendeinName from './modul.js';

  // Alles auf einmal als Namespace importieren:
  import * as Utils from './utils.js';
  Utils.formatiereDatum(...);
  Utils.VERSION

  // Re-Export — ein Modul "bündelt" Exports anderer Module:
  export { formatiereDatum } from './utils.js';
  export * from './weitere-utils.js';

------------------------------------------------------------------------
13.3  TrafkSite mit Modulen — wie könnte das aussehen?
------------------------------------------------------------------------

So, wie TrafkSite AKTUELL strukturiert ist (placeholder.js, nav.js,
volume.js als normale <script>-Tags), funktioniert es — weil die Skripte
SICH GEGENSEITIG KENNEN (nav.js ruft global makeBubbleVideo() auf) und
absichtlich GLOBAL bleiben sollen.

EIN möglicher modularer Umbau (NUR zur Veranschaulichung — keine
Pflichtänderung für dein Projekt!):

  // datei: bubble.js
  export const BUBBLE_VIDEO = "SourceHop-Notes/.../animation60.webm";
  export const BUBBLE_VIDEO_LIGHTER = "SourceHop-Notes/.../Lighter....webm";
  export function makeBubbleVideo(src = BUBBLE_VIDEO, className = 'nav-bubble-video') {
      // ... wie bisher
  }
  export function injectNavVideos(root = document) { /* ... */ }

  // datei: nav.js
  import { makeBubbleVideo } from './bubble.js';
  export function initNav() {
      // ... ruft jetzt das IMPORTIERTE makeBubbleVideo direkt auf,
      // statt typeof-Check (Modul 12.3) — der Import würde bei
      // FEHLENDER Datei selbst einen Fehler werfen, also ist
      // makeBubbleVideo IMMER eine Funktion, wenn nav.js überhaupt lädt.
  }

  // datei: main.js — der "Einstiegspunkt"
  import { initNav } from './nav.js';
  import { injectNavVideos, injectSectionVideos } from './bubble.js';
  // ... Initialisierung

  <script type="module" src="files/main.js"></script>

VORTEILE: explizite Abhängigkeiten (man SIEHT sofort, was nav.js
braucht), kein globaler "Topf" mehr, bessere Testbarkeit (Modul 19),
Tools wie Bundler (Modul 15) können ungenutzten Code automatisch
entfernen ("Tree Shaking").
NACHTEILE: mehr Dateien/Imports zum Pflegen, type="module" hat eigene
CORS-Regeln (ES-Module können bei file://-URLs Probleme machen — man
braucht meist einen lokalen Server zum Testen, z.B. python3 -m http.server).

------------------------------------------------------------------------
13.4  Dynamische Imports — Code "on demand" nachladen
------------------------------------------------------------------------

  button.addEventListener('click', async () => {
      const { schwereFunktion } = await import('./schwer.js');
      schwereFunktion();

================================================================================
MODUL 14: Moderne JavaScript-Features (ES2020+)
================================================================================

JS entwickelt sich ständig weiter. Dieses Modul sammelt nützliche,
modernere Features, die in den vorherigen Modulen schon AUFGETAUCHT sind
oder gut dazu passen — als Referenz UND Vertiefung.

------------------------------------------------------------------------
14.1  Optional Chaining und Nullish Coalescing — Vertiefung
------------------------------------------------------------------------

Bereits in 2.7 eingeführt — hier zwei weitere Varianten:

  // Optional Chaining mit FUNKTIONSAUFRUF:
  obj.methode?.()
  // Ruft methode NUR auf, wenn sie existiert — sonst undefined,
  // KEIN "is not a function"-Fehler.

  // Optional Chaining mit ARRAY-INDEX:
  arr?.[0]
  // Greift auf Index 0 zu, NUR wenn arr nicht null/undefined ist.

  // Logische Zuweisungs-Operatoren — Kombination aus Operator und =:
  let x = null;
  x ??= "Standard";    // x = x ?? "Standard"  → "Standard" (war null)
  x ||= "Anderer";      // x = x || "Anderer"   → "Standard" (truthy, bleibt)
  x &&= "Geändert";     // x = x && "Geändert"  → "Geändert" (truthy → geändert)

------------------------------------------------------------------------
14.2  Array/Object-Neuerungen
------------------------------------------------------------------------

  // Array.flat() — verschachtelte Arrays "flach klopfen":
  [[1,2],[3,[4,5]]].flat()     → [1, 2, 3, [4, 5]]   (1 Ebene tief)
  [[1,2],[3,[4,5]]].flat(2)    → [1, 2, 3, 4, 5]     (2 Ebenen tief)
  [[1,2],[3,[4,5]]].flat(Infinity)  → komplett flach, egal wie tief

  // Array.flatMap() — map + flat(1) in einem:
  [[1,2],[3,4]].flatMap(arr => arr.map(n => n * 2))
  → [2, 4, 6, 8]

  // Array.at() — auch NEGATIVE Indizes (wie Python):
  const arr = [10, 20, 30];
  arr.at(-1)    → 30   (letztes Element — kürzer als arr[arr.length-1]!)
  arr.at(0)     → 10

  // Object.fromEntries() — Umkehrung von Object.entries():
  Object.fromEntries([['a', 1], ['b', 2]])   → { a: 1, b: 2 }
  // (siehst du in 7.4 bei FormData!)

  // Array.from mit MAP-Funktion — Erzeugen UND Transformieren in einem:
  Array.from({ length: 5 }, (_, i) => i * 2)   → [0, 2, 4, 6, 8]
  // { length: 5 } ist ein "array-like" Objekt — Array.from(...) macht
  // ein echtes Array daraus, der zweite Parameter transformiert direkt.

  // structuredClone() — TIEFE Kopie (siehe 6.5!), eingebaut, modern:
  const original = { tags: ["a", "b"], datum: new Date() };
  const kopie = structuredClone(original);
  kopie.tags.push("c");
  original.tags   → ["a", "b"]   (UNVERÄNDERT — echte tiefe Kopie!)

14.3  Tagged Template Literals
------------------------------------------------------------------------

Eine Funktion VOR einem Template Literal (ohne Klammern!) kann den
String VOR der Zusammensetzung verarbeiten — die Grundlage für sicheres
HTML-Escaping (siehe Modul 18!):

  function gross(strings, ...werte) {
      return strings.reduce((acc, str, i) =>
          acc + str + (werte[i] !== undefined ? String(werte[i]).toUpperCase() : ''), '');
  }

  const name = "alfonz";
  gross`Hallo ${name}!`   → "Hallo ALFONZ!"

  // strings = ["Hallo ", "!"]  (die "Text-Teile" um die ${}-Lücken)
  // werte   = ["alfonz"]        (die EINGESETZTEN Werte)

In Modul 18 nutzen wir genau dieses Prinzip für eine html`...`-Funktion,
die eingesetzte Werte AUTOMATISCH escaped — ein zentraler Baustein gegen
XSS bei dynamisch erzeugtem innerHTML.

14.4  Strict Mode
------------------------------------------------------------------------

  'use strict';
  // ODER automatisch aktiv bei: Klassen, ES-Module (type="module"),
  // Arrow Functions im Modul-Kontext

Was ändert sich? (Auswahl)
  - Variable ohne let/const/var → FEHLER statt stille globale Variable:
      'use strict';
      x = 5;   // ReferenceError: x is not defined
      // OHNE strict mode würde dies STILLSCHWEIGEND window.x = 5 erzeugen!

  - this in "normalen" Funktionen ohne Kontext ist undefined statt window
    (verhindert versehentliches Schreiben auf globale Objekte)

  - Doppelte Funktionsparameter-Namen sind FEHLER
  - Bestimmte unsichere Konstrukte (with, ...) sind verboten

Strict Mode deckt VIELE typische Tippfehler SOFORT auf, statt sie
"durchschlüpfen" zu lassen und erst viel später zu seltsamen Bugs zu

MODUL 15: Performance und Best Practices
================================================================================

Code, der FUNKTIONIERT, ist Schritt 1. Code, der auch bei VIELEN
Elementen, SCHWACHEN Geräten oder HÄUFIGEN Events flüssig bleibt, ist
Schritt 2 — und genau das hat dein Team in placeholder.js bereits aktiv
betrieben (Kommentare wie "spart ~50% GPU auf Firefox" sind kein Zufall!).

------------------------------------------------------------------------
15.1  Debounce und Throttle — Events "bremsen"
------------------------------------------------------------------------

Manche Events (scroll, resize, input, mousemove) feuern SEHR HÄUFIG —
teils 60+ mal pro Sekunde. Führt jeder einzelne Event eine teure
Operation aus (DOM-Updates, fetch, Berechnungen), wird die Seite
RUCKELIG.

  DEBOUNCE — "warte, bis RUHE ist":
    Führt die Funktion erst aus, wenn seit dem LETZTEN Aufruf eine
    bestimmte Zeit VERGANGEN ist. Perfekt für Live-Suche: erst suchen,
    wenn der Nutzer 300ms NICHT mehr getippt hat.

      function debounce(fn, ms) {
          let timeoutId;
          return function(...args) {
              clearTimeout(timeoutId);
              timeoutId = setTimeout(() => fn.apply(this, args), ms);
          };
      }

      const sucheLive = debounce(text => {
          console.log("Suche nach:", text);
          // ... teure Operation, z.B. fetch
      }, 300);

      sucheInput.addEventListener('input', e => sucheLive(e.target.value));
      // Tippt der Nutzer "TrafkTux" schnell, läuft sucheLive NUR EINMAL,
      // 300ms NACH dem letzten Tastendruck — nicht 8x für jeden Buchstaben.

  THROTTLE — "höchstens alle X ms":
    Führt die Funktion HÖCHSTENS einmal pro Zeitintervall aus, auch wenn
    der Event häufiger feuert. Perfekt für scroll/resize-Handler.

      function throttle(fn, ms) {
          let letzterLauf = 0;
          return function(...args) {
              const jetzt = Date.now();
              if (jetzt - letzterLauf >= ms) {
                  letzterLauf = jetzt;
                  fn.apply(this, args);
              }
          };
      }

      window.addEventListener('scroll', throttle(() => {
          console.log("Scroll-Position:", window.scrollY);
      }, 200));

15.2  requestAnimationFrame — flüssige Animationen
------------------------------------------------------------------------

Bereits in 8.2 erwähnt — hier vertieft. requestAnimationFrame(callback)
führt callback VOR dem NÄCHSTEN Bildschirm-Repaint aus — synchronisiert
mit der Bildwiederholrate (meist 60Hz = ~16.7ms pro Frame).

  // Aus deinem placeholder.js — die komplette Render-Loop:
  let _frameCount = 0;

  function _drawLoop() {
      requestAnimationFrame(_drawLoop);   // ruft sich selbst für den
                                            // NÄCHSTEN Frame wieder auf

      if (document.hidden) return;        // Tab unsichtbar? → Frame
                                            // überspringen, GPU sparen
                                            // (document.hidden, Modul 5.3)

      _frameCount++;
      if (_frameCount % 2 !== 0) return;   // jeden 2. Frame überspringen
                                             // (% Modulo, Modul 2.4!)
                                             // → effektiv 30fps statt 60fps

      // ... für jeden registrierten Canvas: drawImage aus dem
      //     passenden "Master-Video"
  }
  _drawLoop();   // Erststart

WARUM NICHT setInterval(fn, 16)? setInterval läuft UNABHÄNGIG vom
Bildschirm-Repaint — es könnte Frames "verschwenden" (mehrfach zeichnen,
bevor der Bildschirm überhaupt aktualisiert wird) oder hinter dem
Repaint zurückbleiben. requestAnimationFrame ist IMMER perfekt
synchronisiert UND wird vom Browser AUTOMATISCH pausiert, wenn der Tab
im Hintergrund ist (zusätzlich zur expliziten document.hidden-Prüfung).

------------------------------------------------------------------------
15.3  DOM-Operationen bündeln — Reflow/Repaint vermeiden
------------------------------------------------------------------------

Jede Änderung am DOM (Element hinzufügen, Klasse togglen, .style ändern)
kann den Browser zwingen, das LAYOUT neu zu berechnen ("Reflow") und/oder
neu zu ZEICHNEN ("Repaint") — beides kostet Zeit.

  // SCHLECHT — 100 einzelne appendChild-Aufrufe, potenziell 100 Reflows:
  for (let i = 0; i < 100; i++) {
      const li = document.createElement('li');
      li.textContent = `Item ${i}`;
      liste.appendChild(li);   // jedes Mal potenziell ein Reflow
  }

  // BESSER — DocumentFragment (schon in 3.6 gesehen), EIN Reflow:
  const fragment = document.createDocumentFragment();
  for (let i = 0; i < 100; i++) {
      const li = document.createElement('li');
      li.textContent = `Item ${i}`;
      fragment.appendChild(li);
  }
  liste.appendChild(fragment);   // EIN appendChild, EIN Reflow

  // SCHLECHT — Lesen UND Schreiben abwechselnd ("Layout Thrashing"):
  elements.forEach(el => {
      const breite = el.offsetWidth;   // LESEN → erzwingt Reflow
      el.style.width = breite * 2 + 'px';  // SCHREIBEN → Layout "dirty"
  });
  // Beim NÄCHSTEN .offsetWidth im nächsten Durchlauf MUSS der Browser
  // das Layout NEU berechnen, weil es durch die vorherige Schreib-
  // Operation "schmutzig" wurde — DAS wiederholt sich für JEDES Element!

  // BESSER — ERST alle LESEN, DANN alle SCHREIBEN:
  const breiten = elements.map(el => el.offsetWidth);   // alle LESEN
  elements.forEach((el, i) => {
      el.style.width = breiten[i] * 2 + 'px';            // alle SCHREIBEN
  });

      entries.forEach(entry => {
          if (entry.isIntersecting) {

              observer.unobserve(entry.target);   // einmal reicht
          }
      });
  }, { threshold: 0.1 });   // ab 10% sichtbar

  document.querySelectorAll('section').forEach(sec => observer.observe(sec));

  // Idee für placeholder.js: injectSectionVideos() läuft aktuell für

MODUL 16: Web Components — eigene HTML-Elemente
================================================================================

Web Components erlauben dir, EIGENE, WIEDERVERWENDBARE HTML-Tags zu
erstellen — mit eigenem Verhalten UND (optional) eigenem, GEKAPSELTEM
Styling. Sie bauen direkt auf allem aus den Modulen 3, 5 und 11 auf.

------------------------------------------------------------------------
16.1  Custom Elements — Grundgerüst
------------------------------------------------------------------------

  class TrafkBubble extends HTMLElement {
      constructor() {
          super();   // PFLICHT — wie bei jeder extends-Klasse (Modul 11.2)
      }

      // Lifecycle-Methode: läuft, wenn das Element ins DOM eingefügt wird
      connectedCallback() {
          this.innerHTML = `<span class="bubble-text">${this.getAttribute('text') ?? ''}</span>`;
          this.addEventListener('click', () => {
              this.classList.toggle('aktiv');
          });
      }

      // läuft, wenn das Element aus dem DOM entfernt wird — Aufräumen!
      disconnectedCallback() {
          // z.B. Timer/Listener entfernen, die NICHT automatisch verschwinden
      }
  }

  // Registrieren — der Name MUSS einen Bindestrich enthalten!
  customElements.define('trafk-bubble', TrafkBubble);

  // Jetzt im HTML benutzbar — wie JEDES andere Tag:
  <trafk-bubble text="Hallo"></trafk-bubble>

------------------------------------------------------------------------
16.2  Attribute beobachten — observedAttributes
------------------------------------------------------------------------

  class TrafkBubble extends HTMLElement {
      static get observedAttributes() {
          return ['text', 'aktiv'];   // NUR diese Attribute werden beobachtet
      }

      connectedCallback() {
          this.render();
      }

      // läuft AUTOMATISCH, wenn sich eines der observedAttributes ändert:
      attributeChangedCallback(name, alterWert, neuerWert) {
          this.render();
      }

      render() {
          const text = this.getAttribute('text') ?? '';
          const aktiv = this.hasAttribute('aktiv');
          this.innerHTML = `<span class="${aktiv ? 'aktiv' : ''}">${text}</span>`;
      }
  }
  customElements.define('trafk-bubble', TrafkBubble);

  // Von außen — z.B. per JS:
  document.querySelector('trafk-bubble').setAttribute('text', 'Neu!');
  // → attributeChangedCallback läuft AUTOMATISCH, render() aktualisiert
  //   den Inhalt — OHNE dass der aufrufende Code irgendwas vom internen
  //   Aufbau wissen muss!

------------------------------------------------------------------------
16.3  Shadow DOM — gekapseltes Styling
------------------------------------------------------------------------

Normales innerHTML (wie oben) "vermischt" sich mit dem Rest der Seite —
CSS-Regeln aus standard.css könnten dein Custom Element beeinflussen
(und umgekehrt). Shadow DOM erzeugt einen ISOLIERTEN Teilbaum:

  class TrafkBubble extends HTMLElement {
      constructor() {
          super();
          const shadow = this.attachShadow({ mode: 'open' });
          shadow.innerHTML = `
              <style>
                  /* Diese Styles wirken NUR innerhalb dieses Elements! */
                  span { color: var(--Kraft, gold); font-weight: bold; }
              </style>
              <span><slot></slot></span>
          `;
      }
  }
  customElements.define('trafk-bubble', TrafkBubble);

  // <slot> ist ein Platzhalter für den INHALT, den der NUTZER des Tags
  // hineinschreibt:
  <trafk-bubble>Ask Alfonz</trafk-bubble>
  // "Ask Alfonz" landet im <slot> — span umrahmt es farblich.

  // var(--Kraft, gold) — CSS Custom Properties (aus standard.css!)
  // WERDEN durch Shadow DOM hindurch VERERBT (im Gegensatz zu normalen
  // CSS-Regeln) — :root { --Kraft: #fff495; } aus deinem standard.css
  // würde HIER also automatisch greifen, der Fallback "gold" nur, wenn
  // --Kraft GAR NICHT definiert wäre.

------------------------------------------------------------------------
16.4  Praxisbeispiel: <trafk-nav-item> für nav.js
------------------------------------------------------------------------

Eine denkbare (nicht zwingende!) Umsetzung eines Nav-Eintrags als Web
Component — fasst HTML-Struktur + Klick-Logik + Styling an EINEM Ort:

  class TrafkNavItem extends HTMLElement {
      connectedCallback() {
          const name = this.getAttribute('name') ?? '';
          const href = this.getAttribute('href') ?? '#';

          this.innerHTML = `
              <div class="nav">
                  <a class="do" href="${href}">${name}</a>
              </div>
          `;
          // injectNavVideos (aus placeholder.js) könnte HIER für
          // this.querySelector('.nav') aufgerufen werden — eine
          // Web-Component "weiß", wann sie fertig zusammengebaut ist.
      }
  }
  customElements.define('trafk-nav-item', TrafkNavItem);

  // renderTree() (aus nav.js) könnte STATT createElement('li') + viele
  // einzelne appendChild-Aufrufe NUR EIN <trafk-nav-item> erzeugen:
  const item = document.createElement('trafk-nav-item');
  item.setAttribute('name', n.name);
  item.setAttribute('href', n.path || '#');
  li.appendChild(item);
  // Die GESAMTE innere Struktur (.nav, <a class="do">, ...) ist in der
  // Web Component DEFINIERT — nav.js muss sie nicht mehr "von Hand" bauen.

------------------------------------------------------------------------
16.5  Wann Web Components, wann "normale" Funktionen?
------------------------------------------------------------------------

Web Components lohnen sich, wenn:
  - eine UI-Einheit MEHRFACH auf der Seite (oder über mehrere Seiten)
    vorkommt UND immer GLEICH aussieht/funktioniert,
  - du sie wie EINEN "Baustein" im HTML selbst sehen willst
    (<trafk-bubble>Text</trafk-bubble> statt fünf Zeilen JS),
  - GEKAPSELTES Styling (Shadow DOM) Konflikte mit globalem CSS vermeiden
    soll.

"Normale" Funktionen (wie aktuell in nav.js, placeholder.js) reichen,
wenn:
  - es um EINMALIGE Initialisierung/Manipulation der bestehenden Seite
    geht (Header/Footer laden, Lautstärke anwenden),
  - das Element schon im HTML EXISTIERT und nur "aufgewertet" wird.

WARUM BRAUCHT AskAlfonz.js DIESEN PROXY ÜBERHAUPT?

  Der "GH_TOKEN" ist ein GEHEIMER Schlüssel für die KI-API. Würde
  AskAlfonz.js DIREKT aus dem Browser mit diesem Token an
  models.inference.ai.azure.com senden, könnte JEDER Besucher der Seite
  diesen Token im Network-Tab (Modul 1.5!) auslesen und SELBST missbrauchen
  — auf KOSTEN von TrafkHop Entertainment.

  Der Proxy (index.js) läuft auf einem SERVER, hält den Token GEHEIM
  (process.env), und AskAlfonz.js (im Browser) sendet seine Anfragen NUR
  an PROXY_URL — ohne Token. Der Server fügt den Token serverseitig
  hinzu, bevor er an die KI-API weiterleitet. Aus Sicht des Browsers ist
  index.js einfach "irgendeine API", deren interne Implementierung
  (inkl. Geheimnisse) unsichtbar bleibt.

17.4  Express-Grundlagen — Routen, Middleware
------------------------------------------------------------------------

  app.get('/', (req, res) => res.send('Hallo!'));
  app.get('/api/projekte/:id', (req, res) => {
      const id = req.params.id;     // aus der URL, z.B. /api/projekte/42 → "42"
      res.json({ id, name: "TrafkTux" });
  });
  app.post('/api/kontakt', (req, res) => {
      const { name, email } = req.body;   // dank express.json()

      res.status(201).json({ status: 'ok' });

  // Middleware — Funktionen, die VOR der eigentlichen Route laufen:
  app.use((req, res, next) => {
      console.log(`${req.method} ${req.path}`);
      next();   // WICHTIG — ohne next() "hängt" der Request!
  });

  // Statische Dateien ausliefern (z.B. eine eigene index.html):
  app.use(express.static('public'));

================================================================================
MODUL 18: Sicherheit in JavaScript
================================================================================

Jede Webseite, die Nutzereingaben verarbeitet (Formulare, Chat,
URL-Parameter) oder mit APIs/Tokens arbeitet, hat eine ANGRIFFSFLÄCHE.
Dieses Modul war im alten Kurs nur am Rand vorhanden — für moderne
Websites ist es Pflicht.

------------------------------------------------------------------------
18.1  XSS (Cross-Site Scripting) — die häufigste Web-Schwachstelle
------------------------------------------------------------------------

XSS passiert, wenn NUTZER-KONTROLLIERTER Text als HTML interpretiert
wird und dadurch Skripte ausgeführt werden, die NICHT von dir stammen.

  // GEFÄHRLICH — wenn nutzerName aus einem Formular/URL-Parameter kommt:
  const nutzerName = '<img src=x onerror="alert(\'gehackt!\')">';
  begruessung.innerHTML = `Hallo, ${nutzerName}!`;
  // → Der Browser versucht, das <img> zu laden, scheitert (src=x
  //   existiert nicht), und führt deshalb onerror aus — JEDER Code in
  //   onerror="..." LÄUFT. Mit Zugriff auf document.cookie,
  //   localStorage, etc. könnte ein Angreifer hier ECHTEN Schaden anrichten.

  // SICHER — textContent interpretiert NIEMALS HTML (Modul 3.3!):
  begruessung.textContent = `Hallo, ${nutzerName}!`;
  // → zeigt LITERAL den Text "<img src=x onerror=...>" an, führt NICHTS aus.

REGEL: Nutzereingaben (oder Daten UNBEKANNTER Herkunft — auch aus einer
API!) NIEMALS direkt per innerHTML einsetzen. Für reinen Text:
textContent. Brauchst du HTML-STRUKTUR UM einen Nutzer-Wert herum,
ESCAPE den Wert SELBST:

  function escapeHTML(str) {
      const div = document.createElement('div');
      div.textContent = str;
      return div.innerHTML;
      // Trick: textContent setzen, innerHTML LESEN — der Browser
      // wandelt < > & " ' automatisch in &lt; &gt; &amp; &quot; &#39; um!
  }

  escapeHTML('<b>Hi</b>')   → "&lt;b&gt;Hi&lt;/b&gt;"

  msgDiv.innerHTML = `<b>Nutzer:</b> ${escapeHTML(nutzerEingabe)}`;
  // Jetzt ist nutzerEingabe IMMER reiner, escapeter Text — egal was drinsteht.

Tagged Template (aus 14.3) für automatisches Escaping:

  function html(strings, ...werte) {
      return strings.reduce((acc, str, i) =>
          acc + str + (werte[i] !== undefined ? escapeHTML(String(werte[i])) : ''), '');
  }

  msgDiv.innerHTML = html`<b>Nutzer:</b> ${nutzerEingabe}`;
  // ALLE eingesetzten ${...}-Werte werden automatisch escaped — der
  // "Text um sie herum" (das <b>...</b>) bleibt echtes HTML.

EINSCHÄTZUNG für AskAlfonz.js: msgDiv.innerHTML = `<b ...>Traveler:</b>
<p>${text}</p>` setzt text (die NUTZEREINGABE aus dem Chat-Feld!) DIREKT
per innerHTML. Für ein internes/Demo-Projekt mit überschaubarem Risiko
mag das tolerierbar sein — bei einer Seite mit MEHREREN Nutzern oder
sensiblen Daten wäre escapeHTML(text) bzw. die html-Tagged-Template-
Funktion HIER die korrekte, robuste Lösung.

------------------------------------------------------------------------
18.2  Niemals Secrets im Browser-Code
------------------------------------------------------------------------

ALLES, was im Browser läuft — JEDE Zeile JS, JEDE Variable — ist für
JEDEN Besucher LESBAR (Rechtsklick → "Seitenquelltext" / DevTools
Sources-Tab, Modul 1.5). Es gibt KEINE Möglichkeit, etwas im Frontend-
Code "geheim" zu halten.

  // NIEMALS in Browser-Code:
  const API_KEY = "sk-abc123xyz...";   // JEDER kann das lesen und benutzen!

  // RICHTIG — wie in deinem Projekt:
  //   Browser (AskAlfonz.js) → fetch(PROXY_URL) → Server (index.js) →
  //   nutzt process.env.GH_TOKEN (NUR auf dem Server lesbar) →
  //   echte KI-API

Das ist GENAU der Grund für index.js als Proxy (siehe 17.3) — und der
Grund, warum process.env (Server) und localStorage (Browser, IMMER
einsehbar) fundamental unterschiedliche "Geheimhaltungs-Level" haben.

------------------------------------------------------------------------
18.3  CORS — warum manche fetch()-Aufrufe blockiert werden
------------------------------------------------------------------------

CORS (Cross-Origin Resource Sharing) ist eine BROWSER-Sicherheitsregel:
Eine Seite auf origin-a.com darf standardmäßig KEINE Daten von
origin-b.com per fetch lesen — außer origin-b.com erlaubt es EXPLIZIT.

  // trafkhop-entertainment.github.io versucht, von einem ANDEREN Server
  // Daten zu holen:
  fetch('https://anderer-server.com/api/daten')
  // → Falls anderer-server.com KEINEN passenden
  //   "Access-Control-Allow-Origin"-Header sendet:
  //   Browser BLOCKIERT die Antwort (Network-Tab zeigt oft trotzdem
  //   Status 200 — aber JS bekommt die Daten NICHT, "CORS error" in
  //   der Konsole)

  // Aus deinem index.js — GENAU DESHALB:
  app.use(cors());
  // → Express fügt automatisch "Access-Control-Allow-Origin: *" (oder
  //   spezifischer) zu JEDER Antwort hinzu — erlaubt also AUSDRÜCKLICH,
  //   dass z.B. trafkhop-entertainment.github.io diesen Proxy per fetch
  //   aufrufen darf.

CORS schützt NICHT vor Angriffen wie XSS oder gestohlenen Tokens — es
verhindert nur, dass BELIEBIGE fremde Webseiten STILLSCHWEIGEND IM NAMEN
EINES EINGELOGGTEN NUTZERS Daten von einer API lesen, ohne dass die API
es erlaubt hat.

------------------------------------------------------------------------
18.4  Sichere Verwendung von eval, with und Function()
------------------------------------------------------------------------

  eval("alert('Hallo')")    // führt einen STRING als CODE aus
  new Function('a', 'b', 'return a+b')(1,2)   → 3   // ähnlich wie eval

VERMEIDE eval/Function() KOMPLETT, wenn der String IRGENDWOHER kommt, wo
ein Nutzer Einfluss hat — das ist die DIREKTESTE Form von "beliebigen
Code ausführen". Fast IMMER gibt es eine sicherere Alternative
(JSON.parse statt eval für Daten, switch/Funktionen statt dynamischem
Code).

------------------------------------------------------------------------
18.5  Content Security Policy (CSP) — ein Blick über den Tellerrand
------------------------------------------------------------------------

  <meta http-equiv="Content-Security-Policy"
        content="default-src 'self'; script-src 'self'">

CSP ist ein HTTP-Header (oder <meta>-Tag), der dem Browser sagt, WOHER
Skripte/Styles/Bilder geladen werden dürfen. Eine strikte CSP kann viele
XSS-Angriffe BLOCKIEREN, selbst wenn Schadcode "irgendwie" ins HTML
gelangt — der Browser würde inline-Skripte oder Skripte von fremden
Domains schlicht NICHT ausführen. Für GitHub Pages (wie TrafkSite) lässt
sich CSP per <meta>-Tag setzen — ein guter nächster Schritt, sobald
Formulare mit echten Nutzerdaten hinzukommen.

================================================================================
MODUL 19: Testing und Debugging
================================================================================

------------------------------------------------------------------------
19.1  Die DevTools als Debugging-Werkzeug — Vertiefung
------------------------------------------------------------------------

Aufbauend auf 1.5:

  Breakpoints (Sources-Tab)
    Klick auf eine Zeilennummer → Code STOPPT dort, sobald er erreicht
    wird. Du siehst:
      - Scope-Bereich: ALLE Variablen, die an dieser Stelle sichtbar
        sind (Local, Closure, Global — Modul 4.3!)
      - Call Stack: WER hat diese Funktion aufgerufen, und WER hat
        DIESE Funktion aufgerufen, usw.
      - Watch: eigene Ausdrücke, die du beim Pausieren beobachten willst

    Steuerung: "Step over" (nächste Zeile, Funktionsaufrufe NICHT
    betreten), "Step into" (in aufgerufene Funktion HINEIN springen),
    "Step out" (aktuelle Funktion verlassen), "Resume" (weiterlaufen bis
    zum nächsten Breakpoint).

  Conditional Breakpoints
    Rechtsklick auf Zeilennummer → "Add conditional breakpoint" — hält
    NUR an, wenn eine Bedingung true ist, z.B. i === 50 in einer
    Schleife mit 1000 Durchläufen.

  debugger;-Statement
    Direkt im Code platziert — wirkt wie ein Breakpoint, OHNE dass du
    in den DevTools danach suchen musst:

      function buildTree(urls) {
          debugger;   // Browser pausiert HIER, FALLS DevTools offen sind
          // ...
      }

  Live-Bearbeitung
    Im Sources-Tab kannst du Code-Zeilen DIREKT ändern und sofort
    testen — perfekt zum schnellen Ausprobieren von Fixes, OHNE die
    Datei zu speichern (Änderungen gehen beim Neuladen verloren!).

------------------------------------------------------------------------
19.2  Unit Tests — Grundidee (ohne Framework)
------------------------------------------------------------------------

Ein "Test" ist Code, der ANDEREN Code aufruft und PRÜFT, ob das Ergebnis
stimmt — automatisiert, statt jedes Mal manuell nachzuschauen.

  // Eine MINIMALE eigene "Test-Funktion" — das PRINZIP hinter jedem
  // Test-Framework (Jest, Vitest, ...):
  function assertEqual(tatsaechlich, erwartet, beschreibung) {
      if (tatsaechlich === erwartet) {
          console.log(`✓ ${beschreibung}`);
      } else {
          console.error(`✗ ${beschreibung}: erwartet ${erwartet}, bekam ${tatsaechlich}`);
      }
  }

  // Tests für eine Funktion aus deinem Code:
  function shouldIndexUrl(url) {
      if (!url) return false;
      const lower = url.toLowerCase();
      if (lower.includes('/.git/')) return false;
      return true;
  }

  assertEqual(shouldIndexUrl(''), false, 'Leerer String → false');
  assertEqual(shouldIndexUrl('https://x.com/.git/config'), false, '.git wird ausgeschlossen');
  assertEqual(shouldIndexUrl('https://x.com/index.html'), true, 'Normale URL → true');

WARUM TESTEN — gerade bei FUNKTIONEN OHNE DOM (also "reine" Logik wie
shouldIndexUrl, buildTree, sicherDurch aus Modul 12): solche Funktionen
lassen sich OHNE Browser, OHNE Klicken testen — du tippst EINMAL den
Test, und kannst ihn danach JEDERZEIT erneut laufen lassen (z.B. nachdem
du etwas geändert hast — "hab ich was kaputt gemacht?").

Echte Frameworks (Jest, Vitest, Mocha) bieten describe/it/expect-Syntax,
automatisches Auffinden von Testdateien, Mocking (fetch/DOM "fake"
nachbilden) — das Prinzip "Eingabe → erwartetes Ergebnis → Vergleich"
bleibt aber IDENTISCH zu assertEqual oben.

------------------------------------------------------------------------
19.3  Mocking — DOM und fetch in Tests "vorgaukeln"
------------------------------------------------------------------------

Funktionen wie initNav() oder sendMessage() HÄNGEN vom DOM und von fetch
ab — in einer "Test-Umgebung" (kein echter Browser) existieren beide
nicht automatisch. Mocking ersetzt sie durch einfache "Attrappen":

  // "Fake" fetch — gibt IMMER feste Test-Daten zurück, statt echte
  // Netzwerk-Requests zu machen:
  global.fetch = async (url) => ({
      ok: true,
      json: async () => ({ messages: [{ role: 'assistant', content: 'Test-Antwort' }] }),
      text: async () => '<urlset><url><loc>https://x.com/a.html</loc></url></urlset>'
  });

  // Jetzt kann z.B. initNav() (das fetch('sitemap.xml') aufruft) GETESTET
  // werden, OHNE dass eine echte sitemap.xml existieren muss — und das
  // Ergebnis ist IMMER GLEICH (deterministisch), egal wie das Netzwerk
  // gerade läuft.

------------------------------------------------------------------------
19.4  Häufige Fehlermeldungen — was sie bedeuten
------------------------------------------------------------------------

  "Cannot read properties of null (reading 'xyz')"
    → getElementById/querySelector gab null zurück, du hast direkt
      .xyz darauf aufgerufen. Lösung: Element existiert (noch) nicht —
      Skript-Reihenfolge prüfen (Modul 1.2), oder Optional Chaining
      (Modul 2.7), oder defensive if-Prüfung (Modul 12.3).

  "Cannot read properties of undefined (reading 'xyz')"
    → Ähnlich, aber der Wert war undefined — oft, weil eine Funktion
      "vergessen" hat, etwas zurückzugeben, oder ein Objekt-Key falsch
      geschrieben/nicht vorhanden ist.

  "x is not a function"
    → Du hast versucht, etwas aufzurufen, das KEINE Funktion ist — oft
      Tippfehler (.foreach statt .forEach!) oder das Skript, das die
      Funktion definiert, wurde NICHT (oder zu spät) geladen (Modul 1.3,
      typeof-Check aus 12.3).

  "Unexpected token"
    → Syntaxfehler — fehlende Klammer/Komma/Anführungszeichen. Meist in
      der GENANNTEN Zeile ODER der Zeile DAVOR.

  "Maximum call stack size exceeded"
    → Endlose Rekursion — eine Funktion ruft (direkt oder über Umwege)
      sich selbst auf, OHNE eine Abbruchbedingung zu erreichen (siehe
      collapseSelf/renderTree in nav.js — REKURSIVE Funktionen MÜSSEN
      IMMER einen Fall haben, der NICHT weiter rekursiert!).

  "CORS error" / "blocked by CORS policy"
    → siehe Modul 18.3.

  "Uncaught (in promise) ..."
    → ein Promise wurde REJECTED, aber NIEMAND hat .catch() oder
      try/catch darum gelegt (siehe Modul 12.4, unhandledrejection).

================================================================================
MODUL 20: Fortgeschrittene Patterns
================================================================================

Dieses Modul sammelt wiederkehrende STRUKTUR-Muster, die in größeren
Web-Projekten ständig auftauchen — als Vorbereitung auf das
Abschlussprojekt (Modul 21).

------------------------------------------------------------------------
20.1  Module Pattern / Namespace — strukturierter als IIFE
------------------------------------------------------------------------

Erinnerung an das IIFE aus volume.js (4.5) — das "Module Pattern" ist
dieselbe Idee, aber GIBT EXPLIZIT EIN OBJEKT MIT DER ÖFFENTLICHEN API
zurück, statt alles komplett zu verstecken:

  const VolumeManager = (function() {
      // PRIVAT — von außen unsichtbar
      const STORAGE_KEY = 'site-volume';
      const getStoredVol = () => parseFloat(localStorage.getItem(STORAGE_KEY) ?? '0.5');

      function applyVolume(vol) {
          document.querySelectorAll('audio, video').forEach(m => m.volume = vol);
      }

      // ÖFFENTLICH — wird zurückgegeben:
      return {
          init() {
              applyVolume(getStoredVol());
              const slider = document.getElementById('volume');
              if (slider) {
                  slider.value = getStoredVol();
                  slider.addEventListener('input', e => {
                      const vol = parseFloat(e.target.value);
                      localStorage.setItem(STORAGE_KEY, vol);
                      applyVolume(vol);
                  });
              }
          },
          getVolume: getStoredVol   // gezielt EINE Funktion "exportieren"
      };
  })();

  VolumeManager.init();
  console.log(VolumeManager.getVolume());   // 0.5
  console.log(VolumeManager.STORAGE_KEY);    // undefined — privat!

In modernen Projekten ersetzen ES-Module (Modul 13) dieses Pattern meist
— das GRUNDPRINZIP (klare Trennung privat/öffentlich) bleibt aber
identisch.

------------------------------------------------------------------------
20.2  Observer Pattern — lose gekoppelte Komponenten
------------------------------------------------------------------------

Baut direkt auf CustomEvent (Modul 5.7) auf — Komponenten "lauschen" auf
Änderungen, OHNE die Quelle der Änderung zu kennen:

  // Ein zentraler "EventBus":
  const EventBus = new EventTarget();

  // Komponente A meldet eine Änderung:
  function setLautstaerke(vol) {
      localStorage.setItem('site-volume', vol);
      EventBus.dispatchEvent(new CustomEvent('lautstaerke', { detail: { vol } }));
  }

  // Komponente B reagiert — MUSS NICHTS über setLautstaerke wissen:
  EventBus.addEventListener('lautstaerke', e => {
      console.log('Neue Lautstärke:', e.detail.vol);
      // z.B. ein Icon im Header aktualisieren
  });

  // Komponente C reagiert GENAUSO — beliebig viele "Abonnenten":
  EventBus.addEventListener('lautstaerke', e => {
      document.title = e.detail.vol === 0 ? '🔇 TrafkSite' : 'TrafkSite';
  });

VORTEIL: setLautstaerke MUSS NICHT wissen, WIE VIELE oder WELCHE anderen
Teile der Seite sich für Lautstärke-Änderungen interessieren — neue
"Abonnenten" können JEDERZEIT hinzugefügt werden, OHNE setLautstaerke
selbst anzufassen.

------------------------------------------------------------------------
20.3  State Management — ein zentraler "Wahrheits-Speicher"
------------------------------------------------------------------------

Bei mehreren zusammenhängenden Werten (z.B. "Chat-Verlauf", "Lädt
gerade?", "Index bereit?") hilft EIN zentrales State-Objekt, statt vielen
einzelnen globalen Variablen:

  // STATT vieler einzelner globaler Variablen wie in AskAlfonz.js:

20.4  Render-Funktion — UI aus Zustand "neu zeichnen"
------------------------------------------------------------------------

Statt bei JEDER Zustandsänderung manuell EINZELNE DOM-Stellen zu
aktualisieren (fehleranfällig — man "vergisst" leicht eine Stelle), eine
zentrale render()-Funktion, die den GESAMTEN relevanten Bereich aus dem
aktuellen Zustand neu aufbaut:

  function renderChat(state) {
      const container = document.getElementById('chat-window');
      container.innerHTML = '';   // alles entfernen — Modul 3.6

      state.chatHistory.forEach(msg => {
          const div = document.createElement('div');
          div.className = msg.role === 'user' ? 'msg-user' : 'msg-bot';
          div.textContent = msg.content;   // textContent — Modul 18.1!
          container.appendChild(div);
      });
  }

  AppState.subscribe(renderChat);
  AppState.addMessage('user', 'Hallo!');
  // → renderChat läuft automatisch, BAUT DIE GESAMTE Chat-Liste neu auf

Bei WENIGEN Elementen (wie einem Chat-Verlauf) ist "alles neu bauen"
einfach und ausreichend performant. Bei VIELEN/komplexen Elementen
würden Frameworks (React, Vue, ...) hier gezielter NUR die geänderten
Teile aktualisieren ("Virtual DOM" / "Reactivity") — das Grundprinzip
"Zustand → Render-Funktion → DOM" bleibt aber gleich.

21.3  suche.js — der Suchindex (Arrays, Regex, Module 6+10)

Eine vereinfachte Version von fallbackSearch aus AskAlfonz.js — eine

  // suche.js

  // "Index aufbauen" — bei AskAlfonz.js wäre das buildSearchIndex(),
  // hier reicht ein einfaches .filter + .map, weil unsere Daten schon
  // vollständig in daten.js stehen (kein fetch nötig):
  let suchIndex = [];

  function baueIndex() {
      suchIndex = wissensDatenbank
          .filter(eintrag => shouldIndexUrl(eintrag.url))
          .map(eintrag => ({
              ...eintrag,                          // Spread, Modul 6.5!
              textLower: eintrag.text.toLowerCase(),
              isBackup: /\/backups?\//i.test(eintrag.url)   // Regex, Modul 10!
          }));
      return suchIndex;
  }

  // fallbackSearch — Wort-für-Wort-Bewertung, genau wie in AskAlfonz.js:
  function sucheKontext(frage) {
      const woerter = frage.toLowerCase().split(/\W+/).filter(w => w.length > 3);
      //                                              ^Regex: alles, was
      //                                               KEIN Wortzeichen ist,
      //                                               als Trenner (Modul 10.2)

      const bewertet = suchIndex.map(eintrag => {
          let score = 0;
          woerter.forEach(wort => {
              if (eintrag.titel.toLowerCase().includes(wort)) score += 50;
              if (eintrag.textLower.includes(wort)) score += 10;
          });
          // Backup-Einträge abwerten — ein Bonus-Schritt:
          if (eintrag.isBackup) score = score / 2;
          return { eintrag, score };
      });

      // map+filter+sort+slice+map — die Daten-Pipeline aus Modul 6.2!
      return bewertet
          .filter(x => x.score > 0)
          .sort((a, b) => b.score - a.score)
          .slice(0, 2)
          .map(x => x.eintrag);
  }

  function simuliereKIAntwort(kontext, frage) {
      return new Promise((resolve, reject) => {

              // Künstlicher Fehler in 10% der Fälle — zum Testen von
              // Fehlerbehandlung (Modul 12)!

  async function loadSitemap() {
      const sources = ['sitemap.xml'];
      const seenUrls = new Set();   // Modul 6.6 — Duplikate vermeiden!
      for (const src of sources) {
          try {
              const response = await fetch(src);              // Modul 8.5
              if (!response.ok) { console.warn(...); continue; }
              const xmlText = await response.text();
              const urlRegex = /<loc>(.*?)<\/loc>/gi;          // Modul 10!
              const priorityRegex = /<priority>(.*?)<\/priority>/gi;
              const priorities = [...xmlText.matchAll(priorityRegex)].map(m => m[1]);
              // matchAll (Modul 10.3) + Spread (6.5) + map (6.2) — ALLE
              // <priority>-Werte als Array von Strings extrahieren

              let idx = 0, locMatch;
              while ((locMatch = urlRegex.exec(xmlText)) !== null) {
                  // .exec() in einer while-Schleife — die "klassische"
                  // Art, ALLE Treffer EINES Regex mit /g zu durchlaufen
                  // (matchAll, oben, ist die modernere Alternative dafür)
                  let url = locMatch[1].trim();
                  let priority = (idx < priorities.length) ? priorities[idx] : "0.65";
                  // Ternär (Modul 2.4) mit Index-Fallback
                  idx++;
                  if (!seenUrls.has(url)) {
                      seenUrls.add(url);
                      sitemapUrls.push(url);
                      urlPriorityMap.set(url, parseFloat(priority));
                      // Map.set — Modul 6.6, parseFloat — Modul 2.2
                  }
              }
              console.log(`✅ Sitemap geladen: ${sitemapUrls.length} URLs`);
              break;   // sources hat nur 1 Element — break ist hier
                       // "fertig, nicht weitersuchen"
          } catch (e) {
              console.warn(`Fehler beim Laden von ${src}: ${e.message}`);
              // try/catch — Modul 12.1. EIN fehlgeschlagener Source
              // bricht NICHT das ganze Programm — die for-Schleife geht
              // (theoretisch) zum nächsten Eintrag in sources weiter.
          }

          if (url.endsWith('.html')) {
              const parser = new DOMParser();   // baut ein DOM AUS EINEM
                                                   // STRING — wie nav.js
                                                   // es mit der Sitemap-XML
                                                   // tut (Modul 3 + 5!)
              const doc = parser.parseFromString(text, 'text/html');
              const junk = doc.querySelectorAll('script, style, nav, header, footer, .menu, #sidebar');
              junk.forEach(el => el.remove());   // Modul 3.6 — .remove()
              const contentNode = doc.querySelector('main') || doc.querySelector('.content') || doc.body;
              // ||-Ketten (Modul 2.4) als "versuche A, sonst B, sonst C"
              flatText = contentNode.innerText || contentNode.textContent;
              // innerText vs. textContent — Modul 3.3! innerText würde
              // hier NORMALERWEISE NICHT funktionieren, weil doc NICHT
              // im echten DOM "gerendert" ist — innerText ist auf
              // "Off-Screen"-Dokumenten oft leer ("" → falsy), DESHALB
              // der Fallback || contentNode.textContent.
          }

          const flat = flatText.replace(/\s+/g, ' ').trim();
          // Regex \s+ → EIN Leerzeichen (Modul 10.3) + trim (Modul 2.3)
          return { flat: flat, raw: text };
      } catch (e) {
          return { flat: '', raw: '' };   // defensiv — Modul 12.3: lieber
                                            // LEERE Werte als ein Crash,
                                            // der buildSearchIndex() stoppt
      }
  }

  function extractImagesFromRaw(rawText, docUrl) {
      if (!rawText) return [];   // früh-return, Modul 4.4
      const baseDir = docUrl.substring(0, docUrl.lastIndexOf('/') + 1);
      // String-Methoden, Modul 2.3 — alles VOR dem letzten "/"
      const images = [];
      const seen = new Set();   // Duplikate vermeiden, Modul 6.6
      for (const m of rawText.matchAll(/!\[\[([^\]]+\.(png|jpg|jpeg|gif|webp|bmp|svg))\]\]/gi)) {
          // matchAll mit GRUPPEN (Modul 10.3) — m[1] = kompletter
          // Dateiname, m[2] = nur die Endung
          const filename = m[1].trim();
          if (seen.has(filename)) continue;   // continue = "diesen
                                                 // Durchlauf überspringen,
                                                 // weiter mit dem nächsten"
          seen.add(filename);
          const beforeImg = rawText.substring(0, m.index);
          const labelMatch = beforeImg.match(/####\s*picture description of:\s*(.+)\s*$/im);
          const label = labelMatch ? labelMatch[1].trim() : filename.replace(/\.[^.]+$/, '');
          // Ternär (2.4): WENN eine "Bildbeschreibung" im Markdown
          // davor steht, nimm SIE als Label, SONST den Dateinamen ohne
          // Endung (.replace mit Regex, Modul 10.3)
          images.push({ filename, url: baseDir + encodeURIComponent(filename), label });
          // Object-Shorthand: { filename } statt { filename: filename }
          // — eine kleine, sehr verbreitete Schreibweise
      }
      return images;
  }

ABSCHNITT 6 — buildSearchIndex (Zeilen 136-157, Module 6 + 8 + 10):

  async function buildSearchIndex() {
      const relevantUrls = sitemapUrls.filter(shouldIndexUrl);
      // .filter mit FUNKTIONS-REFERENZ (Modul 4.7!) — shouldIndexUrl
      // wird OHNE () übergeben; .filter ruft sie selbst für JEDES
      // Element auf

      const fetchPromises = relevantUrls.map(async (url) => {
          // .map mit ASYNC Callback — gibt ein Array von PROMISES zurück!
          const { flat, raw } = await fetchFileContent(url);  // Destructuring 6.4
          if (!flat) return null;   // wird später per .filter(Boolean) entfernt
          const priority = urlPriorityMap.get(url) || 0.65;
          // Map.get + || Fallback — Module 6.6 + 2.4. (Würde HIER ??
          // statt || besser passen, wie in Modul 2.7 diskutiert? priority
          // 0 wäre ein gültiger, aber sehr niedriger Wert — || würde 0
          // fälschlich zu 0.65 machen. In der Praxis kommen Prioritäten
          // aber selten als exakt 0 vor, daher ist || hier vertretbar,
          // ?? wäre dennoch "korrekter".)
          return {
              url,                              // Shorthand: url: url
              text: flat.toLowerCase(),
              rawText: raw,
              images: url.endsWith('.md') && raw ? extractImagesFromRaw(raw, url) : [],
              // && und Ternär kombiniert (Modul 2.4) — "NUR bei .md-
              // Dateien MIT Inhalt Bilder extrahieren, sonst []"
              isBackup: /\/backups?\//i.test(url) || /\/old\//i.test(url),
              // Regex .test() (Modul 10.1) + || — "Pfad enthält /backup/
              // ODER /backups/ ODER /old/"
              isGameIdea: /\/gameideas\//i.test(url) || /spieleideen/i.test(url),
              priority: priority
          };
      });
      const results = await Promise.all(fetchPromises);
      // Promise.all (Modul 8.6) — ALLE fetchFileContent-Aufrufe LAUFEN
      // PARALLEL, NICHT nacheinander! Bei z.B. 30 Seiten könnte das
      // sequenziell 30x so lange dauern.
      searchIndex = results.filter(Boolean);
      // .filter(Boolean) — Modul 2.6 (Truthy/Falsy)! Boolean als
      // FUNKTION übergeben (Modul 4.7) — entfernt ALLE null-Einträge
      // (von "if (!flat) return null;" oben), weil Boolean(null) ===
      // false
      indexReady = true;
      console.log(`✅ Index fertig: ${searchIndex.length} Dokumente geladen.`);
  }

  function fallbackSearch(userMessage) {
      const words = userMessage.toLowerCase().split(/\W+/).filter(w => w.length > 3);
      // split mit Regex \W+ (Modul 10.2: \W = KEIN Wortzeichen, also
      // Leerzeichen/Satzzeichen als Trenner) + filter (Modul 6.2) —
      // nur "längere" Wörter (>3 Zeichen) zählen für die Bewertung

      const scored = searchIndex.map(doc => {
          let score = 0;
          words.forEach(word => {
              if (doc.url.toLowerCase().includes(word)) score += 50;
              if (doc.text.includes(word)) score += 10;
              // Treffer im URL/Pfad zählt MEHR (50) als Treffer im
              // Fließtext (10) — eine einfache "Relevanz-Heuristik"
          });
          return { doc, score };
      });

      const topDocs = scored
          .filter(x => x.score > 0)
          .sort((a, b) => b.score - a.score)
          .slice(0, 3)
          .map(x => x.doc);
      // GENAU die Pipeline aus deinem Mini-Alfonz (21.3)! filter → sort
      // (absteigend, Modul 6.1) → slice (Top 3) → map (nur das
      // doc-Objekt, ohne score)

      const context = topDocs
          .map(d => `SOURCE: ${d.url}\nCONTENT:\n${d.rawText.substring(0, 2000)}`)
          .join('\n\n---\n\n');
      // .map dann .join (Modul 6.1/2.3) — Array von Strings → EIN String

      return { context, images: [] };
  }

MODUL 22: TypeScript — JavaScript mit Typsicherheit
================================================================================

Du hast in Modul 2 gelernt: JS ist "dynamisch getypt" — eine Variable
kann ihren Typ wechseln, und Tippfehler bei Property-Namen (proekt.nme
statt projekt.name) fallen oft ERST zur LAUFZEIT auf, als
"undefined is not a function" oder "Cannot read properties of undefined"
(Modul 19.4) — manchmal erst, wenn ein Nutzer genau DIESEN Pfad im Code
auslöst, Wochen nach dem Schreiben.

TypeScript (TS) ist eine SUPERMENGE von JavaScript: JEDES gültige JS ist
auch gültiges TS. TS fügt eine TYP-EBENE hinzu, die der TS-COMPILER
PRÜFT, BEVOR der Code überhaupt läuft — und die danach WIEDER ENTFERNT
wird (TS wird zu "normalem" JS "compiliert"/"transpiliert"). Im Browser
oder in Node.js läuft am Ende IMMER reines JavaScript — TS ist ein
Werkzeug für DICH als Entwickler:in, nicht für den Browser.

  function addiere(a: number, b: number): number {
      return a + b;
  }
  addiere(5, "3");
  // ❌ KOMPILIERFEHLER, SCHON BEVOR der Code läuft:
  // "Argument of type 'string' is not assignable to parameter of type 'number'."

  Der FEHLER aus Modul 2.4 (String-Plus-Falle bei Formular-Werten) wird
  hier zu etwas, das der EDITOR (z.B. VS Code) dir SOFORT beim Schreiben
  unterstreicht — nicht erst, wenn ein Nutzer ihn auslöst.

22.2  Grundtypen und Type Inference
------------------------------------------------------------------------

  let name: string = "Alfonz";
  let alter: number = 400_000_000_000;
  let aktiv: boolean = true;
  let irgendwas: any = "Vorsicht — any deaktiviert ALLE Prüfungen!";
  let unbekannt: unknown = irgendwas;  // sicherer als any — siehe 22.6

  // Arrays:
  let projekte: string[] = ["RSA", "TrafkSite", "TrafkTux"];
  let zahlen: Array<number> = [1, 2, 3];   // alternative Schreibweise

  // TYPE INFERENCE — TS "errät" den Typ oft SELBST, KEINE Annotation nötig:
  let x = 42;          // TS weiß: x ist number (auch ohne ": number")
  x = "Text";          // ❌ Fehler — TS merkt sich den ERSTEN erkannten Typ!

  // Funktions-Rückgabewerte werden meist auch INFERIERT:
  function verdoppeln(n: number) {
      return n * 2;    // TS weiß: Rückgabetyp ist number, OHNE ": number"
  }

In der PRAXIS schreibst du also NICHT überall Typen hin — nur dort, wo
TS sie NICHT selbst herausfinden kann (vor allem: Funktionsparameter,
und komplexere Strukturen wie unten).

22.3  Interfaces und Type Aliases — eigene Objekt-Formen
------------------------------------------------------------------------

Das ist der Teil, der bei Webseiten-Daten (Modul 6!) am meisten bringt —
eine "Form", die ein Objekt haben MUSS:

  interface Projekt {
      name: string;
      jahr: number;
      tags: string[];
      aktiv?: boolean;        // ? = OPTIONAL — darf fehlen
      autor: { name: string; rolle: string };   // verschachtelt
  }

  const p: Projekt = {
      name: "TrafkTux",
      jahr: 2026,
      tags: ["Spiel", "Linux"],
      autor: { name: "Daniel Raberger", rolle: "HoppiTex" }
      // aktiv fehlt — ERLAUBT, weil aktiv? optional ist
  };

  p.jahr = "2027";        // ❌ Fehler: string nicht zu number zuweisbar
  p.neueProperty = "x";   // ❌ Fehler: "neueProperty" existiert nicht auf Projekt!
                           // (würde in PLAIN JS, Modul 6.3, einfach
                           // STILLSCHWEIGEND eine neue Property anlegen —
                           // hier oft ein TIPPFEHLER für "name" o.ä.!)

  // type — ähnlich, oft für UNIONS (siehe 22.4) oder Funktionssignaturen:
  type ID = string | number;
  type Klickhandler = (e: MouseEvent) => void;

  // Faustregel: interface für "Objekt-Formen" (erweiterbar, lesbare
  // Fehlermeldungen), type für ALLES ANDERE (Unions, Funktionstypen,
  // Aliase für primitive Typen). In der Praxis oft austauschbar — viele
  // Teams haben dazu eine STIL-Konvention.

22.4  Union Types und Literal Types — "einer von mehreren bestimmten Werten"
------------------------------------------------------------------------

  // Union — "A ODER B":
  function zeigeStatus(status: "lädt" | "bereit" | "fehler") {
      // status kann NUR diese drei STRINGS sein — sonst Kompilierfehler!
  }
  zeigeStatus("lädt");     // ✓
  zeigeStatus("läd");       // ❌ Tippfehler wird SOFORT erkannt!

  // Aus AskAlfonz.js abgeleitet — eine ChatNachricht (vgl. Modul 11.1):
  interface ChatNachricht {
      role: "user" | "assistant" | "system";   // GENAU diese drei Werte
      content: string;
  }

  let chatHistory: ChatNachricht[] = [];
  chatHistory.push({ role: "user", content: "Hallo!" });
  chatHistory.push({ role: "useer", content: "Tippfehler!" });
  // ❌ "useer" ist nicht "user" | "assistant" | "system" — TS meldet es,
  //    BEVOR der Code an einen Server geschickt wird, der mit
  //    "useer" vielleicht einfach NICHTS anfängt (stiller Bug!)

  // Type Narrowing — TS "merkt sich" Bedingungen:
  function verarbeite(wert: string | number) {
      if (typeof wert === "string") {
          wert.toUpperCase();   // ✓ TS weiß: hier ist wert ein string
      } else {
          wert.toFixed(2);      // ✓ TS weiß: hier ist wert ein number
      }
  }
  // Das ist derselbe typeof-Check aus Modul 2.2/12.3 — TS NUTZT ihn, um
  // den Typ INNERHALB des if-Zweigs zu "verengen" (narrowing).

------------------------------------------------------------------------
22.5  Generics — "Typen als Parameter"
------------------------------------------------------------------------

Generics lassen FUNKTIONEN/KLASSEN mit BELIEBIGEN, aber KONSISTENTEN
Typen arbeiten — <T> ist ein "Platzhalter für einen Typ", der beim
AUFRUF festgelegt wird:

  // OHNE Generics — entweder zu STRENG (nur number) oder zu LOCKER (any):
  function ersterEintrag(arr: any[]): any {
      return arr[0];   // Rückgabetyp ist any — TS "weiß" danach NICHTS mehr
  }

  // MIT Generics — T wird beim Aufruf "automatisch" erkannt:
  function ersterEintrag<T>(arr: T[]): T {
      return arr[0];
  }
  const erstesProjekt = ersterEintrag(["RSA", "TrafkSite"]);
  // TS weiß: erstesProjekt ist ein string — T wurde zu string

  const ersteZahl = ersterEintrag([1, 2, 3]);
  // TS weiß: ersteZahl ist number — T wurde zu number

  // Generische Klasse — z.B. ein "Cache" (verwandt mit Map, Modul 6.6):
  class Cache<T> {
      private daten = new Map<string, T>();   // private — wie # in Modul 11.3,
                                                 // TS-spezifische Schreibweise
      set(key: string, wert: T) { this.daten.set(key, wert); }
      get(key: string): T | undefined { return this.daten.get(key); }
  }

  const projektCache = new Cache<Projekt>();   // T = Projekt
  projektCache.set("tux", p);
  const geladen = projektCache.get("tux");     // Typ: Projekt | undefined

  Generics wirken anfangs abstrakt — die Faustregel: SOBALD eine
  Funktion/Klasse "für JEDEN Typ gleich funktioniert, aber den
  KONKRETEN Typ am Ende WISSEN soll" (Arrays, Caches, Wrapper um
  fetch-Antworten, ...), ist <T> die Lösung.

22.6  any vs. unknown — und warum unknown sicherer ist
------------------------------------------------------------------------

  let a: any = JSON.parse('{"name":"Alfonz"}');
  a.name.toUpperCase();   // ✓ kompiliert — TS prüft GAR NICHTS mehr für a
  a.nameeee.toUpperCase(); // ✓ kompiliert AUCH — aber CRASHT zur Laufzeit!
  // any ist ein "Ausstieg" aus dem Typsystem — nützlich für SCHRITTWEISE
  // Migration von JS zu TS, aber jedes any ist eine "Lücke".

  let u: unknown = JSON.parse('{"name":"Alfonz"}');
  u.name.toUpperCase();   // ❌ Fehler — "Object is of type 'unknown'"
  // ERST nach einer Typ-PRÜFUNG darf man zugreifen:
  if (typeof u === "object" && u !== null && "name" in u) {
      (u as { name: string }).name.toUpperCase();   // jetzt ok — "as" =
                                                       // explizite Typ-
                                                       // Behauptung (mit
                                                       // Vorsicht nutzen!)
  }

Faustregel: unknown für Daten UNBEKANNTER Herkunft (genau wie
fetch-Antworten, JSON.parse-Ergebnisse, Nutzereingaben — siehe Modul 12
und 18!) — ZWINGT dich, die Form zu PRÜFEN, bevor du zugreifst. any nur
als letzten Ausweg, NIEMALS als "Standard".

22.7  TypeScript für DOM-Code — Modul 3 mit Typen

  const input = document.getElementById('chat-input');

  }

  // Event-Handler — das Event-Objekt ist automatisch typisiert:
  button.addEventListener('click', (e: MouseEvent) => {
      e.target   // Typ: EventTarget | null
  });

  inputField.addEventListener('keypress', (e: KeyboardEvent) => {

  });

================================================================================
MODUL 23: Build-Tools, Frameworks und das moderne Web
================================================================================

Dieses letzte Modul verbindet drei Themen, die oft GEMEINSAM auftauchen,
sobald ein Projekt über "ein paar <script>-Tags" hinauswächst: WIE Code
heute gebaut/gebündelt wird, WIE Frameworks die Patterns aus Modul 20
automatisieren — und WIE Webseiten ECHTZEIT-Verbindungen (WebSockets) und
OFFLINE-Fähigkeit (Service Workers) bekommen. Letzteres ist für ein
Chat-Projekt wie AskAlfonz.js besonders relevant.

------------------------------------------------------------------------
23.1  Warum überhaupt ein Build-Schritt?
------------------------------------------------------------------------

TrafkSite funktioniert OHNE Build-Schritt: .html/.css/.js-Dateien werden
1:1 von GitHub Pages ausgeliefert, der Browser lädt sie GENAU SO, wie sie
im Repository liegen. Das ist EINFACH und für dieses Projekt RICHTIG.

Ein "Build-Schritt" (engl. "bundling") wird interessant, wenn:

  - du TypeScript (Modul 22) schreibst — der Browser versteht KEIN TS,
    es muss zu JS COMPILIERT werden.
  - du VIELE kleine ES-Module (Modul 13) hast — ohne Bundler lädt der
    Browser JEDE Datei als EIGENEN HTTP-Request (langsamer bei VIELEN
    Dateien, je nach HTTP-Version).
  - du npm-Pakete (Modul 17.2) im BROWSER nutzen willst — npm-Pakete
    sind oft für Node.js geschrieben (require, fs-Zugriffe, ...), ein
    Bundler "übersetzt" sie für den Browser.
  - du Frameworks wie React/Vue (23.3) nutzt — deren spezielle Syntax
    (JSX, .vue-Dateien) MUSS zu normalem JS übersetzt werden.
  - du Code für PRODUKTION optimieren willst: Minifizierung (Variablen
    umbenennen zu a, b, c — kleinere Dateien), Tree Shaking (UNGENUTZTEN
    Code aus npm-Paketen ENTFERNEN — aus Modul 13.3 bereits angedeutet),
    Code-Splitting (Code in mehrere Dateien aufteilen, die NUR bei
    Bedarf geladen werden — verwandt mit dynamischem import() aus 13.4).

------------------------------------------------------------------------
23.2  Vite — ein moderner Bundler in der Praxis
------------------------------------------------------------------------

Vite (französisch für "schnell") ist ein populäres, modernes Build-Tool.
Der Workflow:

  npm create vite@latest mein-projekt
  cd mein-projekt
  npm install
  npm run dev      // startet einen lokalen Entwicklungsserver mit
                    // "Hot Module Replacement" — Änderungen im Code
                    // erscheinen SOFORT im Browser, OHNE Neuladen!
  npm run build     // erzeugt einen dist/-Ordner mit OPTIMIERTEM,
                     // fertigem HTML/CSS/JS — DAS wird deployed (z.B.
                     // auf GitHub Pages)

  // Projektstruktur (vereinfacht):
  mein-projekt/
    index.html        ← Vite-Einstiegspunkt (mit <script type="module" src="/src/main.js">)
    src/
      main.js          ← dein Code, kann TS, ES-Module, npm-Imports
                          enthalten
      style.css
    package.json
    vite.config.js     ← Konfiguration

  // In src/main.js — IMPORTS, die OHNE Vite im Browser NICHT
  // funktionieren würden (npm-Paket direkt importiert!):
  import { format } from 'date-fns';     // npm-Paket
  import './style.css';                   // CSS direkt importieren — Vite
                                            // baut daraus ein <link>-Tag
  import logoUrl from './logo.png';       // Bilder als Import — Vite gibt
                                            // den FERTIGEN URL zurück

  console.log(format(new Date(), 'dd.MM.yyyy'));

DIE ENTWICKLUNGS-ERFAHRUNG ändert sich GRUNDLEGEND: statt Datei speichern
→ Browser manuell neu laden (wie bisher bei TrafkSite), läuft npm run dev
DAUERHAFT im Terminal, und JEDE Änderung erscheint INNERHALB von
Millisekunden im Browser — bei GLEICHBLEIBENDEM Zustand (ein Formular,
das du gerade ausfüllst, bleibt ausgefüllt, während sich NUR die
geänderte Komponente aktualisiert).

------------------------------------------------------------------------
23.3  Frameworks — Module 20 "eingebaut"
------------------------------------------------------------------------

Erinnerung an Modul 20.3/20.4: AppState (zentraler Zustand) + renderChat
(Render-Funktion, die bei Zustandsänderung NEU aufbaut). Frameworks wie
React und Vue AUTOMATISIEREN genau dieses Muster — und lösen das
Performance-Problem aus 20.4 ("bei VIELEN Elementen ALLES neu bauen ist
langsam") mit einem "Virtual DOM": das Framework vergleicht ALT vs. NEU
und ändert NUR die UNTERSCHIEDE im echten DOM.

REACT — Beispiel, GLEICHE Logik wie Mini-Alfonz (Modul 21.4/21.5):

  import { useState } from 'react';

  function ChatFenster() {
      // useState — wie EIN Eintrag in AppState (20.3), ABER: jede
      // Änderung von chatVerlauf (per setChatVerlauf) löst AUTOMATISCH
      // ein Re-Render aus — KEIN manuelles renderChat() nötig!
      const [chatVerlauf, setChatVerlauf] = useState([]);
      const [eingabe, setEingabe] = useState('');

      function sendeNachricht() {
          if (!eingabe.trim()) return;   // Modul 4.4, früh-return!
          setChatVerlauf([...chatVerlauf, { rolle: 'user', text: eingabe }]);
          // Spread (Modul 6.5) — NEUES Array statt .push (React erkennt
          // Änderungen über NEUE Referenzen, nicht über Mutation!)
          setEingabe('');
      }

      // JSX — sieht aus wie HTML, ist aber JS (wird zu
      // React.createElement(...)-Aufrufen compiliert — DAS ist der
      // Build-Schritt aus 23.1!):
      return (
          <div>
              <div id="chat-window">
                  {chatVerlauf.map((eintrag, i) => (
                      <div key={i} className={eintrag.rolle === 'user' ? 'msg-user' : 'msg-bot'}>
                          {eintrag.text}
                          {/* React escaped {eintrag.text} AUTOMATISCH —
                              wie textContent in Modul 18.1! */}
                      </div>
                  ))}
              </div>
              <input value={eingabe} onChange={e => setEingabe(e.target.value)} />
              {/* onChange feuert bei JEDEM Tastendruck — wie das
                  'input'-Event aus Modul 7.2, NICHT 'change'! React
                  benennt es nur anders. */}
              <button onClick={sendeNachricht}>Senden</button>
          </div>
      );
  }

VUE — DIESELBE Idee, andere Syntax (Template + <script setup>):

  <template>
    <div>
      <div id="chat-window">
        <div v-for="(eintrag, i) in chatVerlauf" :key="i"
             :class="eintrag.rolle === 'user' ? 'msg-user' : 'msg-bot'">
          {{ eintrag.text }}
        </div>
      </div>
      <input v-model="eingabe" />
      <button @click="sendeNachricht">Senden</button>
    </div>
  </template>

  <script setup>
  import { ref } from 'vue';
  const chatVerlauf = ref([]);
  const eingabe = ref('');

  function sendeNachricht() {
      if (!eingabe.value.trim()) return;
      chatVerlauf.value.push({ rolle: 'user', text: eingabe.value });
      eingabe.value = '';
  }
  </script>

WAS BLEIBT GLEICH, WAS ÄNDERT SICH?

  GLEICH:                              ANDERS:
    if (!eingabe.trim()) return;        kein manuelles renderChat()
    (Modul 4.4)                         mehr — "Reaktivität" ist
    Spread/Array-Methoden (Modul 6)     EINGEBAUT
    Events feuern bei denselben         JSX/Templates statt
    Browser-Events (Modul 5) — nur      createElement+appendChild
    andere SCHREIBWEISE (onClick        (Modul 3.6)
    statt addEventListener)             Build-Schritt PFLICHT (23.1)

WANN FRAMEWORK, WANN VANILLA-JS (wie TrafkSite)?

  Vanilla JS (aktuelle TrafkSite) passt, wenn:
    - die Seite ÜBERWIEGEND statisch ist, mit "Inseln" von
      Interaktivität (Lautstärkeregler, Navigation, EIN Chat-Widget)
    - KEIN Build-Schritt gewünscht ist (einfaches Deployment auf GitHub
      Pages, KEIN npm-Setup nötig)
    - die Anzahl der "lebenden" UI-Teile überschaubar bleibt — Module 20
      reicht aus

  Ein Framework lohnt sich, wenn:
    - die GESAMTE Seite eine "Anwendung" ist, mit VIELEN, stark
      verschachtelten, gegenseitig abhängigen UI-Teilen
    - du dieselben "Komponenten" (z.B. eine Projekt-Karte) an VIELEN
      Stellen WIEDERVERWENDEN willst — verwandt mit, aber mächtiger als
      Web Components (Modul 16)
    - du in einem TEAM arbeitest, das von ETABLIERTEN Konventionen
      profitiert

  → TrafkSite, so wie es aktuell aufgebaut ist (statische Sections +
  ein paar Skripte für Nav/Lautstärke/Chat), ist ein LEHRBUCH-Beispiel
  für "Vanilla JS reicht" — und GENAU DESHALB war dieser Kurs darauf
  ausgerichtet.

------------------------------------------------------------------------
23.4  WebSockets — echte Echtzeit-Kommunikation
------------------------------------------------------------------------

fetch (Modul 8) folgt dem Muster "Anfrage → EINE Antwort → fertig". Für
einen CHAT wäre eigentlich SCHÖNER: der SERVER könnte dem Client
JEDERZEIT etwas SCHICKEN — z.B. "Alfonz tippt gerade...", oder Nachrichten
von ANDEREN Nutzern in einem GEMEINSAMEN Chat. Das ist WebSockets.

  // BROWSER-Seite:
  const ws = new WebSocket('wss://trafkhop-entertainment-alfonzproxy.hf.space/ws');
  //         ^wss:// = WebSocket SECURE (wie https:// bei fetch)

  ws.addEventListener('open', () => {
      console.log('Verbindung steht!');
      ws.send(JSON.stringify({ type: 'chat', text: 'Hallo Alfonz!' }));
      // .send() — JEDERZEIT, nicht nur als "Antwort" auf etwas
  });

  ws.addEventListener('message', e => {
      const data = JSON.parse(e.data);
      // KOMMT JEDERZEIT — auch OHNE dass der Browser etwas angefragt hat!
      if (data.type === 'tippt') {
          statusEl.textContent = `${data.sender} tippt...`;
      } else if (data.type === 'chat') {
          fuegeNachrichtHinzu('bot', data.text);   // Modul 21.4!
      }
  });

  ws.addEventListener('close', () => console.log('Verbindung beendet'));
  ws.addEventListener('error', e => console.error('WS-Fehler:', e));

  // NODE.JS-Seite (verwandt mit index.js, Modul 17.3) — mit dem Paket "ws":
  const { WebSocketServer } = require('ws');
  const wss = new WebSocketServer({ port: 8080 });

  wss.on('connection', socket => {
      socket.on('message', raw => {
          const data = JSON.parse(raw);
          // an ALLE verbundenen Clients weiterleiten — "Broadcast":
          wss.clients.forEach(client => {
              client.send(JSON.stringify({ type: 'chat', sender: 'Jemand', text: data.text }));
          });
      });
  });

fetch VS. WebSocket — WANN WAS?

  fetch (Modul 8)                    WebSocket
    "Frag einmal, bekomm einmal       "Bleib verbunden, beide Seiten
    Antwort" — Request/Response       können JEDERZEIT senden"
    Zustandslos pro Aufruf             Verbindung bleibt offen (Zustand!)
    Gut für: Daten laden, Formulare    Gut für: Chat mit MEHREREN
    senden, einmalige Aktionen         Nutzern, Live-Updates, "tippt
                                        gerade", Multiplayer

  AskAlfonz.js (Modul 21.8) nutzt AKTUELL fetch — passend, weil JEDE
  Chat-Anfrage EINE Anfrage → EINE Antwort ist, und es NUR EINEN Nutzer
  pro Browser-Tab gibt. WÜRDE TrafkSite einen "Live-Chat zwischen
  mehreren Besuchern" bekommen (z.B. ein gemeinsamer Chatraum), wäre DAS
  der Moment für WebSockets — fetch könnte "neue Nachrichten von ANDEREN"
  nur durch wiederholtes Nachfragen ("Polling", z.B. alle 2 Sekunden per
  setInterval, Modul 8.2) simulieren — WebSockets liefern sie SOFORT.

23.5  Service Workers — Offline-Fähigkeit und Hintergrund-Logik
------------------------------------------------------------------------

Ein Service Worker ist ein SEPARATES JS-Skript, das der Browser
IM HINTERGRUND ausführt — UNABHÄNGIG von einem geöffneten Tab. Er kann
sich zwischen die Seite und das NETZWERK schalten ("Proxy im Browser") —
ähnlich wie index.js sich zwischen AskAlfonz.js und die KI-API schaltet
(Modul 17.3/18.2), nur LOKAL im Browser des Nutzers.

  // registrierung — in einem normalen <script> (z.B. placeholder.js
  // oder einer neuen Datei):
  if ('serviceWorker' in navigator) {   // Feature-Detection — Modul 12.3!
      navigator.serviceWorker.register('/sw.js');
  }

  // sw.js — läuft in einem EIGENEN Kontext, KEIN document, KEIN window!
  const CACHE_NAME = 'trafksite-v1';
  const DATEIEN = [
      '/TrafkSite/index.html',
      '/TrafkSite/files/standard.css',
      '/TrafkSite/files/nav.js',
      '/TrafkSite/files/volume.js',
      '/TrafkSite/files/placeholder.js'
  ];

  // Beim Installieren — Dateien VORAB in den Cache legen:
  self.addEventListener('install', e => {
      e.waitUntil(
          caches.open(CACHE_NAME).then(cache => cache.addAll(DATEIEN))
      );
  });

  // Bei JEDEM Netzwerk-Request der Seite — "Cache-First"-Strategie:
  self.addEventListener('fetch', e => {
      e.respondWith(
          caches.match(e.request).then(cached => {
              return cached || fetch(e.request);
              // ?? wäre hier NICHT richtig (Modul 2.7) — || passt: ist
              // cached ein "echtes" Response-Objekt (truthy) oder
              // undefined (falsy, kein Cache-Treffer)?
          })
      );
  });

  // Veraltete Caches aufräumen — beim Aktivieren eines NEUEN Service Workers:
  self.addEventListener('activate', e => {
      e.waitUntil(
          caches.keys().then(keys =>
              Promise.all(keys.filter(k => k !== CACHE_NAME).map(k => caches.delete(k)))
              // filter + map + Promise.all — Module 6.2 + 8.6, jetzt im
              // Service-Worker-Kontext!
          )
      );
  });

EFFEKT FÜR TRAFKSITE: Mit diesem (vereinfachten) Service Worker würde
TrafkSite — EINMAL besucht — beim ZWEITEN Besuch SOFORT aus dem
Browser-Cache laden, sogar OHNE Internetverbindung (Logo, Navigation,
Lautstärkeregler funktionieren OFFLINE; NUR der AskAlfonz-Chat würde
weiterhin fetch zum Proxy brauchen und entsprechend OHNE Internet
fehlschlagen — siehe Modul 12, try/catch fängt das ab).

Ein Service Worker ist die TECHNISCHE Basis für PWAs (Progressive Web
Apps) — Webseiten, die sich "wie Apps" installieren lassen (Icon auf dem
Homescreen, Start ohne Browser-Adressleiste), inklusive Push-
Benachrichtigungen (über einen weiteren Mechanismus, die Push API — über
diesen Kurs hinaus, aber jetzt kennst du den EINSTIEGSPUNKT: der Service
Worker).
